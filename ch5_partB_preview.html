P10FlashAttention:IO 感知的精确 attention

对应主书:第 3 章 4.7.3 算子优化 · 第 4 章 5.2.2 / 5.8

FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness
作者Tri Dao, Daniel Y. Fu, Stefano Ermon, Atri Rudra, Christopher Ré
机构Stanford University
发表NeurIPS 2022 · arXiv:2205.14135 · 被引 5,000+
后续FlashAttention-2 (2023), FlashAttention-3 (2024, FP8 + H100 优化)

问题:attention 的真瓶颈不是 FLOPs

Self-attention 在长序列下的时间复杂度是 $O(L^2 d)$,被认为是"计算瓶颈"。但 Tri Dao 观察到:GPU 上 attention 实际是内存带宽瓶颈,不是计算瓶颈。

数据:A100 GPU 计算 FP16 GEMM 有 312 TFLOPs,但 HBM 带宽只有 1.5 TB/s。两者比值(算术强度)要 ~208 FLOP/byte 才能用满算力——标准 attention 实现的算术强度远低于此。大部分时间 GPU 在等 HBM 搬数据。

标准 attention 的问题

标准 PyTorch 实现做四件事:

  1. 计算 $S = QK^\top \in \mathbb{R}^{L \times L}$,写到 HBM
  2. 从 HBM 读 $S$,算 $P = \text{softmax}(S)$,写回 HBM
  3. 从 HBM 读 $P$,和 $V$ 乘得 $O$,写回 HBM
  4. (可能的 dropout、mask 又几次读写)

$L = 4096$ 时,$S$ 矩阵 64 MB。每次读写都是 128 MB 的 HBM 往返。L 大,这个开销线性增长,但算力利用率极低。

核心方法:tiling + 在线 softmax

FlashAttention 把 Q、K、V 切成 tile,每个 tile 大小是 GPU 片上 SRAM 能容纳的。外层循环 Q 的 tile $Q_i$,内层循环 K、V 的 tile $(K_j, V_j)$。 \lt strong>永不物化完整的 $S$ 或 $P$——所有中间结果留在 SRAM 里。

flowchart TB HBM["HBM 大容量慢"] -. "读 Q_i, K_j, V_j" .-> SRAM SRAM["SRAM 小容量快
tile 内计算"] -. "写 O_i" .-> HBM SRAM --> S["S_ij = Q_i K_j^T (在 SRAM)"] S --> SM["在线 softmax 更新 m, l, O"] SM --> NEXT["下一个 K_j, V_j"]

关键难点:在线 softmax

softmax 需要归一化——$\text{softmax}(x_i) = \exp(x_i) / \sum_j \exp(x_j)$。看起来必须等所有 $x_j$ 才能算分母。但 FlashAttention 用 \lt strong>增量更新 \lt /strong>:每看一个新 $S_{ij}$ tile,维护 running max 和 running sum,最后等价于完整 softmax。

设已处理到第 $j-1$ 个 tile,状态是 $(m_{j-1}, l_{j-1}, O_{j-1})$——当前位置已见最大值 $m$,当前分母 $l$,当前加权输出 $O$。新来一个 tile $(K_j, V_j)$:

算 $S_{ij} = Q_i K_j^\top$。令 $\tilde{m}_j = \max(S_{ij})$。新 max:

$m_j = \max(m_{j-1}, \tilde{m}_j)$

新的(和旧的)指数项都用新 max 重缩放:

$l_j = e^{m_{j-1} - m_j} \cdot l_{j-1} + \sum_k e^{S_{ij,k} - m_j}$

新输出(旧贡献要按新 max 重缩放):

$O_j = \frac{1}{l_j}\left[ l_{j-1} e^{m_{j-1} - m_j} O_{j-1} + \sum_k e^{S_{ij,k} - m_j} V_{j,k} \right]$

把所有 K/V tile 扫完,$O_J$ 等价于完整 softmax 的结果。数值上和标准 attention 精确等价(不是近似!)。

伪代码

def flash_attention(Q, K, V, block_size=64):
    """Q,K,V: [L, d].  返回 O: [L, d], 永不物化 LxL 的 S"""
    L, d = Q.shape
    O = torch.zeros_like(Q)
    m = torch.full((L, 1), -float('inf'))  # 每行的 running max
    l = torch.zeros((L, 1))                 # 每行的 running sum

    for j in range(0, L, block_size):
        K_j = K[j:j+block_size]              # tile 载入 SRAM
        V_j = V[j:j+block_size]
        S_ij = Q @ K_j.T / math.sqrt(d)      # 在 SRAM 内算, [L, block]

        # Causal mask (如果需要)
        mask = torch.arange(j, j+K_j.shape[0]).unsqueeze(0) > \
               torch.arange(L).unsqueeze(1)
        S_ij = S_ij.masked_fill(mask, -float('inf'))

        m_new = torch.maximum(m, S_ij.max(dim=-1, keepdim=True).values)
        P_ij = torch.exp(S_ij - m_new)
        l_new = torch.exp(m - m_new) * l + P_ij.sum(dim=-1, keepdim=True)

        # 旧 O 按新 max 重缩放, 加上当前 tile 的贡献
        O = (l / l_new) * torch.exp(m - m_new) * O + (P_ij @ V_j) / l_new
        m, l = m_new, l_new

    return O

真正的 CUDA 实现有大量工程细节——warp 级 shuffling、寄存器复用、pipeline——但核心算法就是上面这段。

收益

  • - HBM 访问:从 $O(L^2 d)$ 降到 $O(L^2 d^2 / M)$($M$ 是 SRAM 大小)——降几个数量级
  • - 速度:A100 上比标准 attention 快 2-4×,长序列更明显
  • - 显存:中间 $S, P$ 不再物化,显存从 $O(L^2)$ 降到 $O(L)$——这是让 128K+ 上下文可行的关键

FlashAttention-2 和 -3

FlashAttention-2(2023):精细化外层循环、减少非 matmul 操作、更好的 warp 分工。再快 2×,总共对比标准 attention 快 4-8×。

FlashAttention-3(2024):针对 H100 Hopper 架构,用上 FP8 Tensor Core、异步执行、更精细的 pipeline。H100 上接近峰值算力的 75%。

这三代是 2022-2024 年 attention 优化最重要的工作,底层影响了 PyTorch SDPA、vLLM、xFormers、TensorRT-LLM 的实现。现代 LLM 训练和推理几乎都默认用 FlashAttention。

启示

FlashAttention 的影响超出 attention 本身——它是"IO-aware 算法设计"理念的典范。很多 GPU 算子都能类似重构:把中间结果留在 SRAM、用数学上等价的增量形式避免大中间量。后来的 FlashDecoding、RingAttention 都是这个思路的延续。

为什么是 2 页:在线 softmax 算法是必须亲手推一遍才能真懂的内容。搞懂它,你会在很多地方发现类似技巧——PagedAttention、StreamingLLM、FlashDecoding 都依赖这套数学。

P11LoRA / QLoRA:让微调变得便宜

对应主书:第 3 章 4.5.3 微调 · 第 4 章 5.4.1 低秩思想

LoRA: Low-Rank Adaptation of Large Language Models
作者Edward Hu, Yelong Shen, Phillip Wallis, Zeyuan Allen-Zhu, Yuanzhi Li, Shean Wang, Lu Wang, Weizhu Chen
机构Microsoft
发表ICLR 2022 · arXiv:2106.09685 · 被引 8,000+
QLoRA: Efficient Finetuning of Quantized LLMs
作者Tim Dettmers 等(University of Washington)
发表NeurIPS 2023 · arXiv:2305.14314

问题:全量微调大模型太贵

微调 70B 模型需要存:参数 140GB(FP16)+ 梯度 140GB + Adam 状态 560GB ≈ 840GB,再加激活。单机多卡都不够——必须 ZeRO-3 级别的分布式。

但很多场景只需要"轻度定制"——给模型加一个领域风格、让它按特定格式输出。是否必须调所有参数?

LoRA 的核心假设

Hu 等人的假设:模型适配到下游任务时,权重变化 $\Delta W$ 具有低秩结构。

即 $\Delta W \in \mathbb{R}^{d \times k}$ 可以分解成 $\Delta W = BA$,其中 $B \in \mathbb{R}^{d \times r}, A \in \mathbb{R}^{r \times k}$,$r \ll \min(d, k)$。

训练时冻结原权重 $W_0$,只训 $A, B$:

$W = W_0 + \Delta W = W_0 + BA, \quad h = W_0 x + BA x$
flowchart LR X[x] --> W0["W_0 (冻结, d x k)"] W0 --> Y1[W_0 x] X --> A["A (r x k, 可训)"] A --> B["B (d x r, 可训)"] B --> Y2["BA x"] Y1 --> ADD((+)) Y2 --> ADD ADD --> H[h]

初始化:$A$ 用高斯初始化、$B$ 初始化为 0——训练起点 $BA = 0$,和全量微调同起点。

参数量对比

GPT-3 175B,某层 $W \in \mathbb{R}^{12288 \times 12288}$,150M 参数。取 $r = 8$:LoRA 参数 $A \in \mathbb{R}^{8 \times 12288}, B \in \mathbb{R}^{12288 \times 8}$,总共 200K 参数——减少 750×。

全模型层面,LoRA 只训约 0.01% 的参数。显存需求:权重冻结不需要 Adam 状态、梯度也只对 $A, B$ 算——比全量微调省 10-20× 显存。

实现(PEFT)

from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM

base = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3-8B")

config = LoraConfig(
    r=16,                          # rank
    lora_alpha=32,                 # 缩放因子, 实际 W += (alpha/r) BA
    target_modules=["q_proj", "v_proj"],  # 只给 Q, V 投影加 LoRA
    lora_dropout=0.05,
    task_type="CAUSAL_LM",
)
model = get_peft_model(base, config)
model.print_trainable_parameters()
# trainable params: 4,194,304 || all params: 8,034,455,552 || trainable%: 0.05%

推理时合并

推理时可以把 $BA$ 合回 $W_0$:$W_{\text{new}} = W_0 + BA$,变回一个普通权重矩阵。推理零额外开销,这是 LoRA 极大的优势——和 Adapter(在层之间插入小模块)相比,LoRA 推理不影响延迟。

QLoRA:再省 4 倍显存

QLoRA(2023)把原模型量化到 4-bit,LoRA 参数保持 FP16。显存:

  • - 冻结权重:140GB(FP16)→ 35GB(INT4),省 4×
  • - LoRA 参数和梯度:几十 MB,可忽略
  • - 总需求:65B 模型单卡 48GB 可微调

QLoRA 的几个技术创新:

  1. NF4(NormalFloat 4-bit):针对正态分布权重设计的 4-bit 数据类型,比普通 INT4 精度高
  2. 双重量化:连量化的 scale 也量化,进一步省内存
  3. Paged Optimizers:借 NVIDIA Unified Memory 在显存不够时 swap 到 CPU

QLoRA 让"用消费级显卡微调 65B 模型"变成现实——是开源 LLM 社区参与微调的门槛拐点。Alpaca、Vicuna 等一系列社区微调项目都用 LoRA/QLoRA。

变种

  • LoRA+:$A, B$ 用不同 lr(B 应该更大),小改动带来 1-2% 效果提升
  • DoRA(Weight-Decomposed LoRA):把权重分解成 magnitude 和 direction,分别调
  • AdaLoRA:动态分配每层的 rank
  • rsLoRA:改进缩放规则,让大 r 也稳定

影响

LoRA 是 2022 年后微调领域最重要的工作。今天几乎所有开源 LLM 的 fine-tune 都用 LoRA 或其变种——HuggingFace PEFT 库就是围绕它建的。LoRA 还激发了架构设计的思想:DeepSeek 的 MLA 用低秩分解压缩 KV Cache(主书 5.4.1),精神一脉相承。

为什么是 2 页:LoRA 是开源社区最重要的"杠杆"——让个人和小团队也能参与前沿 LLM 适配。QLoRA 的 NF4 设计值得一读,对"面向分布设计数据类型"有很好的启发。

P12PagedAttention / vLLM:把 KV Cache 当虚拟内存

对应主书:第 4 章 5.8.3

Efficient Memory Management for Large Language Model Serving with PagedAttention
作者Woosuk Kwon, Zhuohan Li, Siyuan Zhuang, Ying Sheng, Lianmin Zheng, Cody Hao Yu, Joseph E. Gonzalez, Hao Zhang, Ion Stoica
机构UC Berkeley, Stanford, UCSD, CMU
发表SOSP 2023 · arXiv:2309.06180

问题:KV Cache 的内存碎片

LLM 推理每个请求需要连续的 KV Cache 空间。朴素实现按 max_seq_len 预分配:一个请求预留 4096 token 空间,实际只用 500,浪费 87%。系统级上有四种浪费:

  1. 保留浪费:为"可能更长"预留的空间
  2. 内部碎片:每个请求实际长度不同,预留空间用不完
  3. 外部碎片:短请求结束留下的小块,装不下新的大请求
  4. 无法共享:相同前缀的请求各存一份(浪费)

核心方法:OS 虚拟内存搬到 KV Cache

PagedAttention 直接类比 OS 虚拟内存:

  • GPU 显存切成固定大小 block(16 个 token 的 KV 一块)
  • 每个请求维护 block table,记录它的 token 位置 → 物理 block
  • Block 按需分配、按 token 增长动态扩展,不预留
  • Attention kernel 通过 block table 间接访问
flowchart TB subgraph GPU["GPU Memory Pool (blocks of 16 tokens KV)"] B0[Block 0: free] B3[Block 3: R1 tokens 16-19] B5[Block 5: R2 tokens 0-14] B7[Block 7: R1 tokens 0-15] B8[Block 8: free] end R1["Request 1 (20 tokens)
block_table: [7, 3]"] --> B7 R1 --> B3 R2["Request 2 (15 tokens)
block_table: [5]"] --> B5

修改后的 attention kernel

标准 kernel 假设 K、V 连续,PagedAttention 的 kernel 按 block table 间接查位置:

// 简化伪代码
for each query q_i:
    output = 0
    max_score = -inf; sum_exp = 0
    for each logical token j in query's block_table:
        block_id  = block_table[j / BLOCK_SIZE]
        inner_idx = j % BLOCK_SIZE
        k_j = K_pool[block_id * BLOCK_SIZE + inner_idx]
        v_j = V_pool[block_id * BLOCK_SIZE + inner_idx]
        score = q_i @ k_j / sqrt(d)
        // 在线 softmax 更新(同 FlashAttention)
        ...

一次额外的 indirection(block_table[...])带来小开销,但省的内存远超这点开销。

共享与 Copy-on-Write

多个请求前缀相同时(system prompt、few-shot 示例),block_table 可以指向相同物理 block。写时才复制(某个请求开始产生不同 token 时才分裂)。这让prefix caching 成为自然的优化——高并发 LLM 服务直接受益。

beam search 和并行采样同样受益:多个候选共享前缀 KV Cache。

实测收益

论文数据(LLaMA-13B):

  • - 内存利用率:~40% → ~96%
  • - 同显存下并发请求数:4-10×
  • - 吞吐:2-4× 提升

vLLM

论文作者同时开源了 vLLM 项目——把 PagedAttention + 连续批处理 + 推测解码打包成生产级推理引擎。2023 年中发布以来:

  • - 成为开源 LLM 服务的事实标准
  • - 2024 年 OpenRank 上和 SGLang 并列双强
  • - 2025 年推出 v1 大重构,架构全面升级
  • - 被几乎所有主流云厂商、LLM 服务商采用

vLLM 的成功超过 PagedAttention 论文本身的引用价值——它是"研究成果变成工业基础设施"的典范。

启示

跨领域借鉴的典型例子。操作系统的虚拟内存概念在 1961 年就已经成熟(Atlas 计算机),60 多年后被搬到 LLM 推理,解决了一个新问题。学过 OS 的程序员读这篇论文几乎"一眼秒"——这是基础知识的威力。

为什么必读:不读这篇,就不理解为什么 vLLM 能跑到比朴素实现快 5-10 倍。对做系统工程的人,这也是"跨界借鉴"的优秀范本。

P13Switch Transformer 与 Mixtral:把 MoE 做成实用

对应主书:第 4 章 5.6

Switch Transformers: Scaling to Trillion Parameter Models with Simple and Efficient Sparsity
作者William Fedus, Barret Zoph, Noam Shazeer(Google)
发表JMLR 2022 · arXiv:2101.03961
Mixtral of Experts
作者Mistral AI team
发表arXiv:2401.04088 (2024)

MoE 的老思想

MoE 概念可追溯到 Jacobs 1991 年的 "Adaptive Mixtures of Local Experts"。Shazeer 等 2017 年首次用到深度学习(Sparsely-Gated MoE)。但真正证明 MoE 实用的是 Switch Transformer 和 Mixtral。

Switch Transformer:Top-1 路由

Switch 的核心简化:每个 token 只路由到 1 个专家(之前工作多用 top-2)。极度简化 gating,降低通信复杂度。

$\text{expert}(x) = \arg\max_i g_i(x), \quad g(x) = \text{softmax}(W_r x)$

如果选中专家 $i$,输出:

$y = g_i(x) \cdot \text{Expert}_i(x)$

关键工程挑战是负载均衡。如果路由不均匀,一些专家拥挤、一些空闲,GPU 利用率低。Switch 加辅助损失:

$\mathcal{L}_{\text{aux}} = \alpha \cdot N \sum_{i=1}^N f_i \cdot P_i$

其中 $f_i$ 是路由到专家 $i$ 的 token 比例,$P_i$ 是所有 token 对专家 $i$ 的平均 gate 概率。最优时 $f_i = P_i = 1/N$,$\mathcal{L}_{\text{aux}} = 1$。

Switch Transformer 训练了 1.6 万亿参数模型——首个公开的"万亿参数 LLM"。当时震撼业界。但效果相对同算力稠密模型优势不大,社区对"MoE 是否值得"仍有疑问。

Mixtral 8x7B:MoE 真正证明自己

Mistral AI 2023 年底发布 Mixtral 8x7B,改变了 MoE 的地位:

  • - 8 个专家,每个类似 7B 参数
  • - Top-2 路由(而非 Switch 的 Top-1)
  • - 总参数 47B(attention 共享),激活参数约 13B
  • - 能力接近 LLaMA-2 70B(稠密),推理速度像 13B
flowchart TB X[Token x] --> R[Router: softmax W_r x] R -->|top-2 gate| G["g_i=0.6
g_j=0.4"] G --> E1[Expert 1 FFN] G --> E2[Expert 2 FFN] E1 -->|w=0.6| SUM((+)) E2 -->|w=0.4| SUM SUM --> Y[Output] E3[Expert 3 FFN] -.-> SUM E4[Expert 4 FFN] -.-> SUM E5[...] -.-> SUM

实测 Mixtral 在 MMLU、数学、代码各种基准上和 LLaMA-2 70B 持平,但推理快很多。"同等能力下 MoE 便宜"第一次被清楚证明。

MoE 的工程难点

MoE 好看难做,工程挑战:

  1. 显存占用:推理时所有专家都得加载(不知道哪个 token 路由到哪)。Mixtral 8x7B 推理要 80GB+ 显存,不是 13B 的错觉
  2. 负载不均:即使有辅助损失,还是有些专家过载。设"容量因子" 1.25(允许 25% 超额)减少 drop
  3. 小 batch 低效:batch 小时,每个专家只接收少数 token,GPU 利用率低。MoE 更适合大 batch
  4. All-to-all 通信:分布式 MoE 要把 token 按路由发给不同卡——每层两次 all-to-all,通信开销大

后续演进

DeepSeek-V2/V3(2024)推动 MoE 走向极端:

  • 细粒度专家:从 8 个大专家变成 256 个小专家,组合更多样
  • 共享专家:1-2 个所有 token 都过的专家(承担通用知识),其他专家走路由
  • 多 token 预测、MLA 注意力压缩、FP8 训练配合 MoE 进一步提升效率

Qwen3-Next、Kimi K2、GLM-4.5、LLaMA 4 都是细粒度 MoE 路线。2025 年末所有开源前沿模型几乎都是 MoE。

为什么必读:MoE 是大模型规模化的核心技术。读完 Switch + Mixtral 两篇,你会理解"稀疏激活"的工程美——一个点子(只算部分参数)改变了 2024-2026 开源 LLM 的整体格局。

RLHF、DPO、CoT、DeepSeek-V3——让 LLM 从"会说话"到"会做事"的 4 篇关键论文。

P14InstructGPT / RLHF:对齐的起点

对应主书:第 3 章 4.5.3 微调与对齐 · 第 4 章 5.1.1

Training language models to follow instructions with human feedback
作者Long Ouyang, Jeff Wu, Xu Jiang, ... 等 20+ 人
机构OpenAI
发表NeurIPS 2022 · arXiv:2203.02155 · 被引 12,000+

问题:预训练模型和"有用助手"的鸿沟

GPT-3 预训练目标是"预测下一个 token"——模型会续写,但不一定回答。给它 "How do I make a cake?" 它可能续成 "? This is a common question..." 而不是真的教你。

InstructGPT 解决的问题:让预训练模型学会"按人类期望"响应。这是 ChatGPT 的直接前身,整个对齐领域的开山之作。

三阶段流程

flowchart LR PT[Pretrained GPT-3] --> SFT[Stage 1 SFT
人工指令-回答对] SFT --> RM[Stage 2 RM
人工偏好数据] RM --> PPO[Stage 3 PPO
RL 优化] PPO --> FINAL[InstructGPT]

Stage 1: SFT(Supervised Fine-Tuning)

人工写 13K 个 "prompt + 理想回答"对,用这个在 GPT-3 上做常规监督学习。目标:模型初步学会"按指令回应"的格式。

Stage 2: 奖励模型(Reward Model)

对同一个 prompt,让 SFT 模型生成多个答案,人工对它们排序。用偏好数据训练一个打分模型 $r_\phi(x, y)$:

$\mathcal{L}_{\text{RM}} = -\log \sigma(r_\phi(x, y_{\text{win}}) - r_\phi(x, y_{\text{lose}}))$

这是 Bradley-Terry 偏好模型的极大似然——RM 学着给"人类偏好的答案"打更高分。

Stage 3: PPO 强化学习

把 RM 当"奖励函数",用 PPO 训练策略模型。目标:生成 RM 打分高的回答。加 KL 惩罚防止策略漂离 SFT 模型(避免 reward hacking):

$\mathcal{L}_{\text{PPO}} = -\mathbb{E}[r_\phi(x, y) - \beta \cdot D_{\text{KL}}(\pi_\theta(y|x) \| \pi_{\text{SFT}}(y|x))]$

训练一次需要同时装 4 个模型:policy(要训的)、reference(冻结 SFT,算 KL)、reward model、critic(价值函数)。显存吃紧、训练不稳。

效果

InstructGPT 1.3B 参数,人类偏好评估上超过 GPT-3 175B 原版。规模小 100×,效果更好——对齐的价值第一次被量化证明。

几个月后 ChatGPT 发布,本质是 InstructGPT 的产品化版本。

InstructGPT 的遗产

  1. 三阶段流程(SFT → RM → RLHF)成为对齐标准方法,被 Claude、Gemini、LLaMA、Qwen 全部沿用
  2. 偏好数据成为新的"金矿"——标注成本高但价值大
  3. KL 惩罚作为"对齐税"的标准做法
  4. Bradley-Terry 偏好模型被广泛采用

批判视角

  • - Reward hacking:policy 学到"骗 RM"而非真正改进
  • - RM 的偏见:标注人的偏好不代表所有用户
  • - "对齐税":RLHF 后某些能力退化(特别是真实能力 benchmark)
  • - 工程复杂:4 个模型、PPO 难调

这些问题激发了 2023 年之后的各种替代——DPO(下一篇)、KTO、ORPO、RLAIF 等,都试图简化或改进 RLHF。

为什么必读:ChatGPT 为什么能是"助手"而不是续写机?答案在这篇。RLHF 三阶段的细节必须读原文才能真正理解。

P15DPO:把 RLHF 变成分类损失

对应主书:第 3 章 4.5.3

Direct Preference Optimization: Your Language Model is Secretly a Reward Model
作者Rafael Rafailov, Archit Sharma, Eric Mitchell, Stefano Ermon, Christopher D. Manning, Chelsea Finn
机构Stanford University
发表NeurIPS 2023(Outstanding Paper Award)· arXiv:2305.18290

问题:RLHF 的工程复杂度过高

RLHF 要训两个模型(RM + policy)、跑 PPO 强化学习、同时装 4 份模型、调一堆超参数、还担心 reward hacking。一套完整 RLHF 流水线对中小团队几乎不可行。

DPO 的贡献是:整个流程可以化简成一个监督学习损失。

核心数学推导

RLHF 的目标是带 KL 约束的奖励最大化:

$\max_{\pi} \mathbb{E}_{x, y \sim \pi}[r(x,y)] - \beta D_{\text{KL}}(\pi(y|x) \| \pi_{\text{ref}}(y|x))$

这个问题有闭式最优解:

$\pi^*(y|x) = \frac{1}{Z(x)} \pi_{\text{ref}}(y|x) \exp\left(\frac{1}{\beta} r(x, y)\right)$

$Z(x)$ 是归一化常数。反解出 $r$:

$r(x, y) = \beta \log \frac{\pi^*(y|x)}{\pi_{\text{ref}}(y|x)} + \beta \log Z(x)$

关键观察:这个 $r$ 代入 Bradley-Terry 损失时,$\beta \log Z(x)$ 会抵消(因为损失里是 $r(y_w) - r(y_l)$,两个 $\log Z(x)$ 相减为 0)。得到:

$\mathcal{L}_{\text{DPO}} = -\log \sigma\left( \beta \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} \right)$

这是普通的二元分类损失。

结果:不再需要 RM、不再需要 RL

  • - 不训奖励模型
  • - 不用 PPO 和 critic
  • - 只装 2 个模型(policy + 冻结的 reference)
  • - 标准 SGD,不会 reward hacking

实现

def dpo_loss(policy, ref, batch, beta=0.1):
    """batch 包含 (prompt, chosen, rejected) 三元组"""
    # 算 policy 和 reference 对 chosen / rejected 的对数似然
    pi_logps_chosen = compute_logps(policy,  batch.chosen, batch.prompt)
    pi_logps_reject = compute_logps(policy,  batch.rejected, batch.prompt)
    ref_logps_chosen = compute_logps(ref,    batch.chosen, batch.prompt)
    ref_logps_reject = compute_logps(ref,    batch.rejected, batch.prompt)

    pi_logratio  = pi_logps_chosen  - pi_logps_reject
    ref_logratio = ref_logps_chosen - ref_logps_reject

    loss = -F.logsigmoid(beta * (pi_logratio - ref_logratio)).mean()
    return loss

整个 DPO 训练看起来就像 fine-tune——比 PPO 简单一个数量级。

影响

DPO 在 2023 年底、2024 年快速成为主流对齐方法。Zephyr(HuggingFace)、Tulu(AI2)、Nous-Hermes、Qwen、DeepSeek 多家开源模型都用 DPO 或其变种。

但 DPO 不是银弹:

  • - 偏好数据噪声敏感——IPO(2024)改进这一点
  • - 仅需成对数据——KTO(2024)让单点反馈也能用
  • - 合并 SFT 和 DPO 的变种——ORPO、SimPO 继续演进

2024-2025 对齐领域百花齐放,但底层理论都可追溯到 DPO 这篇。最近的推理模型训练(DeepSeek R1 的 GRPO)又回到了 RL 路线——对齐技术并未"定型",仍在演化。

为什么必读:"PPO + RM + KL = 一个分类损失"这个数学 reframing 是对齐领域最漂亮的结果之一。读完你会理解为什么大家都在"摆脱 RL"——能用监督学习解决的问题,何必用 RL。

P16Chain-of-Thought:让模型"想"一步再答

对应主书:第 4 章 5.11.1 · 5.1.1 推理时计算

Chain-of-Thought Prompting Elicits Reasoning in Large Language Models
作者Jason Wei, Xuezhi Wang, Dale Schuurmans, Maarten Bosma, Brian Ichter, Fei Xia, Ed Chi, Quoc Le, Denny Zhou
机构Google Research
发表NeurIPS 2022 · arXiv:2201.11903 · 被引 10,000+

问题:LLM 在多步推理上错得离谱

2021 年 GPT-3 在很多任务上强,但在数学题(小学应用题级别)错误率高。典型题:

Q: Roger has 5 tennis balls. He buys 2 more cans of tennis balls.
   Each can has 3 tennis balls. How many tennis balls does he have now?
模型直接回答: 27  (错)

模型有"计算能力"(能做 5 + 2×3 = 11),但直接问答时跳步骤,错。

核心发现:让模型"写出思考过程"

Wei 等人做的事情简单得荒谬:把 few-shot 例子从"问题 + 答案"改成"问题 + 推理过程 + 答案"。

Q: Roger has 5 tennis balls. He buys 2 more cans. Each can has 3.
   How many balls does he have now?
A: Roger started with 5 balls. 2 cans of 3 balls each is 6 balls.
   5 + 6 = 11. The answer is 11.

Q: The cafeteria had 23 apples. If they used 20 to make lunch and
   bought 6 more, how many apples do they have?
A: [让模型续写]

模型现在会输出类似 "The cafeteria had 23 apples. They used 20, so they had 23 - 20 = 3. Then they bought 6 more, so 3 + 6 = 9. The answer is 9." —— 正确。

涌现现象

CoT 有个关键特性:在小模型上没有效果甚至更差,在大模型上突然大幅提升。Wei 等人的实验数据(GSM8K 数学基准):

模型直接答CoT
GPT-3 0.4B4.5%5.6%
GPT-3 5.7B5.1%11.6%
GPT-3 175B15.6%46.9% (3×)
PaLM 540B17.9%56.9% (3×)

这种"模型大到一定程度后能力突然出现"的现象,被称为"涌现"。CoT 是最早明确记录的涌现能力之一。

为什么 CoT 有效:算力视角

LLM 的每个 forward pass 计算量固定($O(N)$,N 是参数量)。直接答 = 一次 forward 完成多步推理,算力不够。CoT = 每步推理一次 forward,总算力随推理长度线性增加——相当于用 token 长度换算力。

这个视角打开了"推理时计算"(test-time compute)的大门——OpenAI o1、DeepSeek R1 的长推理本质就是 CoT 的极致化。

Zero-shot CoT

Kojima 等 2022 年("Let's think step by step")发现:连 few-shot 例子都不用,只在 prompt 末尾加"Let's think step by step",大模型也会自动做 CoT。这个简单得不真实的 prompt 一度成为圈内 meme。

CoT 的后续演化

  • Self-Consistency(2022):CoT + 多次采样 + 投票——纠错
  • Tree of Thoughts(2023):不是线性推理,而是搜索多路径推理树
  • ReAct(2022):CoT 交替"思考"和"工具调用"——Agent 的基础
  • o1 / R1(2024):通过 RL 把 CoT 长度推到极致,生成数万 token 推理再给答案

影响

CoT 是 LLM 能力的一个关键解锁——让"推理任务"从"几乎做不了"变成"能做"。今天所有数学、编程、推理基准都默认用 CoT 评估。现代 LLM 的"思考模式"(DeepSeek V3.1、Qwen3、Claude thinking、Gemini Deep Think)都是 CoT 的产品化。

为什么必读:CoT 是理解"推理时 scaling"的起点。从 CoT 到 Self-Consistency 到 ToT 到 o1 的演化,是 2022-2026 LLM 能力提升的另一条主线(和 scaling law 并行)。

P17DeepSeek-V3:开源 MoE 的集大成

对应主书:第 4 章 5.4.1 MLA · 5.6 MoE · 5.9.2 FP8

DeepSeek-V3 Technical Report
作者DeepSeek-AI team
发表arXiv:2412.19437(2024.12)
后续DeepSeek-V3.1(2025.8 混合思考)· V3.2(2025.10 DSA 稀疏注意力)

为什么选这篇作为压轴

DeepSeek-V3 不是第一个 MoE、不是第一个用 FP8 训练、不是第一个做混合推理的模型——但它是第一个把所有这些技术集成到一起做到极致的开源模型。671B 总参数、37B 激活、训练只用 2.788M H800 GPU 小时(约 560 万美元),达到 GPT-4o 级别能力。

这是"工程集成"胜过"单点突破"的范本,也是开源赶上闭源的关键节点。

架构总览

flowchart TB IN[Input] --> EMB[Embedding] EMB --> L1[Layer 1 to 61] L1 --> ATT["MLA Attention
低秩 KV 压缩"] ATT --> MOE["Fine-grained MoE

top-8 激活"] MOE --> MTP["Multi-Token Prediction

创新 1: MLA(Multi-head Latent Attention)

主书 5.4.1 详讲过。核心:用低秩分解压缩 KV Cache。

$\mathbf{c}_{\text{kv}} = W_{DKV} \mathbf{x}, \quad \mathbf{k} = W_{UK} \mathbf{c}_{\text{kv}}, \quad \mathbf{v} = W_{UV} \mathbf{c}_{\text{kv}}$

$d_c \ll d_{model}$,KV Cache 只存 $\mathbf{c}_{\text{kv}}$。推理时 $W_{UK}$ 可合并进 Q 投影,零额外运算。

对比等效 MHA,KV Cache 缩小 90%+,是长上下文推理的核心优势。

创新 2: 细粒度 MoE + 共享专家

V3 每层有 256 个 routed 专家 + 1 个 shared 专家,top-8 激活:

  • - 细粒度:256 小专家的组合数 $C_{256}^8 = 4 \times 10^{14}$,远超 Mixtral 的 28 种组合
  • - 共享专家:承担通用知识,所有 token 都过
  • - 无辅助损失的负载均衡:V3 创新地在路由分数上加可学习的 bias,动态调整让每个专家负载均匀,不再需要传统的辅助损失

创新 3: Multi-Token Prediction (MTP)

训练时不只预测下一个 token,也预测接下来 $D$ 个 token(V3 用 $D = 1$ 即额外预测 1 个)。每个预测位置一个专门的 MTP 模块(带独立 Transformer 层)。好处:

  • - 训练信号密度更高,同样 token 数学到更多
  • - 推理时可直接用 MTP 模块做推测解码,省一个独立 draft model

创新 4: FP8 训练的大规模工程化

V3 是首个 671B 级别完全 FP8 训练的模型。核心技术:

  1. Fine-grained quantization:Per-tensor 不够,V3 用 per-block(1×128 个元素一组 scale)
  2. 高精度累加:FP8 × FP8 → BF16 累加,减少数值误差
  3. 选择性 FP8:attention 的 softmax、归一化等敏感操作保留 BF16
  4. Pipeline 优化:All-to-All 和计算重叠,跨节点通信隐藏

实测 FP8 相比 BF16,训练速度快 1.5-2×,最终 loss 几乎无损。

约 560 万美元训一个 GPT-4 级别模型——比闭源估计节省 10×+。这是 "DeepSeek 时刻"(2025.2)的经济基础——成本低到让全球震惊。

后续演进

  • DeepSeek-R1(2025.1):基于 V3,用纯 RL(GRPO 算法)训练出推理模型。无需人工标注 CoT 数据——纯 RL 涌现出推理能力
  • V3-0324(2025.3):后训练改进,吸收 R1 的 RL 经验,推理能力大幅提升
  • R1-0528(2025.5):R1 升级版,AIME 数学基准超过同期 GPT-4o
  • V3.1(2025.8):混合思考模式——一个模型支持 thinking / non-thinking 切换,合并 V3 和 R1
  • V3.2(2025.10):引入 DSA(DeepSeek Sparse Attention),长上下文推理成本再降 50%

意义

V3 证明了三件事:

  1. 开源能追上闭源:不是 6 个月,是能抗衡
  2. 架构创新的复利:MLA + MoE + MTP + FP8 叠加,整体效率远超单项
  3. 中国 AI 实验室的工程实力:V3 的训练效率和架构细节在国际上被全面认可

此后 2025 年开源生态迅速跟进——Qwen3、Kimi K2、GLM-4.5、LLaMA 4 都采用类似架构哲学(细粒度 MoE + 长上下文优化 + FP8 训练)。

批判视角

  • - 训练成本数字有争议——$560 万可能只是算 GPU 时间,不含数据、基础设施、人力
  • - "GPT-4 级别"在不同基准上表现参差——某些基准超过,某些仍差距明显
  • - MLA 的工程复杂度高,小团队未必能复现

但无论怎么质疑,V3 是当代开源 LLM 史上最重要的里程碑之一。

为什么压轴:V3 技术报告是 2024 年最值得读的 LLM 工程论文——它把前面 P06-P13 的所有技术(Transformer、MoE、LoRA 思想、FP8、推理优化)集大成,并展示"它们组合起来能做到什么"。读完这篇,前面 16 篇的相互关系会变得极其清晰。
