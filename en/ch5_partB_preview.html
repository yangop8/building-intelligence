P10FlashAttention: IO-Aware Exact Attention

Main text reference: Chapter 3, §4.7.3 Operator Optimization · Chapter 4, §4.2.2 / §5.8

FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness
AuthorsTri Dao, Daniel Y. Fu, Stefano Ermon, Atri Rudra, Christopher Ré
AffiliationStanford University
PublishedNeurIPS 2022 · arXiv:2205.14135 · 5,000+ citations
Follow-upFlashAttention-2 (2023), FlashAttention-3 (2024, FP8 + H100 optimizations)

The Problem: Attention's Real Bottleneck Is Not FLOPs

Self-attention has time complexity $O(L^2 d)$ for long sequences, often labeled the "compute bottleneck." But Tri Dao observed that on GPUs, attention is actually a memory-bandwidth bottleneck, not a compute bottleneck.

The numbers: an A100 GPU delivers 312 TFLOPs for FP16 GEMM, but its HBM bandwidth is only 1.5 TB/s. The ratio (arithmetic intensity) needs to reach ~208 FLOP/byte to saturate compute — and standard attention falls far below this. Most of the time, the GPU is stalled waiting for data to move from HBM.

The Problem with Standard Attention

A standard PyTorch implementation does four things:

  1. Compute $S = QK^\top \in \mathbb{R}^{L \times L}$, write it to HBM
  2. Read $S$ from HBM, compute $P = \text{softmax}(S)$, write it back to HBM
  3. Read $P$ from HBM, multiply by $V$ to get $O$, write it back to HBM
  4. (Possible dropout and masking operations add further read/write passes)

At $L = 4096$, the $S$ matrix alone is 64 MB. Each read/write is a 128 MB round-trip to HBM. As $L$ grows, this cost scales linearly while compute utilization stays extremely low.

Core Method: Tiling + Online Softmax

FlashAttention splits Q, K, V into tiles sized to fit in the GPU's on-chip SRAM. The outer loop iterates over Q tiles $Q_i$; the inner loop iterates over K, V tiles $(K_j, V_j)$. The full $S$ or $P$ matrix is never materialized — all intermediate results stay in SRAM.

flowchart TB HBM["HBM — large, slow"] -. "read Q_i, K_j, V_j" .-> SRAM SRAM["SRAM — small, fast
tile-level computation"] -. "write O_i" .-> HBM SRAM --> S["S_ij = Q_i K_j^T (in SRAM)"] S --> SM["online softmax: update m, l, O"] SM --> NEXT["next K_j, V_j tile"]

The Key Challenge: Online Softmax

Softmax requires normalization — $\text{softmax}(x_i) = \exp(x_i) / \sum_j \exp(x_j)$. It seems like you must see all $x_j$ before computing the denominator. FlashAttention uses an incremental update instead: each time a new $S_{ij}$ tile arrives, it maintains a running max and running sum, producing a result that is mathematically equivalent to the full softmax.

Say we have processed up to tile $j-1$, with state $(m_{j-1}, l_{j-1}, O_{j-1})$ — the current running max $m$, running denominator $l$, and current weighted output $O$. A new tile $(K_j, V_j)$ arrives:

Compute $S_{ij} = Q_i K_j^\top$. Let $\tilde{m}_j = \max(S_{ij})$. New max:

$m_j = \max(m_{j-1}, \tilde{m}_j)$

Rescale both the new and old exponential terms using the new max:

$l_j = e^{m_{j-1} - m_j} \cdot l_{j-1} + \sum_k e^{S_{ij,k} - m_j}$

New output (old contribution is rescaled to the new max):

$O_j = \frac{1}{l_j}\left[ l_{j-1} e^{m_{j-1} - m_j} O_{j-1} + \sum_k e^{S_{ij,k} - m_j} V_{j,k} \right]$

After scanning all K/V tiles, $O_J$ is exactly equivalent to the full softmax result. It is numerically identical to standard attention — this is not an approximation.

Pseudocode

def flash_attention(Q, K, V, block_size=64):
    """Q, K, V: [L, d].  Returns O: [L, d], never materializing the LxL matrix S"""
    L, d = Q.shape
    O = torch.zeros_like(Q)
    m = torch.full((L, 1), -float('inf'))  # running max per row
    l = torch.zeros((L, 1))                 # running sum per row

    for j in range(0, L, block_size):
        K_j = K[j:j+block_size]              # load tile into SRAM
        V_j = V[j:j+block_size]
        S_ij = Q @ K_j.T / math.sqrt(d)      # computed in SRAM, [L, block]

        # Causal mask (if needed)
        mask = torch.arange(j, j+K_j.shape[0]).unsqueeze(0) > \
               torch.arange(L).unsqueeze(1)
        S_ij = S_ij.masked_fill(mask, -float('inf'))

        m_new = torch.maximum(m, S_ij.max(dim=-1, keepdim=True).values)
        P_ij = torch.exp(S_ij - m_new)
        l_new = torch.exp(m - m_new) * l + P_ij.sum(dim=-1, keepdim=True)

        # Rescale old O to new max, add current tile's contribution
        O = (l / l_new) * torch.exp(m - m_new) * O + (P_ij @ V_j) / l_new
        m, l = m_new, l_new

    return O

The actual CUDA implementation involves substantial engineering — warp-level shuffling, register reuse, pipelining — but the core algorithm is exactly what is shown above.

Performance Gains

  • - HBM accesses: reduced from $O(L^2 d)$ to $O(L^2 d^2 / M)$ ($M$ = SRAM size) — several orders of magnitude fewer
  • - Speed: 2–4× faster than standard attention on A100, with greater gains at longer sequences
  • - Memory: intermediate $S$ and $P$ are never materialized, reducing peak memory from $O(L^2)$ to $O(L)$ — this is what makes 128K+ context windows feasible

FlashAttention-2 and -3

FlashAttention-2 (2023): Refined outer loop ordering, reduced non-matmul operations, improved warp partitioning. Another 2× speedup, totaling 4–8× over standard attention.

FlashAttention-3 (2024): Targets the H100 Hopper architecture, leveraging FP8 Tensor Cores, asynchronous execution, and more granular pipelining. Achieves ~75% of peak throughput on H100.

These three generations represent the most important attention optimization work from 2022 to 2024, and directly influenced the implementations of PyTorch SDPA, vLLM, xFormers, and TensorRT-LLM. Modern LLM training and inference almost universally defaults to FlashAttention.

Key Insight

FlashAttention's impact extends well beyond attention itself — it is the exemplar of IO-aware algorithm design. Many GPU kernels can be restructured similarly: keep intermediate results in SRAM, and use mathematically equivalent incremental forms to avoid large intermediate buffers. FlashDecoding and RingAttention are direct successors of this idea.

Why You Must Read This: The online softmax algorithm is something you need to work through by hand to truly understand. Once you grasp it, you will recognize the same technique appearing in many other places — PagedAttention, StreamingLLM, and FlashDecoding all rely on the same mathematical structure.

P11LoRA / QLoRA: Making Fine-Tuning Affordable

Main text reference: Chapter 3, §4.5.3 Fine-Tuning · Chapter 4, §4.4.1 Low-Rank Ideas

LoRA: Low-Rank Adaptation of Large Language Models
AuthorsEdward Hu, Yelong Shen, Phillip Wallis, Zeyuan Allen-Zhu, Yuanzhi Li, Shean Wang, Lu Wang, Weizhu Chen
AffiliationMicrosoft
PublishedICLR 2022 · arXiv:2106.09685 · 8,000+ citations
QLoRA: Efficient Finetuning of Quantized LLMs
AuthorsTim Dettmers et al. (University of Washington)
PublishedNeurIPS 2023 · arXiv:2305.14314

The Problem: Full Fine-Tuning a Large Model Is Too Expensive

Fine-tuning a 70B model requires storing: parameters at 140 GB (FP16) + gradients at 140 GB + Adam optimizer state at 560 GB ≈ 840 GB total, plus activations. Even a multi-GPU single node isn't enough — you need ZeRO-3-level distributed training.

Yet many use cases only need light customization — adding a domain-specific style, or making the model follow a particular output format. Is it really necessary to update every parameter?

LoRA's Core Hypothesis

Hu et al. hypothesize that when a model adapts to a downstream task, the weight update $\Delta W$ has a low-rank structure.

That is, $\Delta W \in \mathbb{R}^{d \times k}$ can be decomposed as $\Delta W = BA$, where $B \in \mathbb{R}^{d \times r}, A \in \mathbb{R}^{r \times k}$, and $r \ll \min(d, k)$.

During training, the original weights $W_0$ are frozen; only $A$ and $B$ are trained:

$W = W_0 + \Delta W = W_0 + BA, \quad h = W_0 x + BA x$
flowchart LR X[x] --> W0["W_0 (frozen, d x k)"] W0 --> Y1[W_0 x] X --> A["A (r x k, trainable)"] A --> B["B (d x r, trainable)"] B --> Y2["BA x"] Y1 --> ADD((+)) Y2 --> ADD ADD --> H[h]

Initialization: $A$ is initialized with Gaussian noise; $B$ is initialized to zero — so the starting point $BA = 0$, matching full fine-tuning's initial state.

Parameter Count Comparison

For GPT-3 175B, consider a layer where $W \in \mathbb{R}^{12288 \times 12288}$, with 150M parameters. With $r = 8$: LoRA introduces $A \in \mathbb{R}^{8 \times 12288}$ and $B \in \mathbb{R}^{12288 \times 8}$, for a total of 200K parameters — a 750× reduction.

Across the full model, LoRA trains approximately 0.01% of parameters. Memory requirements are dramatically reduced: frozen weights require no Adam state or gradient storage — only $A$ and $B$ need gradients — yielding 10–20× memory savings over full fine-tuning.

Implementation (PEFT)

from peft import LoraConfig, get_peft_model
from transformers import AutoModelForCausalLM

base = AutoModelForCausalLM.from_pretrained("meta-llama/Llama-3-8B")

config = LoraConfig(
    r=16,                          # rank
    lora_alpha=32,                 # scaling factor; effective W += (alpha/r) BA
    target_modules=["q_proj", "v_proj"],  # apply LoRA only to Q and V projections
    lora_dropout=0.05,
    task_type="CAUSAL_LM",
)
model = get_peft_model(base, config)
model.print_trainable_parameters()
# trainable params: 4,194,304 || all params: 8,034,455,552 || trainable%: 0.05%

Merging at Inference Time

At inference, $BA$ can be merged back into $W_0$: $W_{\text{new}} = W_0 + BA$, producing a standard weight matrix. Zero additional inference overhead — this is a major advantage over Adapters, which insert small modules between layers and increase latency at inference time.

QLoRA: Another 4× Memory Reduction

QLoRA (2023) quantizes the base model to 4-bit while keeping the LoRA parameters in FP16. Memory breakdown:

  • - Frozen weights: 140 GB (FP16) → 35 GB (INT4), 4× saving
  • - LoRA parameters and gradients: tens of MB, negligible
  • - Total: a 65B model fits on a single 48 GB GPU

QLoRA introduces several technical innovations:

  1. NF4 (NormalFloat 4-bit): a 4-bit data type designed for normally distributed weights, achieving higher precision than standard INT4
  2. Double quantization: quantizes the quantization scales themselves, further reducing memory
  3. Paged optimizers: uses NVIDIA Unified Memory to swap optimizer state to CPU when GPU memory is insufficient

QLoRA made "fine-tuning a 65B model on a consumer GPU" a reality — it was the inflection point that opened large-scale fine-tuning to the open-source community. Projects like Alpaca and Vicuna all relied on LoRA/QLoRA.

Variants

  • LoRA+: uses different learning rates for $A$ and $B$ (B should be larger), a small change that yields 1–2% quality improvement
  • DoRA (Weight-Decomposed LoRA): decomposes weights into magnitude and direction, adapting each separately
  • AdaLoRA: dynamically allocates rank per layer
  • rsLoRA: improved scaling rules to stabilize training at larger ranks

Impact

LoRA is the most important fine-tuning work since 2022. Nearly every open-source LLM fine-tune today uses LoRA or one of its variants — HuggingFace's PEFT library was built around it. LoRA also inspired architectural thinking: DeepSeek's MLA uses low-rank decomposition to compress the KV Cache (main text §4.4.1), a direct conceptual descendant.

Why You Must Read This: LoRA is the most important "lever" in the open-source community — it lets individuals and small teams participate in frontier LLM adaptation. QLoRA's NF4 design is worth reading carefully; it offers excellent intuition about data type design tailored to a distribution.

P12PagedAttention / vLLM: Treating KV Cache Like Virtual Memory

Main text reference: Chapter 4, §4.8.3

Efficient Memory Management for Large Language Model Serving with PagedAttention
AuthorsWoosuk Kwon, Zhuohan Li, Siyuan Zhuang, Ying Sheng, Lianmin Zheng, Cody Hao Yu, Joseph E. Gonzalez, Hao Zhang, Ion Stoica
AffiliationUC Berkeley, Stanford, UCSD, CMU
PublishedSOSP 2023 · arXiv:2309.06180

The Problem: Memory Fragmentation in KV Cache

LLM inference requires contiguous KV Cache space per request. A naive implementation pre-allocates by max_seq_len: a request reserving space for 4096 tokens might only use 500, wasting 87%. There are four categories of waste at the system level:

  1. Reservation waste: space reserved for sequences that might grow longer
  2. Internal fragmentation: each request has a different actual length, so reserved space goes unused
  3. External fragmentation: small gaps left by completed short requests can't fit new larger ones
  4. No sharing: requests with identical prefixes each store their own copy (pure waste)

Core Method: Bringing OS Virtual Memory to KV Cache

PagedAttention directly mirrors OS virtual memory:

  • GPU memory is divided into fixed-size blocks (KV data for 16 tokens per block)
  • Each request maintains a block table mapping logical token positions to physical blocks
  • Blocks are allocated on demand and grow dynamically with token generation — no pre-reservation
  • The attention kernel accesses KV data indirectly via the block table
flowchart TB subgraph GPU["GPU Memory Pool (blocks of 16 tokens KV)"] B0[Block 0: free] B3[Block 3: R1 tokens 16-19] B5[Block 5: R2 tokens 0-14] B7[Block 7: R1 tokens 0-15] B8[Block 8: free] end R1["Request 1 (20 tokens)
block_table: [7, 3]"] --> B7 R1 --> B3 R2["Request 2 (15 tokens)
block_table: [5]"] --> B5

Modified Attention Kernel

The standard kernel assumes K and V are contiguous. PagedAttention's kernel uses the block table for indirect lookup:

// Simplified pseudocode
for each query q_i:
    output = 0
    max_score = -inf; sum_exp = 0
    for each logical token j in query's block_table:
        block_id  = block_table[j / BLOCK_SIZE]
        inner_idx = j % BLOCK_SIZE
        k_j = K_pool[block_id * BLOCK_SIZE + inner_idx]
        v_j = V_pool[block_id * BLOCK_SIZE + inner_idx]
        score = q_i @ k_j / sqrt(d)
        // online softmax update (same as FlashAttention)
        ...

The one extra indirection (block_table[...]) adds a small overhead, but the memory savings far outweigh this cost.

Sharing and Copy-on-Write

When multiple requests share the same prefix (e.g., a system prompt or few-shot examples), their block tables can point to the same physical blocks. A copy is made only when divergence occurs — when a request starts generating different tokens. This makes prefix caching a natural optimization, directly benefiting high-concurrency LLM serving.

Beam search and parallel sampling benefit similarly: multiple candidates share the prefix KV Cache.

Measured Gains

Paper results (LLaMA-13B):

  • - Memory utilization: ~40% → ~96%
  • - Concurrent requests per GPU: 4–10×
  • - Throughput: 2–4× improvement

vLLM

The paper authors simultaneously open-sourced vLLM — a production-grade inference engine combining PagedAttention, continuous batching, and speculative decoding. Since its mid-2023 release:

  • - It became the de facto standard for open-source LLM serving
  • - In 2024, it ranked alongside SGLang as one of the top two inference frameworks by OpenRank
  • - A major v1 architectural overhaul shipped in 2025
  • - Adopted by virtually every major cloud provider and LLM service

vLLM's success exceeds the citation value of the PagedAttention paper itself — it is a textbook example of "research result becoming industrial infrastructure."

Key Insight

This is a classic example of cross-domain borrowing. The virtual memory concept was mature by 1961 (the Atlas computer) — over 60 years later, it was transplanted into LLM inference to solve a new problem. A programmer who studied operating systems can read this paper and understand it almost instantly. That is the power of foundational knowledge.

Why You Must Read This: Without this paper, you won't understand why vLLM can be 5–10× faster than a naive implementation. For systems engineers, it is also an excellent model of cross-domain knowledge transfer.

P13Switch Transformer & Mixtral: Making MoE Practical

Main text reference: Chapter 4, §5.6

Switch Transformers: Scaling to Trillion Parameter Models with Simple and Efficient Sparsity
AuthorsWilliam Fedus, Barret Zoph, Noam Shazeer (Google)
PublishedJMLR 2022 · arXiv:2101.03961
Mixtral of Experts
AuthorsMistral AI team
PublishedarXiv:2401.04088 (2024)

MoE Is an Old Idea

The MoE concept traces back to Jacobs et al.'s 1991 "Adaptive Mixtures of Local Experts." Shazeer et al. first applied it to deep learning in 2017 (Sparsely-Gated MoE). But it was Switch Transformer and Mixtral that truly demonstrated MoE's practical value.

Switch Transformer: Top-1 Routing

Switch's core simplification: each token is routed to exactly 1 expert (prior work typically used top-2). This drastically simplifies the gating mechanism and reduces communication complexity.

$\text{expert}(x) = \arg\max_i g_i(x), \quad g(x) = \text{softmax}(W_r x)$

If expert $i$ is selected, the output is:

$y = g_i(x) \cdot \text{Expert}_i(x)$

The key engineering challenge is load balancing. If routing is uneven — some experts overloaded, others idle — GPU utilization suffers. Switch adds an auxiliary loss:

$\mathcal{L}_{\text{aux}} = \alpha \cdot N \sum_{i=1}^N f_i \cdot P_i$

Here $f_i$ is the fraction of tokens routed to expert $i$, and $P_i$ is the average gate probability for expert $i$ across all tokens. At the optimum, $f_i = P_i = 1/N$ and $\mathcal{L}_{\text{aux}} = 1$.

Switch Transformer trained a 1.6 trillion parameter model — the first publicly disclosed "trillion-parameter LLM." It shocked the field at the time. However, its quality advantage over dense models of the same compute budget was modest, and the community remained skeptical about whether MoE was truly worthwhile.

Mixtral 8x7B: MoE Proves Itself

Mistral AI's late-2023 release of Mixtral 8x7B changed the status of MoE entirely:

  • - 8 experts, each roughly the size of a 7B parameter model
  • - Top-2 routing (rather than Switch's Top-1)
  • - Total parameters: 47B (with shared attention); activated parameters per token: ~13B
  • - Quality comparable to LLaMA-2 70B (dense), at the inference speed of a 13B model
flowchart TB X[Token x] --> R[Router: softmax W_r x] R -->|top-2 gate| G["g_i=0.6
g_j=0.4"] G --> E1[Expert 1 FFN] G --> E2[Expert 2 FFN] E1 -->|w=0.6| SUM((+)) E2 -->|w=0.4| SUM SUM --> Y[Output] E3[Expert 3 FFN] -.-> SUM E4[Expert 4 FFN] -.-> SUM E5[...] -.-> SUM

Mixtral matched LLaMA-2 70B on MMLU, math, and coding benchmarks, while being significantly faster to run. This was the first clear demonstration that "MoE achieves the same capability at lower cost."

Engineering Challenges of MoE

MoE looks appealing on paper but is difficult in practice. Key engineering challenges:

  1. Memory footprint: all experts must be loaded at inference time (you don't know routing in advance). Mixtral 8x7B requires 80 GB+ of GPU memory — it is not as lean as the "13B activated" framing suggests
  2. Load imbalance: even with auxiliary loss, some experts still get overloaded. A "capacity factor" of 1.25 (allowing 25% overflow) reduces token dropping
  3. Small-batch inefficiency: with small batches, each expert receives very few tokens, and GPU utilization is low. MoE works best at large batch sizes
  4. All-to-all communication: in distributed MoE, tokens must be routed to different devices per layer — two all-to-all operations per layer, with significant communication overhead

Subsequent Evolution

DeepSeek-V2/V3 (2024) pushed MoE further:

  • Fine-grained experts: moved from 8 large experts to 256 small experts, enabling far more diverse combinations
  • Shared experts: 1–2 experts that every token passes through (handling general knowledge), with other experts gated
  • Multi-token prediction, MLA attention compression, and FP8 training combined with MoE to further improve efficiency

Qwen3-Next, Kimi K2, GLM-4.5, and LLaMA 4 all follow the fine-grained MoE path. By late 2025, nearly every frontier open-source model is a MoE.

Why You Must Read This: MoE is the core technology enabling large-scale model scaling. Reading both Switch and Mixtral gives you a full picture of "sparse activation" as an engineering principle — one idea (compute only part of the parameters) reshaped the entire open-source LLM landscape from 2024 to 2026.

P14InstructGPT / RLHF: The Starting Point of Alignment

Main text reference: Chapter 3, §4.5.3 Fine-Tuning & Alignment · Chapter 4, §4.1.1

Training language models to follow instructions with human feedback
AuthorsLong Ouyang, Jeff Wu, Xu Jiang, et al. (20+ authors)
AffiliationOpenAI
PublishedNeurIPS 2022 · arXiv:2203.02155 · 12,000+ citations

The Problem: The Gap Between a Pretrained Model and a "Helpful Assistant"

GPT-3's pretraining objective is "predict the next token" — the model can continue text, but doesn't necessarily answer. Ask it "How do I make a cake?" and it might continue with "? This is a common question..." rather than actually teaching you how to bake.

InstructGPT addresses the problem of teaching a pretrained model to respond "as humans expect." It is the direct predecessor of ChatGPT, and the foundational work of the entire alignment field.

Three-Stage Pipeline

flowchart LR PT[Pretrained GPT-3] --> SFT[Stage 1 SFT
human instruction–response pairs] SFT --> RM[Stage 2 RM
human preference data] RM --> PPO[Stage 3 PPO
RL optimization] PPO --> FINAL[InstructGPT]

Stage 1: SFT (Supervised Fine-Tuning)

Human annotators write 13K "prompt + ideal response" pairs, and GPT-3 is fine-tuned on these with standard supervised learning. Goal: teach the model the basic format of "respond to instructions."

Stage 2: Reward Model

For each prompt, the SFT model generates multiple candidate responses, and human annotators rank them. A scoring model $r_\phi(x, y)$ is trained on this preference data:

$\mathcal{L}_{\text{RM}} = -\log \sigma(r_\phi(x, y_{\text{win}}) - r_\phi(x, y_{\text{lose}}))$

This is maximum likelihood estimation under the Bradley-Terry preference model — the RM learns to assign higher scores to responses humans prefer.

Stage 3: PPO Reinforcement Learning

The reward model serves as the reward function; PPO trains the policy model to generate responses with higher RM scores. A KL penalty prevents the policy from drifting too far from the SFT model (preventing reward hacking):

$\mathcal{L}_{\text{PPO}} = -\mathbb{E}[r_\phi(x, y) - \beta \cdot D_{\text{KL}}(\pi_\theta(y|x) \| \pi_{\text{SFT}}(y|x))]$

Training requires four models loaded simultaneously: the policy (being trained), the reference (frozen SFT, for KL computation), the reward model, and the critic (value function). This is memory-intensive and training is unstable.

Results

InstructGPT at 1.3B parameters outperforms GPT-3 at 175B on human preference evaluations. 100× smaller, yet better — the value of alignment was quantified for the first time.

ChatGPT launched a few months later, essentially the productized version of InstructGPT.

InstructGPT's Legacy

  1. The three-stage pipeline (SFT → RM → RLHF) became the standard alignment recipe, adopted by Claude, Gemini, LLaMA, Qwen, and virtually every major LLM
  2. Preference data became the new "gold" — expensive to label, but immensely valuable
  3. KL penalty as the standard "alignment tax" mechanism
  4. The Bradley-Terry preference model was widely adopted

Critical Perspectives

  • - Reward hacking: the policy can learn to fool the RM rather than genuinely improve
  • - RM bias: annotator preferences don't represent all users
  • - "Alignment tax": RLHF can cause regression on certain capabilities (particularly on factual benchmarks)
  • - Engineering complexity: four models, unstable PPO training

These problems motivated a wave of alternatives after 2023 — DPO (next paper), KTO, ORPO, RLAIF, and others — all attempting to simplify or improve upon RLHF.

Why You Must Read This: Why is ChatGPT an "assistant" rather than a text completion engine? The answer is in this paper. The details of the three-stage RLHF pipeline must be read in the original to be truly understood.

P15DPO: Turning RLHF into a Classification Loss

Main text reference: Chapter 3, §4.5.3

Direct Preference Optimization: Your Language Model is Secretly a Reward Model
AuthorsRafael Rafailov, Archit Sharma, Eric Mitchell, Stefano Ermon, Christopher D. Manning, Chelsea Finn
AffiliationStanford University
PublishedNeurIPS 2023 (Outstanding Paper Award) · arXiv:2305.18290

The Problem: RLHF's Engineering Complexity Is Too High

RLHF requires training two models (RM + policy), running PPO reinforcement learning, loading four model copies simultaneously, tuning a large number of hyperparameters, and worrying about reward hacking. A full RLHF pipeline is practically infeasible for most small and medium teams.

DPO's contribution is showing that the entire process can be reduced to a single supervised learning loss.

Core Mathematical Derivation

RLHF's objective is reward maximization with a KL constraint:

$\max_{\pi} \mathbb{E}_{x, y \sim \pi}[r(x,y)] - \beta D_{\text{KL}}(\pi(y|x) \| \pi_{\text{ref}}(y|x))$

This problem has a closed-form optimal solution:

$\pi^*(y|x) = \frac{1}{Z(x)} \pi_{\text{ref}}(y|x) \exp\left(\frac{1}{\beta} r(x, y)\right)$

$Z(x)$ is the normalization constant. Inverting for $r$:

$r(x, y) = \beta \log \frac{\pi^*(y|x)}{\pi_{\text{ref}}(y|x)} + \beta \log Z(x)$

Key observation: when this $r$ is substituted into the Bradley-Terry loss, $\beta \log Z(x)$ cancels out (since the loss takes $r(y_w) - r(y_l)$, the two $\log Z(x)$ terms subtract to zero). This yields:

$\mathcal{L}_{\text{DPO}} = -\log \sigma\left( \beta \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)} \right)$

This is an ordinary binary classification loss.

Result: No RM Needed, No RL Needed

  • - No reward model to train
  • - No PPO, no critic
  • - Only 2 models loaded (policy + frozen reference)
  • - Standard SGD, no reward hacking

Implementation

def dpo_loss(policy, ref, batch, beta=0.1):
    """batch contains (prompt, chosen, rejected) triples"""
    # compute log-probabilities of chosen / rejected under policy and reference
    pi_logps_chosen = compute_logps(policy,  batch.chosen, batch.prompt)
    pi_logps_reject = compute_logps(policy,  batch.rejected, batch.prompt)
    ref_logps_chosen = compute_logps(ref,    batch.chosen, batch.prompt)
    ref_logps_reject = compute_logps(ref,    batch.rejected, batch.prompt)

    pi_logratio  = pi_logps_chosen  - pi_logps_reject
    ref_logratio = ref_logps_chosen - ref_logps_reject

    loss = -F.logsigmoid(beta * (pi_logratio - ref_logratio)).mean()
    return loss

An entire DPO training run looks just like a standard fine-tuning job — an order of magnitude simpler than PPO.

Impact

DPO rapidly became the dominant alignment method in late 2023 and throughout 2024. Open-source models including Zephyr (HuggingFace), Tulu (AI2), Nous-Hermes, Qwen, and DeepSeek all use DPO or variants.

But DPO is not a silver bullet:

  • - Sensitive to noisy preference data — IPO (2024) improves on this
  • - Requires paired data — KTO (2024) enables single-point feedback
  • - Variants merging SFT and DPO — ORPO and SimPO continue to evolve

The alignment field saw explosive diversity in 2024–2025, but the theoretical underpinnings all trace back to DPO. Recent reasoning model training (DeepSeek R1's GRPO) returned to RL-based methods — alignment techniques have not settled, and they continue to evolve.

Why You Must Read This: "PPO + RM + KL = one classification loss" is one of the most elegant reframings in the alignment field. Reading this will show you why everyone was "escaping RL" — when a problem can be solved with supervised learning, why introduce RL at all.

P16Chain-of-Thought: Letting the Model "Think" Before Answering

Main text reference: Chapter 4, §4.11.1 · §4.1.1 Test-Time Compute

Chain-of-Thought Prompting Elicits Reasoning in Large Language Models
AuthorsJason Wei, Xuezhi Wang, Dale Schuurmans, Maarten Bosma, Brian Ichter, Fei Xia, Ed Chi, Quoc Le, Denny Zhou
AffiliationGoogle Research
PublishedNeurIPS 2022 · arXiv:2201.11903 · 10,000+ citations

The Problem: LLMs Are Badly Wrong on Multi-Step Reasoning

In 2021, GPT-3 was strong at many tasks, but had a high error rate on math word problems at the elementary school level. A typical example:

Q: Roger has 5 tennis balls. He buys 2 more cans of tennis balls.
   Each can has 3 tennis balls. How many tennis balls does he have now?
Model answers directly: 27  (wrong)

The model has the "computational ability" (can compute 5 + 2×3 = 11), but when asked to answer directly, it skips steps and gets it wrong.

Core Finding: Let the Model "Write Out Its Reasoning"

What Wei et al. did sounds almost trivially simple: change the few-shot examples from "question + answer" to "question + reasoning steps + answer."

Q: Roger has 5 tennis balls. He buys 2 more cans. Each can has 3.
   How many balls does he have now?
A: Roger started with 5 balls. 2 cans of 3 balls each is 6 balls.
   5 + 6 = 11. The answer is 11.

Q: The cafeteria had 23 apples. If they used 20 to make lunch and
   bought 6 more, how many apples do they have?
A: [let the model continue]

The model now outputs something like: "The cafeteria had 23 apples. They used 20, so they had 23 - 20 = 3. Then they bought 6 more, so 3 + 6 = 9. The answer is 9." — correct.

An Emergent Phenomenon

CoT has a key characteristic: it has little or no effect on small models, but produces dramatic improvements on large ones. Experimental data from Wei et al. (GSM8K math benchmark):

ModelDirect AnswerCoT
GPT-3 0.4B4.5%5.6%
GPT-3 5.7B5.1%11.6%
GPT-3 175B15.6%46.9% (3×)
PaLM 540B17.9%56.9% (3×)

This phenomenon — a capability appearing suddenly once a model reaches sufficient scale — is called "emergence." CoT is one of the earliest clearly documented emergent abilities.

Why CoT Works: A Compute Perspective

Each LLM forward pass has a fixed compute budget ($O(N)$, where $N$ is the number of parameters). Direct answering = completing multi-step reasoning in one forward pass, which is compute-insufficient. CoT = one forward pass per reasoning step, with total compute scaling linearly with reasoning length — effectively trading token count for compute.

This framing opened the door to "test-time compute" — OpenAI o1 and DeepSeek R1's extended reasoning are essentially CoT taken to its logical extreme.

Zero-Shot CoT

Kojima et al. (2022) ("Let's think step by step") found that even without few-shot examples, simply appending "Let's think step by step" to a prompt causes large models to produce CoT output automatically. This absurdly simple prompt briefly became a sensation in the field.

CoT's Subsequent Evolution

  • Self-Consistency (2022): CoT + multiple samples + majority voting — improves error correction
  • Tree of Thoughts (2023): non-linear reasoning via a search over multiple reasoning paths
  • ReAct (2022): CoT interleaved with "reasoning" and "tool calling" — the foundation of agents
  • o1 / R1 (2024): uses RL to push CoT length to extremes, generating tens of thousands of reasoning tokens before producing an answer

Impact

CoT is a key unlock for LLM capability — it transformed "reasoning tasks" from "nearly impossible" to "doable." Today all math, coding, and reasoning benchmarks default to CoT evaluation. The "thinking mode" in modern LLMs (DeepSeek V3.1, Qwen3, Claude thinking, Gemini Deep Think) are all productized forms of CoT.

Why You Must Read This: CoT is the starting point for understanding "test-time scaling." The progression from CoT → Self-Consistency → ToT → o1 is one of the two major threads of LLM capability improvement from 2022 to 2026 (running in parallel with scaling laws).

P17DeepSeek-V3: The Culmination of Open-Source MoE

Main text reference: Chapter 4, §4.4.1 MLA · §5.6 MoE · §4.9.2 FP8

DeepSeek-V3 Technical Report
AuthorsDeepSeek-AI team
PublishedarXiv:2412.19437 (2024.12)
Follow-upDeepSeek-V3.1 (2025.8, hybrid thinking) · V3.2 (2025.10, DSA sparse attention)

Why This Paper Is the Capstone

DeepSeek-V3 is not the first MoE, the first FP8 training run, or the first hybrid reasoning model — but it is the first open-source model to integrate all of these techniques together and execute each to its limits. 671B total parameters, 37B activated per token, trained in just 2.788M H800 GPU-hours (approximately $5.6M), reaching GPT-4o-level capability.

It is a case study in "engineering integration" winning over "single-point breakthroughs," and a decisive moment in open-source catching up to closed-source.

Innovation 1: MLA (Multi-head Latent Attention)

Covered in depth in main text §4.4.1. Core idea: use low-rank decomposition to compress the KV Cache.

$\mathbf{c}_{\text{kv}} = W_{DKV} \mathbf{x}, \quad \mathbf{k} = W_{UK} \mathbf{c}_{\text{kv}}, \quad \mathbf{v} = W_{UV} \mathbf{c}_{\text{kv}}$

With $d_c \ll d_{model}$, the KV Cache stores only $\mathbf{c}_{\text{kv}}$. At inference, $W_{UK}$ can be absorbed into the Q projection, adding zero extra computation.

Compared to equivalent MHA, KV Cache is reduced by 90%+ — the core advantage enabling long-context inference.

Innovation 2: Fine-Grained MoE + Shared Expert

Each layer in V3 has 256 routed experts + 1 shared expert, with top-8 routing:

  • - Fine-grained: 256 small experts yield $C_{256}^8 = 4 \times 10^{14}$ possible combinations — vastly more diverse than Mixtral's 28
  • - Shared expert: handles general knowledge; every token passes through it
  • - Load balancing without auxiliary loss: V3 innovates by adding learnable bias terms to routing scores, dynamically adjusting to equalize expert load — eliminating the traditional auxiliary loss

Innovation 3: Multi-Token Prediction (MTP)

During training, V3 predicts not just the next token but also the next $D$ tokens ($D = 1$ in V3, i.e., one extra token predicted). Each prediction position has a dedicated MTP module (with its own Transformer layer). Benefits:

  • - Denser training signal: the model learns more from the same number of tokens
  • - At inference, the MTP module can directly serve as a speculative decoding draft head, eliminating the need for a separate draft model

Innovation 4: FP8 Training at Scale

V3 is the first model at the 671B scale to train fully in FP8. Core techniques:

  1. Fine-grained quantization: per-tensor granularity is insufficient; V3 uses per-block (groups of 1×128 elements share a scale)
  2. High-precision accumulation: FP8 × FP8 → BF16 accumulation, reducing numerical error
  3. Selective FP8: numerically sensitive operations (softmax in attention, layer normalization) retain BF16
  4. Pipeline optimization: overlapping all-to-all communication with computation, hiding cross-node latency

In practice, FP8 achieves 1.5–2× training speedup over BF16, with negligible loss degradation.

Subsequent Evolution

  • DeepSeek-R1 (2025.1): built on V3, trained as a reasoning model using pure RL (GRPO algorithm). No human-labeled CoT data needed — reasoning ability emerged from RL alone
  • V3-0324 (2025.3): post-training improvements incorporating lessons from R1's RL, with major reasoning gains
  • R1-0528 (2025.5): upgraded R1, surpassing GPT-4o on the AIME math benchmark at that time
  • V3.1 (2025.8): hybrid thinking mode — a single model supporting switchable thinking / non-thinking, merging V3 and R1
  • V3.2 (2025.10): introduces DSA (DeepSeek Sparse Attention), reducing long-context inference cost by another 50%

Significance

V3 proved three things:

  1. Open-source can match closed-source: not in 6 months, but on competitive terms
  2. Compounding returns from architectural innovation: MLA + MoE + MTP + FP8 combined yield efficiency far beyond any single component
  3. Engineering excellence from Chinese AI labs: V3's training efficiency and architectural details received broad international recognition

The open-source ecosystem moved quickly to follow in 2025 — Qwen3, Kimi K2, GLM-4.5, and LLaMA 4 all adopt similar architectural philosophy (fine-grained MoE + long-context optimization + FP8 training).

Critical Perspectives

  • - The training cost figures are debated — $5.6M may count only GPU time, excluding data, infrastructure, and labor
  • - "GPT-4 level" varies across benchmarks — V3 leads on some and trails on others
  • - MLA's engineering complexity is high; small teams may find it difficult to reproduce

Whatever the caveats, V3 is one of the most important milestones in the history of open-source LLMs.
