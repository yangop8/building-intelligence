P18HNSW: The Retrieval Engine of Vector Databases

Corresponds to main text: Chapter 2, Section 3.5.4 — Vector Databases

Efficient and robust approximate nearest neighbor using Hierarchical Navigable Small World graphs
AuthorsYu. A. Malkov, D. A. Yashunin
AffiliationYandex (independent research)
PublishedIEEE TPAMI 2020 (preprint 2016/2018)
Citations3,500+
LinkarXiv:1603.09320

The Problem: The Curse of High-Dimensional Nearest Neighbor Search

Given a 768-dimensional vector (say, the output embedding of BERT), find the top-10 most similar vectors among one million candidates — this is the Approximate Nearest Neighbor (ANN) problem. Exact methods require linear scan, and $O(N)$ is unacceptable. Traditional approaches like KD-trees degrade to linear time in high dimensions; LSH (Locality-Sensitive Hashing) demands large memory for multiple hash tables and offers a poor recall-speed tradeoff.

HNSW's starting point: organize vectors into a multi-layer graph and perform greedy search on that graph. The graph construction guarantees the "small-world" property — any two nodes are reachable in a small number of hops.

Core Method: Hierarchical Navigable Small World Graphs

The Intuition Behind NSW (Single Layer)

Start with the single-layer Navigable Small World graph: each vector is a node, and each node is connected to several "neighbors." Search starts from an entry point and greedily jumps to the neighbor closest to the query until no closer neighbor can be found — this is greedy search. The key insight of NSW: if the graph simultaneously contains short-range links (local precision) and long-range links (spanning large distances), the average number of hops in greedy search is $O(\log N)$.

Layered Structure (What the "H" Means)

HNSW adds a hierarchical structure on top of NSW, analogous to a skip list: the top layer holds only a small number of nodes connected by long-range edges for rapid coarse localization; each successive layer down adds more nodes with shorter edges for increasingly fine-grained approximation of the target.

Insertion Algorithm

To insert a new vector $q$:

  1. Randomly draw a layer $l = \lfloor -\ln(\text{uniform}(0,1)) \cdot m_L \rfloor$, where $m_L = 1/\ln(M)$. This ensures an exponentially decaying distribution of layer assignments.
  2. Starting from the entry point at the highest layer, greedily search down to the closest node to $q$, descending layer by layer until reaching layer $l$.
  3. From layer $l$ down to layer 0: at each layer, find the $ef_{\text{Construction}}$ nearest neighbors of $q$, select the best $M$ of them as $q$'s neighbors, and add bidirectional edges.

Search Algorithm Pseudocode

def hnsw_search(query, entry_point, top_k, ef):
    """HNSW approximate nearest neighbor search"""
    current_nearest = entry_point

    # Phase 1: greedy descent from the top layer down to layer 1
    for layer in range(top_layer, 0, -1):
        changed = True
        while changed:
            changed = False
            for neighbor in get_neighbors(current_nearest, layer):
                if dist(neighbor, query) < dist(current_nearest, query):
                    current_nearest = neighbor
                    changed = True

    # Phase 2: beam search on layer 0 (ef controls beam width)
    candidates = MinHeap([current_nearest])   # candidate set
    visited = set([current_nearest])
    result = MaxHeap([current_nearest])       # result set, ordered by descending distance

    while len(candidates) > 0:
        c = candidates.pop_min()
        f = result.peek_max()  # furthest node currently in result set
        if dist(c, query) > dist(f, query):
            break  # candidate is farther than the worst result, stop

        for neighbor in get_neighbors(c, layer=0):
            if neighbor not in visited:
                visited.add(neighbor)
                if dist(neighbor, query) < dist(f, query) or len(result) < ef:
                    candidates.push(neighbor)
                    result.push(neighbor)
                    if len(result) > ef:
                        result.pop_max()

    return result.top(top_k)

Experimental Results

On ANN-Benchmarks (128-dimensional SIFT 1M dataset), HNSW achieves a recall@10 of 0.99 while exceeding all contemporary methods (Annoy, FLANN, IVF-PQ) by an order of magnitude in QPS. The advantage is even more pronounced in higher dimensions (768-dim GloVe), because quantization-based methods (PQ) suffer severe precision degradation in high dimensions while HNSW's graph search is not directly affected by dimensionality.

Impact

HNSW is the default index in every mainstream vector database today:

  • Milvus: HNSW is the recommended in-memory index type
  • Qdrant: entirely built on HNSW under the hood
  • FAISS: HNSW used both as a standalone index and as the coarse quantizer in IVF
  • Pinecone, Weaviate, Chroma: all support HNSW
  • PostgreSQL pgvector: HNSW index added in version 0.5+

In a RAG pipeline (main text, Section 4.10), the "retrieval" step of "embedding + vector retrieval" runs on HNSW under the hood. Without this paper, the vector database boom of 2023–2025 would not have happened.

Critical Perspective

HNSW's main drawback is large memory footprint — the entire graph must reside in memory. One billion vectors × 768 dimensions × float32 ≈ 3 TB, and that does not even count the graph edge storage. In large-scale settings, HNSW is therefore typically combined with quantization (PQ/SQ) or sharding. Additionally, HNSW does not support dynamic deletion (vectors can only be soft-deleted), so long-running indices must eventually be rebuilt. DiskANN (Microsoft, 2019) is an alternative specifically designed for ANN on disk.

Why You Must Read This: If you are building RAG, semantic search, or recommendation systems, the vector queries in your stack are almost certainly running on HNSW under the hood. Understanding its graph construction and search logic is what lets you tune parameters correctly ($M$, $ef$) and decide when to switch to IVF-PQ or DiskANN.

P19Raft: The Understandable Consensus Algorithm

Corresponds to main text: Chapter 2 — Distributed Systems Foundations

AuthorsDiego Ongaro, John Ousterhout
AffiliationStanford University
PublishedUSENIX ATC 2014
Citations6,000+
Linkraft.github.io

The Problem: Paxos Is Too Hard to Understand

Distributed consensus — getting multiple machines to agree on a single, ordered log — is the foundational problem of distributed systems. Lamport's Paxos (1989/1998) is the theoretical gold standard, but is notorious for being "extremely difficult to understand": the original paper uses a Greek parliament as a metaphor, real-world engineering requires numerous patches (Multi-Paxos), and no two Paxos implementations are quite the same.

The goal of Ongaro's doctoral dissertation is explicit: design a consensus algorithm equivalent to Paxos but far easier to understand and implement.

Core Method: Leader Election + Log Replication + Safety

Raft decomposes consensus into three relatively independent subproblems:

1. Leader Election: The cluster always has exactly one Leader. Each node is in one of three states: Follower, Candidate, or Leader. A Follower that times out without receiving a heartbeat becomes a Candidate and initiates an election; it becomes Leader upon receiving a majority of votes. Terms (monotonically increasing integers) ensure that an old Leader automatically steps down when it sees a higher term.

2. Log Replication: All write requests are forwarded to the Leader, which appends the log entry locally and then sends it in parallel to all Followers. Once a majority of nodes acknowledge the entry, it is "committed" and the Leader notifies the client.

3. Safety: The core constraint is the Election Restriction: only a node that already holds all committed log entries can be elected Leader. This guarantees that a new Leader will never overwrite committed data.

stateDiagram-v2 [*] --> Follower Follower --> Candidate: Election timeout Candidate --> Leader: Receives majority votes Candidate --> Follower: Discovers higher term Candidate --> Candidate: Election timeout (retry) Leader --> Follower: Discovers higher term

Impact

Raft's industrial impact far exceeds its academic reception — it is the standard consensus layer in today's distributed infrastructure:

  • etcd: Kubernetes' metadata store, built on Raft
  • TiKV: TiDB's storage engine, using Multi-Raft (one Raft Group per Region)
  • CockroachDB: distributed SQL database with Raft consensus
  • Consul: HashiCorp's service discovery, using Raft consensus
  • RabbitMQ Quorum Queues: durable queues based on Raft

The success of the Raft paper validates an important principle in systems research: understandability is itself a form of value. An algorithm that is "good enough" but easy to implement correctly is more valuable in engineering practice than one that is "theoretically optimal" but easy to implement incorrectly.

Why You Must Read This: Distributed systems are the substrate of AI infrastructure — training clusters, inference services, vector databases, and configuration centers all depend on consensus. Understanding Raft's leader election and log replication is the entry point for understanding the reliability guarantees of all these systems.

P20GAN: The Birth of Adversarial Training

Corresponds to main text: Chapter 3, Section 4.4.4 — Generative Models

Generative Adversarial Nets
AuthorsIan J. Goodfellow, Jean Pouget-Abadie, Mehdi Mirza, Bing Xu, David Warde-Farley, Sherjil Ozair, Aaron Courville, Yoshua Bengio
AffiliationUniversite de Montreal
PublishedNeurIPS 2014
Citations70,000+
LinkarXiv:1406.2661

The Problem: Why Generative Models Were So Hard

Before 2014, the dominant generative models were: Restricted Boltzmann Machines (RBMs), Variational Autoencoders (VAEs, introduced the same year), and autoregressive models (PixelRNN would not arrive for another two years). Their shared limitation: either they required explicitly defining a likelihood function and performing complex inference, or they produced blurry outputs. Goodfellow, while discussing the problem with a friend at a bar, conceived a way to circumvent the likelihood function entirely — have two networks engage in adversarial competition.

Core Method: Two-Player Zero-Sum Game

Two networks are trained simultaneously:

  • Generator $G$ : generates fake samples $G(z)$ from noise $z \sim p_z(z)$, with the goal of "fooling the discriminator"
  • Discriminator $D$: judges whether an input is a real or generated sample, with the goal of "not being fooled"

Objective Function

$\min_G \max_D V(D, G) = \mathbb{E}_{x \sim p_{\text{data}}}[\log D(x)] + \mathbb{E}_{z \sim p_z}[\log(1 - D(G(z)))]$

The game-theoretic meaning of this minimax: $D$ wants to maximize its discriminative ability (pulling $V$ up), while $G$ wants to minimize $D$'s discriminative ability (pushing $V$ down).

Theoretical Results

Goodfellow proves two key theorems:

  1. Optimal discriminator: given $G$, the optimal $D^*(x) = \frac{p_{\text{data}}(x)}{p_{\text{data}}(x) + p_g(x)}$
  2. Global optimum: when $p_g = p_{\text{data}}$, $D^* = 1/2$ and $V$ reaches its global minimum of $-\log 4$. That is, at equilibrium the generated distribution perfectly matches the data distribution.

Substituting $D^*$ into $V$ reveals that $V$ is equivalent to a transformation of the Jensen-Shannon divergence between the data distribution and the generated distribution:

$V(D^*, G) = -\log 4 + 2 \cdot \text{JSD}(p_{\text{data}} \| p_g)$

Training Loop

import torch
import torch.nn as nn

class Generator(nn.Module):
    def __init__(self, z_dim=100, img_dim=784):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(z_dim, 256), nn.LeakyReLU(0.2),
            nn.Linear(256, 512), nn.LeakyReLU(0.2),
            nn.Linear(512, img_dim), nn.Tanh(),
        )

    def forward(self, z):
        return self.net(z)

class Discriminator(nn.Module):
    def __init__(self, img_dim=784):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(img_dim, 512), nn.LeakyReLU(0.2),
            nn.Linear(512, 256), nn.LeakyReLU(0.2),
            nn.Linear(256, 1), nn.Sigmoid(),
        )

    def forward(self, x):
        return self.net(x)

G = Generator(z_dim)
D = Discriminator()

opt_D = torch.optim.Adam(D.parameters(), lr=2e-4, betas=(0.5, 0.999))
bce = nn.BCELoss()

for epoch in range(num_epochs):

        real_batch = real_batch.view(batch_size, -1)

        z = torch.randn(batch_size, z_dim)

        loss_D = bce(D(real_batch), torch.ones(batch_size, 1)) + \

        opt_D.zero_grad(); loss_D.backward(); opt_D.step()

        # --- Train generator ---
        z = torch.randn(batch_size, z_dim)
        fake = G(z)
        # Trick: use -log D(G(z)) instead of log(1 - D(G(z))) for stronger gradients
        loss_G = bce(D(fake), torch.ones(batch_size, 1))
        opt_G.zero_grad(); loss_G.backward(); opt_G.step()

Training Instability: GAN's "Original Sin"

GAN's theory is elegant, but training in practice is extremely difficult:

  • Mode collapse: $G$ learns to produce only a few types of samples that are sufficient to fool $D$, sacrificing diversity
  • Training oscillation: $G$ and $D$ must stay balanced in capability — if one becomes too strong, the other receives no useful learning signal
  • Vanishing gradients: when $D$ is too accurate, $\log(1 - D(G(z))) \to 0$ and $G$ receives no gradient

A large body of subsequent work addressed these issues: WGAN (replacing JSD with Wasserstein distance), SNGAN (spectral normalization), Progressive GAN (progressively increasing resolution), StyleGAN (disentangling style and content).

Impact and Decline

From 2014 to 2021, GANs were the undisputed leaders in image generation. StyleGAN2 (2019) produced synthetic faces that remain the benchmark for "fake faces." But in 2022, diffusion models (DDPM / Stable Diffusion) surpassed GANs comprehensively in both quality and diversity, and GAN's dominant position rapidly eroded. By 2025, GANs are primarily used for:

  1. Real-time applications (inference requires only one forward pass, roughly 100× faster than diffusion models)
  2. Specific tasks such as super-resolution and style transfer
  3. Accelerating diffusion models (e.g., consistency distillation uses adversarial training)

Critical Perspective

The "adversarial training" idea outlasts the GAN architecture itself. The reward model in RLHF that discriminates between "good" and "bad" responses is, in a sense, a discriminator; adversarial example attacks and defenses (adversarial robustness) draw directly from GAN's game-theoretic framework. Goodfellow himself later shifted toward adversarial security research.

Why You Must Read This: "Adversarial training" is one of the most important paradigms in deep learning. Even though GANs have ceded ground in image generation to diffusion models, their game-theoretic perspective has influenced RLHF, adversarial robustness, and data augmentation, among other areas. Understanding GANs is what lets you understand why RLHF requires a reward model.

P21VAE: Variational Inference Meets Deep Learning

Corresponds to main text: Chapter 3, Section 4.4.4 — Generative Models

Auto-Encoding Variational Bayes
AuthorsDiederik P. Kingma, Max Welling
AffiliationUniversity of Amsterdam
PublishedICLR 2014
Citations45,000+
LinkarXiv:1312.6114

The Problem: A Missing Probabilistic Framework for Deep Generative Models

GANs sidestep the likelihood function, but at the cost of training instability and an inability to compute probabilities. Traditional Bayesian methods support probabilistic inference but cannot handle high-dimensional data (images, text). Kingma and Welling's question: can we use neural networks for variational inference while preserving a probabilistic framework?

Core Method: The Variational Lower Bound (ELBO)

Generative Model Assumption

Assume data $x$ is generated by latent variable $z$: first sample $z \sim p_\theta(z)$ (prior, typically a standard Gaussian), then decode $x \sim p_\theta(x|z)$ using a neural network. We want to maximize the marginal likelihood of the data:

$\log p_\theta(x) = \log \int p_\theta(x|z) p_\theta(z) \, dz$

This integral is intractable for high-dimensional $z$.

Variational Inference

Introduce an "encoder" $q_\phi(z|x)$ to approximate the true posterior $p_\theta(z|x)$. Applying Jensen's inequality yields the ELBO (Evidence Lower BOund):

$\log p_\theta(x) \geq \underbrace{\mathbb{E}_{q_\phi(z|x)}[\log p_\theta(x|z)]}_{\text{reconstruction term}} - \underbrace{D_{\text{KL}}(q_\phi(z|x) \| p_\theta(z))}_{\text{regularization term}} = \text{ELBO}$

The intuition behind each term:

  • Reconstruction term: how well $x$ can be recovered from $z$ (maximize)
  • KL divergence term: prevents the encoder's output distribution from straying too far from the prior (minimize; encourages a structured latent space)

The Reparameterization Trick

The key technical innovation: sampling from $q_\phi(z|x) = \mathcal{N}(\mu_\phi(x), \sigma^2_\phi(x))$ does not allow gradients to flow back through the sampling operation (which is non-differentiable). Kingma's solution: factor out the randomness:

$z = \mu_\phi(x) + \sigma_\phi(x) \odot \epsilon, \quad \epsilon \sim \mathcal{N}(0, I)$

Now $z$ is a deterministic function of $\phi$ (with external noise $\epsilon$), and gradients can backpropagate normally.

PyTorch Implementation

import torch
import torch.nn as nn
import torch.nn.functional as F

class VAE(nn.Module):
    def __init__(self, input_dim=784, z_dim=20):
        super().__init__()
        # Encoder
        self.fc1 = nn.Linear(input_dim, 400)
        self.fc_mu = nn.Linear(400, z_dim)
        self.fc_logvar = nn.Linear(400, z_dim)
        # Decoder
        self.fc3 = nn.Linear(z_dim, 400)
        self.fc4 = nn.Linear(400, input_dim)

    def encode(self, x):
        h = F.relu(self.fc1(x))
        return self.fc_mu(h), self.fc_logvar(h)

    def reparameterize(self, mu, logvar):
        std = torch.exp(0.5 * logvar)
        eps = torch.randn_like(std)       # external noise
        return mu + std * eps             # reparameterization

    def decode(self, z):
        h = F.relu(self.fc3(z))
        return torch.sigmoid(self.fc4(h))

    def forward(self, x):
        mu, logvar = self.encode(x.view(-1, 784))
        z = self.reparameterize(mu, logvar)
        return self.decode(z), mu, logvar

def vae_loss(recon_x, x, mu, logvar):
    # Reconstruction loss (binary cross-entropy)
    BCE = F.binary_cross_entropy(recon_x, x.view(-1, 784), reduction='sum')
    # KL divergence: -0.5 * sum(1 + log(sigma^2) - mu^2 - sigma^2)
    KLD = -0.5 * torch.sum(1 + logvar - mu.pow(2) - logvar.exp())
    return BCE + KLD

VAE's "Blurriness" Problem and Subsequent Developments

VAE's generation quality (especially for images) has always lagged behind GANs — outputs tend to be blurry. The cause: pixel-level reconstruction loss (MSE/BCE) encourages "averaging" over all plausible outputs. Subsequent improvements include:

  • VQ-VAE (van den Oord, 2017): discrete latent space replacing the continuous Gaussian, yielding sharper generations
  • VQ-VAE-2 (Razavi, 2019): hierarchical discrete codes, quality approaching BigGAN
  • Stable Diffusion: diffusion performed in VQ-VAE's latent space — the VAE handles compression, diffusion handles generation. This is the standard image generation pipeline of 2023–2025.

Impact

VAE's influence extends well beyond the application of image generation:

  1. Reparameterization trick: has become the standard method for any setting that requires backpropagating gradients through sampling operations (reinforcement learning, Gumbel-Softmax, etc.)
  2. Latent space learning: VAE defines the "encoder–latent space–decoder" paradigm, which directly shaped the Latent Diffusion architecture of Stable Diffusion
  3. Variational inference + deep learning: opened the door to probabilistic deep learning, spawning variational sequence models, variational graph neural networks, and more
Why You Must Read This: VAE is a prerequisite for understanding the Stable Diffusion architecture — the "Latent" in Latent Diffusion is exactly the VAE's latent space. The reparameterization trick is also widely used in the LLM domain (e.g., Gumbel-Softmax for discrete selection). Kingma is the author of both Adam and VAE; together, those two papers may constitute the highest-citation combination in all of deep learning.

P22DDPM: Diffusion Models, the Foundation of Modern Image Generation

Corresponds to main text: Chapter 3, Section 4.4.4 — Generative Models

Denoising Diffusion Probabilistic Models
AuthorsJonathan Ho, Ajay Jain, Pieter Abbeel
AffiliationUC Berkeley
PublishedNeurIPS 2020
Citations15,000+
LinkarXiv:2006.11239

The Problem: Where Is the Alternative to GANs?

In 2020, GANs (particularly StyleGAN2) held the highest image generation quality, but training instability and mode collapse continued to frustrate the community. The theoretical framework of diffusion models had been proposed as early as 2015 by Sohl-Dickstein, but generation quality remained far below GANs. Ho et al.'s contribution was the first time a diffusion model matched GAN quality — and with dramatically simpler training.

Core Method: Forward Noising + Reverse Denoising

Forward Process (Diffusion)

Starting from a real image $x_0$, Gaussian noise is added incrementally over $T$ steps until the result is pure noise:

$q(x_t | x_{t-1}) = \mathcal{N}(x_t; \sqrt{1 - \beta_t} \, x_{t-1}, \beta_t I)$

Here $\beta_t$ is a predefined noise schedule that increases from small to large (e.g., $\beta_1 = 10^{-4}$ to $\beta_T = 0.02$). A key property: one can jump directly from $x_0$ to any $x_t$ without sequential computation:

$q(x_t | x_0) = \mathcal{N}(x_t; \sqrt{\bar{\alpha}_t} \, x_0, (1 - \bar{\alpha}_t) I)$

where $\alpha_t = 1 - \beta_t$ and $\bar{\alpha}_t = \prod_{s=1}^{t} \alpha_s$. For sufficiently large $T$, $x_T \approx \mathcal{N}(0, I)$.

Reverse Process (Denoising)

If we can learn to reverse each step — predicting $x_{t-1}$ from $x_t$ — we can generate images starting from pure noise. The reverse process is also Gaussian:

$p_\theta(x_{t-1} | x_t) = \mathcal{N}(x_{t-1}; \mu_\theta(x_t, t), \sigma_t^2 I)$

Ho et al.'s key simplification: instead of directly predicting $\mu$, the network predicts the noise $\epsilon$ . The network $\epsilon_\theta(x_t, t)$ takes a noisy image and a timestep as input and outputs the predicted noise.

Training Objective: Radical Simplification

DDPM's training loss is the simplest of any deep generative model:

$L_{\text{simple}} = \mathbb{E}_{t, x_0, \epsilon}\left[\|\epsilon - \epsilon_\theta(\sqrt{\bar{\alpha}_t} x_0 + \sqrt{1-\bar{\alpha}_t} \epsilon, \, t)\|^2\right]$

In plain terms: randomly sample a timestep $t$, randomly sample noise $\epsilon$, add the noise to $x_0$ to obtain $x_t$, have the network predict the added noise, and train with MSE. That is all.

Training and Sampling Code

import torch
import torch.nn as nn

# Noise schedule
def linear_schedule(T, beta_start=1e-4, beta_end=0.02):
    return torch.linspace(beta_start, beta_end, T)

T = 1000
betas = linear_schedule(T)
alphas = 1.0 - betas
alpha_bars = torch.cumprod(alphas, dim=0)

# Forward noising: jump directly from x_0 to x_t
def q_sample(x_0, t, noise=None):
    if noise is None:
        noise = torch.randn_like(x_0)
    sqrt_alpha_bar = alpha_bars[t].sqrt().view(-1, 1, 1, 1)
    sqrt_one_minus = (1 - alpha_bars[t]).sqrt().view(-1, 1, 1, 1)
    return sqrt_alpha_bar * x_0 + sqrt_one_minus * noise, noise

# Training loop
model = UNet(...)  # U-Net that predicts noise
optimizer = torch.optim.Adam(model.parameters(), lr=2e-4)

for batch in dataloader:
    x_0 = batch['image']
    t = torch.randint(0, T, (x_0.size(0),))       # random timestep
    x_t, noise = q_sample(x_0, t)                   # add noise
    pred_noise = model(x_t, t)                       # predict noise
    loss = nn.functional.mse_loss(pred_noise, noise) # just MSE
    optimizer.zero_grad(); loss.backward(); optimizer.step()

# Sampling: iteratively denoise from pure noise
@torch.no_grad()
def sample(model, shape):
    x = torch.randn(shape)  # x_T ~ N(0, I)
    for t in reversed(range(T)):
        t_batch = torch.full((shape[0],), t, dtype=torch.long)
        pred_noise = model(x, t_batch)
        # Compute mean of x_{t-1}
        alpha = alphas[t]; alpha_bar = alpha_bars[t]
        x = (1 / alpha.sqrt()) * (x - (1 - alpha) / (1 - alpha_bar).sqrt() * pred_noise)
        if t > 0:
            x = x + betas[t].sqrt() * torch.randn_like(x)  # reintroduce stochasticity
    return x

Downstream Development Chain

  1. DDIM (Song, 2020): deterministic sampling — 50 steps is enough, no need for 1000
  2. Classifier-Free Guidance (Ho & Salimans, 2022): conditional generation without a separate classifier, dramatically improving quality
  3. Latent Diffusion / Stable Diffusion (Rombach, 2022): diffusion in VAE latent space; 64× compression makes the process tractable
  4. DALL-E 2/3, Midjourney, Imagen: text-to-image generation, all based on diffusion
  5. Sora (OpenAI, 2024): video generation using diffusion + Transformer (DiT)

Critical Perspective

The inference speed problem of diffusion models is not fully solved — compressing 1000 steps to 4–8 (consistency distillation) incurs quality degradation. Additionally, the theoretical framework of diffusion models limits their applicability to discrete data (text) — text generation remains the domain of autoregressive models (GPT). There are some 2024–2025 attempts at discrete diffusion models (e.g., MDLM), but no substantial breakthrough has emerged yet.

Why You Must Read This: Stable Diffusion, DALL-E, Midjourney, Sora — every stunning image and video generation result from 2022–2025 builds on DDPM's ideas. Once you understand the mathematics of forward noising and reverse denoising, you can grasp why text-to-image generation requires so many inference steps, and why Latent Diffusion moves the diffusion process into latent space.

P23Word2Vec: The Starting Point of Word Embeddings

Corresponds to main text: Chapter 3, Section 4.5.2 — Embeddings

Efficient Estimation of Word Representations in Vector Space
AuthorsTomas Mikolov, Kai Chen, Greg Corrado, Jeffrey Dean
AffiliationGoogle
PublishedICLR 2013 Workshop
Citations40,000+
LinkarXiv:1301.3781

The Problem: A Mathematical Representation of Words

The first challenge in NLP: how do we turn words into numbers? Before 2013, the mainstream approach was one-hot vectors (a $V$-dimensional vector with a single 1, where $V$ is the vocabulary size) or TF-IDF. The problems: one-hot vectors are extremely high-dimensional (vocabularies of tens to hundreds of thousands), and the distance between any two words is identical (zero semantic information).

Mikolov's goal: train dense, low-dimensional (100–300 dimensions) vectors such that semantically similar words are close together in the vector space.

Core Method: Two Architectures

CBOW (Continuous Bag of Words): predict the center word from the words in a context window. For example, given "the cat sat on the ___", use {the, cat, sat, on, the} to predict "mat."

Skip-gram: the reverse — predict the context from the center word. Given "sat", predict {cat, on}, etc. In practice, Skip-gram performs better (especially for rare words) and became the standard.

$P(w_o | w_c) = \frac{\exp(\mathbf{v}_{w_o}' \cdot \mathbf{v}_{w_c})}{\sum_{w \in V} \exp(\mathbf{v}_w' \cdot \mathbf{v}_{w_c})}$

Summing the denominator over the entire vocabulary is too slow. Two speedup strategies were introduced: Hierarchical Softmax (using a Huffman tree) and Negative Sampling (randomly sample a few "negatives" from the vocabulary and update only those). Negative Sampling is extremely simple and efficient and has become the de facto standard.

A Surprising Property: Vector Arithmetic

The most famous phenomenon discovered by Word2Vec:

$\text{king} - \text{man} + \text{woman} \approx \text{queen}$

Semantic relationships are encoded as directions in vector space. This finding directly shaped the design objectives of every subsequent embedding method.

Impact

Word2Vec ushered in the era of "everything can be embedded":

  • GloVe (Stanford, 2014): replaces the local window with a global co-occurrence matrix for a more elegant mathematical framework
  • FastText (Facebook, 2016): subword-level embeddings that handle out-of-vocabulary words
  • ELMo (Peters, 2018): contextual word vectors (the same word has different vectors in different sentences)
  • BERT/GPT: essentially upgrading "static embeddings" to "dynamic contextual embeddings"

By 2025, no one uses Word2Vec for NLP tasks — BERT/GPT contextual embeddings have replaced them entirely. Yet Word2Vec's ideas live on: Item2Vec in recommendation systems, Node2Vec in graph embeddings, Code2Vec in code embeddings — all are extensions of Word2Vec's paradigm of "mapping discrete objects into continuous space."

Why You Must Read This: Word2Vec is the best entry point for understanding the concept of "embedding." The token embeddings in BERT, the positional encodings in Transformers, the vector retrieval in RAG — all of these trace their intellectual origins to "representing discrete symbols with low-dimensional dense vectors." Mikolov's paper transformed this idea from an academic concept into an engineering standard.

P24Mamba: Linear-Time Sequence Modeling

Corresponds to main text: Chapter 4, Section 4.4.3 — Non-Transformer Architectures

Mamba: Linear-Time Sequence Modeling with Selective State Spaces
AuthorsAlbert Gu, Tri Dao
AffiliationCarnegie Mellon University, Princeton University
PublishedCOLM 2024 (preprint 2023)
Citations3,000+
LinkarXiv:2312.00752

The Problem: Transformer's Quadratic Complexity

Transformer's self-attention is $O(N^2)$ in both time and memory with respect to sequence length $N$ — a fundamental limitation. FlashAttention (P10) reduces the constant factor through hardware-level optimization, but the quadratic asymptotic complexity remains. For ultra-long sequences at the million-token scale (genomes, long videos, code repositories), Transformers remain impractical.

From 2020 to 2023, Gu explored State Space Models (SSMs) — a sequence modeling framework rooted in control theory with $O(N)$ complexity. Prior work S4 (2021) and H3 (2022) demonstrated feasibility, but these models consistently fell short of Transformers in language modeling. Mamba is the breakthrough in this line of research.

Background: Structured State Space Models (S4)

The continuous-time SSM is defined as:

$h'(t) = Ah(t) + Bx(t)$
$y(t) = Ch(t) + Dx(t)$

After discretization (using zero-order hold or bilinear transform):

$h_t = \bar{A} h_{t-1} + \bar{B} x_t$
$y_t = C h_t + D x_t$

This is a linear recurrence — similar to an RNN, but with fixed parameters $\bar{A}, \bar{B}, C$ that do not depend on the input. Fixed parameters mean: the recurrence can be unrolled into a convolution and accelerated with FFT ($O(N \log N)$ training), while also supporting $O(1)$ per-step inference via the recurrence form.

S4's limitation: because $A, B, C$ do not depend on the input, the model cannot selectively remember or forget information based on content. This is precisely the core strength of Transformer attention (attention weights are entirely determined by the input).

Core Innovation: Selective State Spaces

Mamba's key insight: make $B$, $C$, and the discretization step size $\Delta$ input-dependent:

$B_t = \text{Linear}(x_t), \quad C_t = \text{Linear}(x_t), \quad \Delta_t = \text{softplus}(\text{Linear}(x_t))$

This allows the model to decide "how much to remember" and "what to output" based on the content of the current token. $\Delta_t$ controls the discretization step size — when $\Delta$ is large, $\bar{A}$ approaches 0 (forgetting history); when $\Delta$ is small, $\bar{A}$ approaches 1 (retaining history).

Input dependence means FFT can no longer be used (the convolution kernel is no longer fixed). Gu and Dao (also the author of FlashAttention) designed a hardware-aware selective scan algorithm — essentially replacing the naive recurrence with parallel prefix sums (parallel scan) for efficient GPU execution.

Mamba Block Architecture

The full Mamba block is more than just the SSM — it also wraps a structure similar to the Transformer FFN:

    def __init__(self, d_model, d_state=16, d_conv=4, expand=2):
        super().__init__()
        d_inner = d_model * expand
        self.in_proj = nn.Linear(d_model, d_inner * 2, bias=False)  # split into two branches
        self.conv1d = nn.Conv1d(d_inner, d_inner, d_conv, padding=d_conv-1,
                                groups=d_inner)  # local convolution
        self.act = nn.SiLU()

        # SSM parameters (input-dependent)
        self.x_proj = nn.Linear(d_inner, d_state * 2 + 1, bias=False)  # B, C, delta
        self.A_log = nn.Parameter(torch.log(torch.arange(1, d_state+1).float()))
        self.D = nn.Parameter(torch.ones(d_inner))

        self.out_proj = nn.Linear(d_inner, d_model, bias=False)

        xz = self.in_proj(x)
        x_branch, z = xz.chunk(2, dim=-1)

        # Local convolution (captures nearby context)
        x_branch = self.conv1d(x_branch.transpose(1, 2))[:, :, :x.size(1)]
        x_branch = x_branch.transpose(1, 2)

        # Selective SSM
        y = self.selective_ssm(x_branch)
        y = y + self.D.unsqueeze(0).unsqueeze(0) * x_branch  # skip connection

        output = y * self.act(z)

Impact and Controversy

Mamba sparked a wave of interest in "Transformer alternatives":

  • Mamba-2 (Gu & Dao, 2024): unifies SSMs into a matrix multiplication framework for further speedup
  • Jamba (AI21, 2024): hybrid Mamba + Transformer architecture
  • RWKV (Peng, 2023): another RNN revival approach, complementary in spirit to Mamba
  • Zamba (Zyphra, 2024): Mamba backbone with a small number of attention layers

Yet the controversy is significant: as of 2025, no Mamba-only model has genuinely surpassed a same-scale Transformer in large-scale (>10B) language modeling. The mainstream direction has shifted toward hybrid architectures (a few attention layers combined with many SSM layers). Mamba's value is more likely as a "complementary component" to Transformers than as a "replacement."

Why You Must Read This: Understanding "why Transformer's quadratic complexity is a problem" and "where the tradeoffs in linear alternatives lie" is foundational for evaluating the trajectory of LLM architecture evolution. Mamba's selective state space idea — letting the model decide its memory strategy based on content — is the modern 2024 revival of LSTM's gating concept.

P25RAG: Giving LLMs Access to External Knowledge

Corresponds to main text: Chapter 4, Section 4.10 — RAG

Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks
AuthorsPatrick Lewis, Ethan Perez, Aleksandra Piktus, Fabio Petroni, Vladimir Karpukhin, Naman Goyal, Heinrich Kuttler, Mike Lewis, Wen-tau Yih, Tim Rocktaschel, Sebastian Riedel, Douwe Kiela
AffiliationFacebook AI Research, University College London
PublishedNeurIPS 2020
Citations4,000+
LinkarXiv:2005.11401

The Problem: The Closed-Knowledge Nature of LLMs

Pretrained language models "bake" knowledge into their parameters. The problems this creates:

  1. Hallucination: knowledge not seen during training gets fabricated
  2. Staleness: information after the training cutoff is unknown
  3. Traceability: answers cannot cite their sources

Lewis et al. propose a simple but far-reaching paradigm: retrieve relevant documents first, then feed those documents into the generative model's input.

Core Method: Retrieve-then-Generate

The RAG model consists of two components:

  • Retriever: given a query $q$, use DPR (Dense Passage Retrieval) to retrieve top-$k$ documents $\{d_1, \ldots, d_k\}$ from a knowledge base
  • Generator: concatenate the query with the retrieved documents and use a seq2seq model (the paper uses BART) to generate the answer

The paper defines two marginalization strategies:

$p_{\text{RAG-Sequence}}(y|x) = \sum_{z \in \text{top-}k} p_\eta(z|x) \, p_\theta(y|x, z)$
$p_{\text{RAG-Token}}(y|x) = \prod_i \sum_{z \in \text{top-}k} p_\eta(z|x) \, p_\theta(y_i|x, z, y_{1:i-1})$

RAG-Sequence: each document independently generates a complete answer, then results are weighted and combined. RAG-Token: each token can attend to a different document. In engineering practice, most RAG systems use a further simplified version — concatenating retrieved documents directly into the prompt without any probabilistic marginalization.

Impact: From Paper to Industry Paradigm

From 2023 to 2025, RAG evolved from an academic paper into the standard architecture for LLM applications:

  • Enterprise knowledge Q&A: embed company documents into a vector store and use RAG to answer employee questions
  • Code assistants: retrieve relevant files from a code repository to assist code generation (Cursor, GitHub Copilot)
  • Search engines: Perplexity and Bing Chat are both "search + generation" RAG variants
  • Vector database boom: the rise of Milvus, Pinecone, Qdrant, and others was directly driven by RAG demand (connecting to P18 HNSW)

In engineering practice, the main challenges of RAG are not the probabilistic framework discussed in the original paper, but rather: chunking granularity (how large should chunks be), retrieval quality (semantic search vs. keyword search), context window limitations (too many retrieved chunks won't fit), and the "lost-in-the-middle" problem of retrieving context that the model fails to use effectively.

Why You Must Read This: RAG is the dominant paradigm for deploying LLMs in production from 2023 to 2025 — almost every enterprise AI application starts with "build a RAG." Understanding the retrieve-then-generate framework from the original paper, and the gap between that framework and current engineering practice, is a fundamental skill for any AI engineer.

P26ReAct: Synergizing Reasoning and Acting in LLMs

Corresponds to main text: Chapter 4, Section 4.11 — Agents

ReAct: Synergizing Reasoning and Acting in Language Models
AuthorsShunyu Yao, Jeffrey Zhao, Dian Yu, Nan Du, Izhak Shafran, Karthik Narasimhan, Yuan Cao
AffiliationPrinceton University, Google Research
PublishedICLR 2023
Citations3,000+
LinkarXiv:2210.03629

The Problem: The Disconnect Between Reasoning and Acting

Chain-of-Thought (P16) taught LLMs to "think" — but only as internal reasoning, with no ability to interact with the external world. On the other hand, "tool use" (having LLMs call APIs) provides action without reasoning — the model offers no explanation for why it is invoking a particular API. Yao et al.'s observation: when humans solve problems, reasoning and action alternate — "I think I should look this up (reasoning) → search (action) → the search results show... (observation) → so the answer is... (reasoning)."

Core Method: Thought-Action-Observation Loop

ReAct has the LLM interleave three types of content within the prompt:

Thought: reason about the current state and decide the next step ("I need to find X's birth year")

Action: invoke an external tool (search, calculator, database query, etc.)

Observation: the result returned by the tool (search results, computed value, etc.)

A typical ReAct trace:

Question: Which was founded earlier — Obama's alma mater or Trump's alma mater?

Thought 1: I need to find Obama's alma mater first.
Action 1: Search[Barack Obama education]
Observation 1: Barack Obama attended Columbia University and Harvard Law School.

Thought 2: Obama's alma mater is Columbia University and Harvard Law School. Now find Trump's.
Action 2: Search[Donald Trump education]
Observation 2: Donald Trump attended the Wharton School at the University of Pennsylvania.

Thought 3: Trump's alma mater is Penn. Now compare founding dates. Columbia was founded in 1754, Penn in 1740.
Action 3: Lookup[University of Pennsylvania founding year]
Observation 3: The University of Pennsylvania was founded in 1740.

Thought 4: Penn (1740) was founded earlier than Columbia (1754).
Action 4: Finish[Trump's alma mater, the University of Pennsylvania (1740), was founded earlier than Obama's alma mater, Columbia University (1754).]

Impact: The Starting Point of the Agent Paradigm

ReAct directly defined the basic pattern of LLM Agents from 2023 to 2025:

  • LangChain Agent: the default agent type is the ReAct pattern
  • AutoGPT, BabyAGI: essentially automated versions of the ReAct loop (with added task decomposition and memory)
  • OpenAI Function Calling: formats ReAct's Action from free text into structured JSON calls
  • Claude Code, Cursor Agent: ReAct in a programming context — "analyze the problem → read files → modify code → run tests"

The essence of ReAct is not any specific technology, but a paradigm shift: an LLM is not just "a model that answers questions" but "an Agent capable of reasoning and acting." This shift in perspective gave rise to the entire AI Agent ecosystem.

Why You Must Read This: If you are building LLM Agents — whether customer service bots, code assistants, or automated workflows — the architecture underneath is almost certainly a variant of the ReAct pattern. Understanding this simple but powerful "alternating reasoning and action" framework is the first step toward designing reliable Agents.
