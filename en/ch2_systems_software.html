Systems Software

The Systems Perspective for AI Engineers — From CPU to DPDK/VPP
Chapter 2 · 28 sections

What does systems software mean for AI engineers? Three things. First, training and inference both run on top of systems — no matter how fast the GPU, data traveling from disk to device memory must pass through PCIe, the kernel, and the file-system cache; every layer can become a bottleneck. Second, AI infrastructure is itself systems software — CUDA/CANN are driver-layer stacks, the vLLM scheduler borrows from OS process scheduling, and PagedAttention maps directly onto virtual-memory paging. Third, in the era of AI-assisted programming, judgment about low-level systems is worth more than raw implementation skill — AI can write the code, but architectural decisions, performance analysis, and fault diagnosis still require a human.

This chapter unfolds 28 sections in stack order: chips → OS → languages/compilers → data storage → communication. As in Chapter 1, coverage is intentionally asymmetric: memory management, concurrency and synchronization, Rust ownership, compiler optimizations, and DPDK/VPP receive deep treatment; device drivers, traditional relational databases, and certain transport-protocol details are covered more briefly. Proprietary internal technology stacks (specific internal middleware, private protocols, etc.) are replaced throughout with open-source general-purpose equivalents — master DPDK, understanding any proprietary equivalent is just one more layer of abstraction.

PART ONE

2.1 CPU & Core Architecture

AI training depends on GPUs and NPUs, but the CPU is the prerequisite for understanding everything else.

2.1.1CPU Overall Architecture

Scope: ISA, pipeline, cache, multi-core — the skeleton of modern processors.

ISA: The Contract CPU Exposes to Software

The Instruction Set Architecture (ISA) is the programming model a CPU exposes to software. It defines which instructions exist, how many registers are available, and how memory is addressed. The three mainstream ISAs:

ISAStyleExamplesUse Cases
x86 / x86-64CISCIntel / AMDPCs, servers
ARMRISCApple M-series, AmpereMobile, cloud, embedded
RISC-VRISC (open)SiFive, Alibaba T-HeadEmerging: IoT to HPC

The CISC (Complex Instruction Set) vs. RISC (Reduced Instruction Set) divide is no longer as sharp as it once was — modern x86 CPUs decompose complex instructions into micro-operations (μops) at the front end, making the back-end execution units essentially identical to RISC. From a software perspective, though, the ISA still determines which assembly instructions you can use, whether vector extensions are AVX-512 or SVE, and whether a TSO memory model is in play.

RISC-V occupies a special position because it is open — anyone can implement it freely and add their own extensions. It is reshaping the landscape of embedded and accelerator chips at a remarkable pace.

Microarchitecture: The Reality Beneath the ISA

The same ISA can yield several times the performance difference depending on the microarchitecture. Modern high-performance CPU cores contain these key components:

Pipeline: Overlaps the five stages — fetch, decode, execute, memory access, writeback — so that one instruction completes per cycle. Branches disrupt the pipeline, which is why there is a branch predictor that uses historical statistics to guess the next instruction. Accuracy routinely exceeds 95%. A misprediction flushes the pipeline at a cost of 15–20 cycles.

Out-of-order execution: Hardware detects when instructions have no dependencies and issues them in parallel. Code written sequentially by the programmer is reordered by hardware according to the dataflow. The Spectre/Meltdown vulnerabilities exploited side effects of this out-of-order execution.

SIMD: Single Instruction, Multiple Data. AVX-512 can compute 8 doubles with a single instruction. Before GPUs dominated, CPU inference relied on SIMD as its primary workhorse for deep learning.

Cache Hierarchy

Memory is two orders of magnitude slower than the CPU (100+ ns vs. <1 ns). The solution is a three-level cache:

LevelSizeLatencyScope
L1 (data + instr)32KB + 32KB~4 cyclesper-core
L2256KB–1MB~12 cyclesper-core
L3 (LLC)several MB – tens of MB~40 cyclesshared
DRAMGB~200 cyclesglobal

Understanding the cache is the key to understanding performance. Sequential array access lets the prefetcher predict the next cache line accurately, at near-zero cost; random linked-list traversal produces a cache miss on almost every access, slowing things down by tens of times. This is why tensors in AI almost universally use contiguous memory layouts.

// The dramatic effect of cache on performance
#define N 16384
int a[N][N];

// Version 1: row-major access, cache-friendly
for (int i = 0; i < N; i++)
    for (int j = 0; j < N; j++) sum += a[i][j];

// Version 2: column-major access, strides N*sizeof(int) bytes each time
for (int j = 0; j < N; j++)
    for (int i = 0; i < N; i++) sum += a[i][j];

// Version 2 is typically 5–10x slower than Version 1 — purely because of cache misses

Multi-Core and Cache Coherence

When multiple cores share memory, the system must guarantee that "what I wrote, you can see." Hardware maintains cache-line state through the MESI protocol (Modified/Exclusive/Shared/Invalid). If one core modifies a cache line, copies held by other cores are marked Invalid; the next access on those cores must reload the line.

This introduces a subtle trap — False Sharing. Two unrelated variables happen to reside on the same cache line (64 bytes). Two cores each write their respective variables frequently, and each write triggers a cache invalidation on the other core. The fix is cache-line alignment — annotate critical variables with alignas(64), or pad them to 64 bytes.

In AI

AI training's data preprocessing, tokenization, and sampling logic all run on the CPU. One frequently overlooked detail: DataLoader CPU performance is often the training bottleneck. The GPU computes at blazing speed while the CPU cannot feed it data fast enough. The remedies are multi-process DataLoaders, binding workers to NUMA nodes, and using pinned memory to reduce copies — all of which rest on a basic understanding of CPU architecture.

2.1.2Network Processing Units

Scope: dedicated hardware for accelerating network packet processing — from NP chips to smart NICs.

Why Dedicated Units Are Needed

Line-rate processing of 100 Gbps traffic, assuming 64-byte small packets, means 148.8 Mpps (million packets per second). The per-packet budget is 6.7 ns — not even enough time for a single DRAM access. General-purpose CPUs cannot keep up with pure software approaches; hardware is mandatory.

Hardware Acceleration Approaches

NP (Network Processor): A multi-core processor designed specifically for packet processing, with an instruction set that includes header parsing, table lookups, and checksum operations. Each core runs a segment of microcode, processing packets in a pipeline fashion. Examples: Broadcom XLP, Intel IXP.

ASIC (switching chip): Fully hardened for fixed protocols. The Broadcom Tomahawk and Marvell Teralynx chips inside Ethernet switches are ASICs — table lookups, forwarding, and VLAN processing are all done in hardware. Latency can reach the hundreds-of-nanoseconds range, but functionality is fixed at fabrication time.

Smart NIC / DPU: NVIDIA BlueField, Intel IPU, AWS Nitro. Essentially a NIC with an ARM SoC on board that can run a full OS, offloading networking, storage, and security functions from the host CPU to the NIC. This is the dominant trend in data-center-scale acceleration.

P4 programmable pipeline: Positioned between ASICs and NPs. The hardware is a fixed pipeline, but the behavior of each pipeline stage is defined in P4 and loaded at power-on. Example: Barefoot Tofino.

Common Hardware Acceleration Features

  • RSS (Receive Side Scaling): The NIC hashes packets on the 5-tuple (source IP, destination IP, protocol, source port, destination port) and distributes them to receive queues on different CPU cores, enabling parallel processing without conflicts.
  • Hardware LRO/GRO: Combines multiple small packets from the same flow into a larger packet before handing them to the protocol stack, reducing the number of syscalls.
  • TSO/GSO: The reverse — the protocol stack hands a large segment to the NIC, which splits it into MTU-sized frames on its own.
  • Checksum/Crypto offload: TCP checksums and TLS encryption/decryption are performed entirely by the NIC.
  • ECMP hardware implementation: Equal-Cost Multi-Path routing; the hardware distributes traffic by hash.

In AI

Large-model training clusters (where GPUs communicate over RDMA via InfiniBand/RoCE) depend on NIC hardware capabilities. GPUDirect RDMA lets the NIC read and write GPU memory directly, bypassing CPU memory. The latency of a single AllReduce operation is dominated by the NIC's RDMA performance, not GPU compute throughput.

2.1.3GPU and NPU Acceleration

Scope: the carrier of AI compute — hardware for parallel computation and tensor acceleration. Brief here; details in Chapter 4.

This section overlaps with Chapter 4 / Section 5.2. Here we only establish the basic reference frame.

The GPU Parallel Model

The GPU's DNA is graphics rendering — shading a massive number of independent pixels is inherently parallel. Modern GPUs repurpose this capability for general-purpose computation (GPGPU). An NVIDIA H100 has approximately 16,896 CUDA cores + 528 Tensor Cores, with FP16 Tensor Core throughput of roughly 989 TFLOPS (dense) / 1,979 TFLOPS (2:4 sparse).

The GPU programming model is called SIMT (Single Instruction, Multiple Threads) — 32 threads form a warp and a single instruction acts on all 32 data elements simultaneously. This is conceptually the same as SIMD; the difference is that if threads within a warp diverge at a branch (if statement), the GPU serializes execution of both paths — known as warp divergence — causing a performance collapse.

NPU: Tensor-First Design

An NPU (Neural Processing Unit) is designed from the ground up for tensor computation. The core architecture is the systolic array: a two-dimensional grid of MAC (multiply-accumulate) units through which data flows like a pulse — each unit performs a multiply-accumulate on a fixed schedule. Google's TPU is the most well-known systolic array design.

Huawei's Ascend uses the DaVinci architecture: a Cube unit for matrix multiplication, a Vector unit for element-wise operations, and a Scalar unit for control flow. Operator developers must explicitly dispatch computation to the appropriate unit.

Memory Structure

GPU/NPU memory hierarchies are more extreme:

LevelCapacityBandwidth
Registers64KB per threadextremely fast
Shared memory / SRAM~100KB per block~several TB/s
HBM80GB (H100)~3 TB/s

HBM bandwidth is large, but still 10× slower than on-chip SRAM. The essence of operator optimizations like FlashAttention is to keep computation in SRAM as much as possible and minimize round-trips to HBM.

2.1.4SoC Heterogeneous Computing

Scope: a single chip integrating CPU/GPU/NPU/accelerators — the paradigm for modern embedded and edge AI. Brief.

SoC (System on Chip) is not a new concept. Mobile processors have been SoCs since the iPhone era — a single chip integrating CPU big and little cores, a GPU, an ISP (image signal processor), a neural network accelerator, and a modem. The data-center SoC trend is even more pronounced: AWS Graviton and Ampere Altra both integrate CPUs, memory controllers, PCIe, and network subsystems on a single die.

Challenges of Heterogeneous Computing

When multiple compute units share memory and buses on a single chip, the challenges are:

  1. Data movement: Copying data between CPU memory and GPU/NPU memory is the dominant overhead. Unified memory (CUDA Unified Memory, Apple Silicon's Unified Memory Architecture) attempts to solve this, but truly zero-copy operation requires hardware coherence support.
  2. Scheduling: Which code runs on the CPU and which runs on the accelerator? Frameworks like PyTorch and TensorFlow decide automatically, but edge cases are numerous.
  3. Power wall: All units share the chip's power budget. Running the NPU at full tilt forces the CPU to throttle.

Common Offload Engines

Network device SoCs commonly include, beyond the CPU and NP:

  • - Compression engine (hardware-accelerated DEFLATE/LZ4)
  • - Encryption/decryption engine (AES, SHA, Chinese national standard SM4)
  • - Regex matching engine (DFA execution, used for IPS intrusion detection)
  • - Table-lookup engine (TCAM, used for route lookups)

These are not directly relevant to AI scenarios, but understanding the concept of "offloadable computation" is highly instructive for understanding DPUs and smart NICs — they are all, at their core, extensions of the SoC philosophy into the networking domain.

PART TWO

2.2 Operating Systems

The OS is the invisible infrastructure beneath all higher-level software — AI framework scheduling, containers, and networking all pass through it.

2.2.1OS Overview

Scope: what an OS is, what it does, and the differences between the major OS families.

An operating system does two things: manage hardware resources (CPU, memory, disk, network) and provide abstractions to applications (processes, files, sockets). Dijkstra's old saying: "Abstraction is not about hiding details — it is about creating a new, precise level of description." The OS is exactly such a level.

Major OS Families

FamilyKernelExamplesUse Cases
Unix-likeMonolithicLinux, FreeBSDServers, containers, supercomputers
macOS / iOSXNU (hybrid)DarwinDesktop, mobile
WindowsNT (hybrid)Windows 10/11Desktop, enterprise servers
AndroidCustom LinuxAOSPMobile
RTOSReal-time microkernelFreeRTOS, VxWorks, ZephyrEmbedded, industrial control

AI infrastructure runs on Linux virtually 100% of the time. The Linux kernel spans 30 million lines of source code, but the core subsystems you interact with day-to-day number only five or six: process scheduling, memory management, VFS, the network stack, block devices, and character devices.

Monolithic Kernel vs. Microkernel

Monolithic kernel (Linux): the file system, network stack, and drivers all live in kernel space and communicate via direct function calls. Fast, but a bug in any subsystem can crash the entire machine.

Microkernel (seL4, Mach): the kernel retains only the most essential IPC, scheduling, and memory-protection primitives; all other services run in user space and communicate through message passing. More resilient, but IPC overhead is significant.

In practice, most modern OSes are hybrids — Linux itself has grown loadable kernel modules (LKMs) and eBPF as dynamic mechanisms, nudging it incrementally toward the microkernel direction.

Real-Time OS

The primary metric for an RTOS is not throughput but worst-case response latency. Scheduling must be predictable — a high-priority task that becomes ready must receive CPU time within a bounded interval. Linux is not an RTOS by default, but the PREEMPT_RT patch brings it close to real-time capability. Industrial control, automotive electronics, and robotics all require an RTOS.

2.2.2Processes and Scheduling

Scope: processes are the basic unit of CPU resource allocation; the scheduler decides who runs and who waits. Focus on CFS and modern scheduling challenges.

Processes and Threads

In Linux, processes and threads look the same to the kernel — both are represented by a task_struct. The creation call determines whether it is a process (fork) or a thread (clone with flags like CLONE_VM). The essential distinction: processes have independent address spaces; threads share one.

A task_struct contains:

  • - PID, parent process, user/group information
  • - Run state (RUNNING, INTERRUPTIBLE, UNINTERRUPTIBLE, ZOMBIE, ...)
  • - Scheduling policy and priority
  • - Open file table, signal handler table
  • - Memory map (mm_struct)
  • - Context (registers, stack)

fork and Copy-on-Write

fork() creates a child process; the traditional semantics copy the entire address space. A naive implementation would be very slow — because the vast majority of child processes immediately exec a new program, making the copied memory useless from the start.

Copy-on-write (COW) is the elegant solution: at fork time nothing is copied; all pages are simply marked read-only with their reference count incremented. When either the parent or child writes to a page, a page fault fires and only then does the kernel copy that one page. Result: fork is nearly as fast as spawning a new thread.

Orphans and Zombies

If the parent dies before the child, the child becomes an orphan process and is adopted by init (PID 1).

If the child dies before the parent calls wait, the child becomes a zombie process — the kernel retains its PID and exit status until the parent reads them. Zombies consume no memory but do occupy a PID table entry. Orphan vs. zombie is a classic Unix interview question.

CFS: Linux's Default Scheduler

The core idea of CFS (Completely Fair Scheduler) is the "ideal multi-tasking CPU" — if the CPU were infinitely fast, $N$ tasks should each receive exactly $1/N$ of the compute simultaneously. The real CPU is time-shared; CFS approximates this ideal.

Implementation: each task maintains a virtual runtime (vruntime); each time it is scheduled, vruntime += actual run time × weight. The scheduler always picks the task with the lowest vruntime. Tasks are organized in a red-black tree, so selecting the next task costs $O(\log n)$.

The nice value ([-20, 19]) influences the rate at which vruntime advances via weight: a lower nice value means higher weight, slower vruntime growth, and more frequent selection. This is a smooth implementation of "priority."

Note: Since Linux 6.6 (late 2023), the default fair scheduler has switched from CFS to EEVDF (Earliest Eligible Virtual Deadline First). EEVDF preserves virtual-time fairness while introducing virtual deadlines, improving tail latency and reducing scheduling jitter for interactive tasks. The underlying data structure is still a red-black tree, so $O(\log n)$ complexity is unchanged.

Real-Time Scheduling

Beyond CFS, Linux also has real-time scheduling classes (SCHED_FIFO, SCHED_RR). FIFO has no time slice — a high-priority task runs until it voluntarily yields or blocks, making it suitable for hard real-time scenarios. RR is FIFO with time-slice round-robin.

There is also DEADLINE scheduling (EDF algorithm): each task declares (runtime, deadline, period), and the kernel guarantees that runtime is completed within each period without exceeding the deadline. This is currently the closest Linux comes to hard real-time scheduling.

Priority Inversion

A classic failure mode: low-priority task L holds a lock; medium-priority task M preempts L (without releasing M's hold on L); high-priority task H waits for the lock — H is indirectly blocked by M, a priority inversion. The 1997 Mars Pathfinder mission suffered exactly this bug.

Fix: priority inheritance — while L holds the lock, it is temporarily elevated to H's priority so M cannot preempt it. Priority ceiling — statically assigns a higher ceiling. Both mechanisms are implemented in RT-mutex.

Cost of Context Switching

A single context switch costs roughly 1–5 μs and involves: saving registers, switching the page table (reloading CR3, partially invalidating the TLB), switching the kernel stack, and the scheduling decision. That sounds fast, but accumulated overhead is significant under a 1,000 Hz timer interrupt with frequent preemption.

In AI training, CPU affinity (via taskset or sched_setaffinity) is commonly used to pin DataLoader workers to fixed cores, reducing both context switches and cache misses.

2.2.3Memory Management

Scope: virtual memory, page tables, allocators, HugePages, OOM — the section most worth digging into in the entire chapter.

The Virtual Memory Contract

Every process sees an independent, contiguous address space that appears to belong entirely to it (128 TB of user space on x86-64). This is the illusion that virtual memory provides. Hardware (the MMU) and the OS (kernel page-table management) cooperate to maintain it.

4-level page tables (x86-64): a 48-bit virtual address is split into 9+9+9+9+12 bits. The first four 9-bit groups index four levels of page tables; the final 12 bits are the offset within the page. Why this design? A single-level page table covering the entire address space would require $2^{36} \times 8$ B = 512 GB per process — clearly impractical. Multi-level tables allocate on demand — only virtual memory ranges that are actually used have page table entries.

TLB

Walking all four levels of page tables on every memory access means four extra memory accesses — far too slow. Hardware uses a TLB (Translation Lookaside Buffer) to cache recently used virtual-to-physical mappings. A TLB hit incurs ~0 overhead; a miss triggers a full page walk costing ~100 cycles.

A TLB holds only tens to hundreds of entries — random access across a large array causes the TLB miss rate to spike.

HugePages

Normal pages are 4 KB; 512 TLB entries cover 4 KB × 512 = 2 MB of data. HugePages are 2 MB (THP) or 1 GB (hugetlbfs), so a single TLB entry covers 2 MB or 1 GB. The benefit is pronounced for large-memory workloads such as databases and AI training.

DPDK, databases, and the JVM all strongly recommend HugePages — not to save memory, but to reduce TLB misses.

Physical Memory Allocation

The kernel's physical page allocator is the Buddy System: physical memory is divided into power-of-two-sized blocks (4 KB, 8 KB, 16 KB, … 4 MB), with a free list for each size. On allocation, the smallest block greater than or equal to the request is found; if it is too large, it is split in half (into buddies) — one half goes to the requester and the other enters the next-smaller free list. On release, the kernel checks whether the buddy is also free; if so, they are merged.

This algorithm balances avoiding external fragmentation with fast allocation speed. But 4 KB granularity is too coarse for small objects (tens of bytes), so a second layer is added:

Slab/Slub: a dedicated allocator for kernel objects. A set of pages is pre-allocated as a "slab" and divided into fixed-size objects (e.g., a task_struct at 2 KB), with a free list maintained at object granularity. Allocation and deallocation are O(1) and cache-friendly.

User-Space malloc

User-program malloc does not call the kernel directly. glibc's ptmalloc, jemalloc, and tcmalloc all obtain large blocks from the kernel via brk() or mmap() and then manage small-object allocation themselves. The core idea behind jemalloc (Redis, Rust default) and tcmalloc (Google) is per-thread caches — each thread maintains a local small-object cache so allocation is lock-free; the global pool is consulted only when the cache is exhausted. This is far faster than ptmalloc under multi-threaded workloads.

Page Reclamation

When physical memory runs low, the kernel must reclaim pages. Linux maintains two LRU lists: active and inactive. Accessing a page promotes it to the head of the active list; pages that go unaccessed for a long time migrate to the tail of the inactive list. Reclamation begins from the inactive tail.

The kswapd kernel thread reclaims pages in the background, maintaining a certain amount of free memory. The reclamation paths are:

  1. Clean page-cache pages: discarded immediately (re-read from file on next access)
  2. Dirty page-cache pages: written back to disk first, then freed
  3. Anonymous memory (heap, stack): swapped out to the swap partition (if enabled)

Swap is commonly disabled on server machines — an OOM kill is preferable to letting a program silently degrade in performance. AI training machines must have swap disabled.

OOM Killer

When even swap cannot help, the kernel invokes the OOM Killer to select and terminate a process. The scoring algorithm weighs memory consumption, running time, and oom_score_adj (tunable). In practice: critical services should be given a negative oom_score_adj to reduce the chance of being killed.

In AI

Most AI training memory problems arise in GPU VRAM; CPU memory is relatively abundant. However, these scenarios trip teams up frequently:

  • - fork with multiple DataLoader workers: Due to COW, large objects in the parent process (such as dataset indices) are copied page by page as child processes "touch" them, eventually duplicating all of that memory. Python multi-process training OOMs are often caused by this.
  • - page cache overwhelming low-memory machines: Reading large datasets fills memory with page cache; on some kernel versions it is not released quickly enough.
  • - zero-copy requirements: Ideally, training data travels from disk to GPU without the four-copy path of "disk → kernel → user space → GPU." Combining mmap + O_DIRECT + GPU pinned memory can reduce the number of copies.
# PyTorch pinned memory example: enables DMA for host→device copies
train_loader = DataLoader(
    dataset,
    batch_size=32,
    num_workers=8,
    pin_memory=True,   # allocate page-locked memory; GPU can DMA directly
)
# Pair with non_blocking=True in the training loop for asynchronous copies
for x, y in train_loader:
    x = x.cuda(non_blocking=True)
    y = y.cuda(non_blocking=True)

2.2.4Storage and File Systems

Scope: VFS abstraction, inodes, page cache, I/O scheduling — the foundation for loading large datasets.

VFS: Everything Is a File

One of Unix's core philosophies. Files, directories, pipes, sockets, devices, and /proc information — all are accessed via open/read/write/close. VFS (Virtual File System) is the unified interface layer provided by the kernel, beneath which different concrete file systems are mounted: ext4, xfs, btrfs, overlayfs (containers), nfs (network).

inode

An inode is the metadata object for a file, containing: size, owner, permissions, timestamps, and pointers to data blocks. The filename is not in the inode — filenames are stored in the directory (which is itself a file whose content is a name-to-inode mapping).

Hard link: two directory entries point to the same inode; the inode's reference count is incremented. Deleting a file actually removes a directory entry and decrements the reference count; the file is truly freed only when the count reaches zero.

Symbolic link (symlink): a special file type whose content is the target path. Resolved recursively.

Page Cache

When a file is read, the kernel caches the data in the page cache (in physical memory). A second read of the same data is served directly from cache with no disk I/O. Writes also go to the page cache first by default (write-back), with the kernel flushing to disk in the background.

The "buff/cache" shown by the free command is the page cache. It is not "wasted" memory — it is memory put to efficient use and can be reclaimed at any time.

I/O Scheduling

In the era of spinning disks, I/O schedulers (CFQ, deadline) focused on merging adjacent requests and minimizing seek time. With SSDs, seek cost has disappeared; Linux now defaults to mq-deadline or none (NVMe multi-queue with direct dispatch).

Synchronous vs. Asynchronous I/O

Standard read/write is synchronous and blocking. High-performance scenarios use:

  • - O_DIRECT: bypasses the page cache and DMA's directly into the user buffer. Commonly used by databases.
  • - aio (Linux AIO): traditional asynchronous I/O, only truly asynchronous under O_DIRECT; limited capabilities.
  • - io_uring (2019+): modern asynchronous I/O using shared ring-buffer submission/completion queues, capable of syscall-less I/O in polling mode. AI training data loaders, databases, and high-performance web servers are all migrating to it.
// io_uring core ring-buffer structure (simplified)
struct io_uring_sqe {  // Submission Queue Entry
    __u8  opcode;      // IORING_OP_READ, _WRITE, ...
    __s32 fd;
    __u64 off;
    __u64 addr;
    __u32 len;
    // ...
};
struct io_uring_cqe {  // Completion Queue Entry
    __u64 user_data;
    __s32 res;
    __u32 flags;
};
// User space fills SQEs and reads CQEs directly — no syscall needed (in polling mode)

In AI

Training datasets can easily reach terabytes, making load performance critical. A few practical points:

  • - Packed formats like WebDataset and TFRecord: pack tens of millions of small files into a small number of large files, avoiding inode/directory-traversal overhead.
  • - Prefetching + multiple workers: DataLoader's prefetch_factor controls how many batches each worker prefetches ahead.
  • - Beyond POSIX semantics: in distributed training, backends such as NFS, Lustre, and S3 have performance profiles that differ drastically from local disks — you must benchmark to know where the bottleneck is.

2.2.5Exceptions and Interrupts

Scope: the mechanism by which the CPU communicates with the outside world — hardware interrupts, soft interrupts, tasklets, and workqueues.

Interrupt Classification

Hardware interrupts (IRQ): triggered by hardware; the CPU immediately suspends its current work and jumps to the interrupt handler. An interrupt vector table maps each interrupt number to a handler function.

Software interrupts / exceptions: triggered by the CPU itself — divide by zero, page fault, system call (a syscall is essentially a trap).

Two-Phase Interrupt Handling

There is an iron rule for interrupt handling: it must be fast — interrupt context cannot sleep or acquire potentially blocking locks. But events like a network packet arriving sometimes require complex processing (parsing protocols, routing, copying data).

Linux's solution is "top half + bottom half":

  • Top half: the interrupt handler itself. Does only the bare minimum — acknowledges the hardware, moves data into a memory buffer, schedules the bottom half. Runs with interrupts disabled; must be as short as possible.
  • Bottom half: interrupts re-enabled, schedulable. Three mechanisms:
MechanismCharacteristicsUse Cases
softirqStatic types, per-CPU, cannot sleepNetwork RX/TX, timers, block I/O
taskletBuilt on softirq, dynamically created; a given tasklet runs on only one core at a timeMedium-overhead drivers
workqueueKernel thread context, can sleepDeferred work that needs to sleep

The network packet receive path is a classic softirq user — after a hardware interrupt enqueues the skb, NET_RX_SOFTIRQ hands it to the protocol stack in the bottom half. This is why ksoftirqd frequently pins an entire core under high network load.

Interrupt Affinity and Load Balancing

By default, all interrupts may be handled by a single core (especially with cheap NICs). /proc/irq/<n>/smp_affinity can bind an interrupt to specific cores. Multi-queue NICs, combined with RSS, distribute different queues' interrupts across different cores for true parallelism.

In AI

AI relevance here is limited. However, GPU drivers are interrupt-intensive — when a CUDA kernel completes, the GPU fires an interrupt, and the driver notifies user space in the bottom half. Interrupt handling overhead is non-negligible in high-QPS inference services (thousands of requests per second).

2.2.6Concurrency, Mutual Exclusion, and Synchronization

Scope: the basic tools for multi-core concurrency — atomics, locks, RCU, memory model. Worth digging into deeply.

The Nature of Races

When multiple threads read and write shared data without synchronization, things go wrong. The simplest counterexample:

int counter = 0;
// Two threads, each running:
for (int i = 0; i < 1000000; i++) counter++;
// Final counter is much less than 2000000

counter++ is not atomic — it is actually three steps: load, add, store. When two threads interleave these steps, updates are lost.

Atomic Operations

Hardware provides atomic instructions: the LOCK prefix on x86, LDREX/STREX on ARM. C11's _Atomic, C++11's std::atomic, and Rust's AtomicUsize are all language-level wrappers around these instructions.

The most important atomic primitive is CAS (Compare-And-Swap):

// Atomically: if *ptr == expected, write new and return true;
// otherwise, update expected with *ptr and return false
bool cas(int *ptr, int *expected, int new);

CAS is the foundation of all lock-free data structures. The typical pattern is a "loop until CAS succeeds":

int old, new;
do {
    old = *counter;
    new = old + 1;
} while (!cas(counter, &old, new));

Types of Locks

Spinlock: busy-waits on failure. Suitable for extremely short critical sections on multi-core systems. Must be used in kernel interrupt handlers (cannot sleep).

Mutex: sleeps on failure and is woken when the lock becomes available. Suitable for longer critical sections. Higher overhead than a spinlock (context switch), but does not waste CPU cycles.

Read-write lock: multiple readers share access; writers get exclusive access. Advantageous in read-heavy, write-rare scenarios. Downside: writer starvation (a continuous stream of readers can keep a writer waiting indefinitely).

Semaphore: a lock with a count, allowing N concurrent holders. Commonly used for resource pools.

Condition variable: not a lock, but a primitive for "waiting until a condition becomes true." Typically used together with a mutex.

RCU

Read-Copy-Update is the most elegant concurrency primitive in the Linux kernel. It is suited for scenarios with overwhelmingly more reads than writes. The core idea:

  1. Readers take no lock at all — they read the old-version pointer directly
  2. The writer copies the data, modifies the copy, then atomically replaces the pointer
  3. The old version is freed with a delay — only after all readers that had already started finish (the grace period)

The RCU read path has near-zero overhead, almost identical to single-threaded access. The cost is write complexity, a temporary doubling of memory, and delayed reclamation. Linux's routing table, namespace structures, and module list all use RCU.

// RCU reader: completely lock-free
rcu_read_lock();
struct foo *p = rcu_dereference(global_ptr);
// use p...
rcu_read_unlock();

// RCU writer
struct foo *new = kmalloc(...);
*new = (struct foo){ .x = 42, ... };
struct foo *old = rcu_dereference(global_ptr);
rcu_assign_pointer(global_ptr, new);   // atomic pointer replacement
synchronize_rcu();                      // wait for all readers to finish
kfree(old);                             // safe to free

Memory Model

Programmers assume their code executes in order, but both CPUs and compilers reorder instructions. In a single thread this is imperceptible (data dependencies are preserved), but in a multi-threaded context reordering can make another core observe an "impossible" state.

Example:

// Thread A:                Thread B:
x = 1;                       if (flag) {
flag = 1;                        assert(x == 1);  // may fail!
                             }

The CPU or compiler may move flag = 1 ahead of x = 1 — Thread B sees flag=1 but x is still 0.

Fix: memory barriers. x86 uses the TSO (Total Store Order) memory model, which is relatively strong — stores are ordered, but loads can be reordered. ARM is weakly ordered — almost anything can be reordered, and explicit barriers are mandatory.

Higher-level language abstractions: C++11's std::memory_order and Rust's Ordering offer five levels: Relaxed, Acquire, Release, AcqRel, and SeqCst. SeqCst is the strongest (sequentially consistent) and most expensive; Relaxed is the weakest (only atomicity is guaranteed) and fastest. Using these correctly is the hallmark of a high-level systems engineer.

MESI and Cache Coherence

Multi-core cache coherence is guaranteed by the hardware MESI protocol. Cache lines migrate between cores through Modified/Exclusive/Shared/Invalid state transitions.

A frequently overlooked observation: the cost of a lock is not "the lock primitive itself" — it is cache-line migration. After core A holds a lock and core B then contends for it, the lock variable's cache line migrates from A to B, costing ~100 cycles per migration. This explains why performance collapses under heavy lock contention.

Lock-Free Data Structures

Lock-free queues, stacks, and hash tables exist but are hard to write correctly — subtle mistakes lead to ABA problems and memory reclamation hazards. Industrial-grade implementations (Folly, Crossbeam) have been tuned over many years, but writing your own lock-free structures is not worth the risk.

In AI

PyTorch's internals make extensive use of lock-free atomics (Tensor reference counting, parameter server updates). The c10::intrusive_ptr in the C10 library is faster than std::shared_ptr because it embeds the reference count, avoiding an extra allocation.

Parameter servers and asynchronous SGD in training clusters are concurrency problems through and through. NCCL synchronization for single-machine multi-GPU training also implicitly involves memory barriers — ensuring that AllReduce results are visible to all ranks.

2.2.7Device Management and System Calls

Scope: syscalls are the only interface between user space and the kernel — io_uring and eBPF are reshaping it.

Device Driver Model

Linux divides devices into three classes:

  • Character devices: streaming reads and writes (serial ports, keyboard, /dev/random)
  • Block devices: random block access (HDDs, SSDs)
  • Network devices: do not use read/write; use the socket API

A driver registers an operations table (file_operations) with the kernel — function pointers for open, read, write, ioctl, mmap, and others. Applications reach the device through the file system (/dev/xxx); calls are dispatched to the driver.

DMA and Zero-Copy

DMA (Direct Memory Access) lets a device read and write memory without involving the CPU. The traditional network receive path: NIC DMA's into a kernel buffer → kernel copies into the socket buffer → user-space read copies again into the user buffer. Three copies total.

Zero-copy techniques: sendfile (file to socket entirely within the kernel), splice (zero-copy through a pipe), mmap+write (maps the file into user space). Nginx and Kafka use sendfile extensively.

System Calls

A syscall is the only way user space enters the kernel. x86-64 uses the syscall instruction (the old int 0x80 is obsolete), with arguments passed in specific registers. One syscall costs ~100 ns — not slow, but the cumulative cost becomes significant across tens of millions of iterations in a loop.

vDSO (virtual Dynamic Shared Object): implements a few high-frequency, lightweight syscalls (such as gettimeofday) as user-space functions, avoiding a kernel entry. Provided transparently through dynamic linking.

eBPF

eBPF (extended Berkeley Packet Filter) is the most important new mechanism added to Linux in recent years. It allows users to inject verified small programs into kernel hot paths (networking, tracing, security) without modifying the kernel or loading a module.

Applications: tcpdump uses BPF for packet filtering; bpftrace for dynamic tracing; Cilium for container networking (replacing iptables); Katran for Layer-4 load balancing (Facebook). AI observability platforms (fine-grained monitoring of training jobs) are increasingly adopting eBPF.

eBPF programs pass through a verifier that guarantees: no loops, no out-of-bounds accesses, bounded execution time. This makes them safe to run in kernel space.

Containers

Containers are not virtual machines — they are ordinary Linux processes that happen to use namespaces (for view isolation) and cgroups (for resource limits).

  • namespaces: eight kinds — PID, net, mount, IPC, UTS, user, cgroup (4.6+), and time (5.6+) — each isolating one dimension of resource visibility. A container seeing itself as PID 1 is the effect of the PID namespace.
  • cgroups: enforce quotas on CPU, memory, block I/O, and network bandwidth. Kubernetes resource limits are translated directly into cgroup settings.

Docker, containerd, and Podman are all higher-level wrappers around these two primitives. AI training almost always runs inside containers (for reproducibility and scheduling); understanding namespaces and cgroups is indispensable when troubleshooting performance or permission issues.

PART THREE

2.3 Programming Languages & Compilers

Language determines what you can think; the compiler determines how fast you can run.

2.3.1The Rust Language

Position: ownership model + no GC + zero-cost abstractions — the new baseline for systems programming.

Ownership: The Core of Rust

Rust's most distinctive feature is its ownership system. Three rules:

  1. Every value has exactly one owner.
  2. When the owner goes out of scope, the value is automatically dropped (resources freed).
  3. Assignment or passing a value as an argument moves ownership by default (unless the type implements Copy).

These rules eliminate three classic C/C++ bugs at the root: use-after-free, double-free, and memory leaks. The compiler performs static checks at compile time with no runtime overhead.

fn main() {
    let s1 = String::from("hello");
    let s2 = s1;           // ownership moves; s1 is no longer valid
    // println!("{}", s1); // compile error!
    println!("{}", s2);    // OK
} // s2 goes out of scope; the string is automatically freed

Borrowing: Temporary Access Without Moving

Moving ownership everywhere would be cumbersome — borrowing lets you "use it temporarily without owning it":

fn len(s: &String) -> usize { s.len() }

fn main() {
    let s = String::from("hello");
    let n = len(&s);       // pass an immutable borrow
    println!("{}, len {}", s, n);  // s is still valid
}

There are two kinds of borrows: immutable &T (multiple can coexist) and mutable &mut T (exclusive). The core rule: a mutable borrow is mutually exclusive with any other borrow. This is enforced at compile time, directly eliminating data races — if concurrent code compiles, it is (at the ownership + borrow level) race-free.

Lifetimes

A borrow must not outlive the value it borrows from. Most of the time the compiler infers this automatically; complex cases require explicit annotations:

// 'a is a lifetime parameter: the returned borrow cannot outlive
// the shorter of the two inputs
fn longer<'a>(s1: &'a str, s2: &'a str) -> &'a str {
    if s1.len() > s2.len() { s1 } else { s2 }
}

Lifetimes are the first mental hurdle for Rust beginners. Once you clear it, you realize how much "is this pointer still valid?" in C/C++ relies on intuition and convention — Rust turns it into the compiler's job.

Zero-Cost Abstractions

Rust's traits, iterators, and generics are zero-cost abstractions: after compilation they run as fast as hand-written C. An iterator chain like v.iter().filter(...).map(...).sum(), after LLVM optimization, produces assembly essentially identical to a hand-written for loop — no virtual dispatch overhead, no boxing. This is fundamentally different from the abstractions in Java or Python.

No GC, But No Manual malloc/free Either

Rust has no garbage collector — the ownership model makes memory management a compile-time decision. The compiler knows every drop point; resource cleanup is implemented in the Drop trait (similar to C++ RAII). Box<T>, Rc<T>, and Arc<T> handle exclusive heap allocation, single-threaded reference counting, and multi-threaded reference counting, respectively.

Async

Rust's async/await produces zero-cost Futures — each async function compiles into a state machine. The runtime (tokio, async-std) is responsible for scheduling and advancing state machines. This lets Rust async squeeze out performance close to C, though the ecosystem is still maturing.

In AI

PyTorch is Python + C++, not Rust. But Rust is heavily used in AI infrastructure surrounding it: tokenizers (HuggingFace), burn (a pure-Rust deep learning framework), candle (a Rust inference framework), certain components of vLLM, and various vector databases (Qdrant, LanceDB).

Rust-Python interop is bridged via PyO3, now the preferred way to write Python extensions — far safer than writing C extensions.

2.3.2The Modern Evolution of C

Position: C is still the general-purpose low-level language — let's look at how it's evolving, and at C++ as an alternative path for systems programming. Brief.

This section covers the broader "C-family evolution." C is not dead — kernels, drivers, DPDK, and Redis are still written in C. But the C toolchain and its ecosystem keep improving:

  • Sanitizer suite: AddressSanitizer (ASan) detects out-of-bounds accesses and use-after-free; ThreadSanitizer (TSan) detects data races; UBSan detects undefined behavior. Enable with -fsanitize=address at compile time; runtime overhead is 2–5×, but it catches the vast majority of memory bugs.
  • Static analysis: Clang Static Analyzer, Coverity, PVS-Studio. Catches code that compiles cleanly but fails at runtime.
  • Fuzz testing: libFuzzer, AFL++. Automatically generates inputs to probe for crashes. OpenSSL and SQLite have both found large numbers of bugs through fuzzing.

C++ in Systems Programming

C++ is far more expressive than C, but its complexity has exploded. Modern C++ (C++17/20/23) makes writing contemporary code much more pleasant with std::unique_ptr, std::optional, ranges, and more. The main problem is that dangerous features cannot be disabled — the safe subset of C++ exists, but must be enforced by discipline.

Performance-wise, C++ and Rust are close (both have zero-cost abstractions, neither has a GC). The fundamental difference is that Rust performs ownership checks at compile time, while C++ relies on the programmer. Databases, browsers, and game engines still make heavy use of C++ (LLVM, Chromium, Unreal).

Language Selection Considerations

If you were starting a network device firmware, operating system, or database from scratch today, the general guidance on language choice is:

  • - Kernel/drivers: C (mainstream on Linux) or Rust (Rust for Linux has been merged into the mainline; multiple subsystem drivers have been rewritten in Rust; the ecosystem should mature after 2025).
  • - Middleware/services: Rust first, Go second (GC is acceptable), C++ when the team has experience.
  • - Upper-layer applications: Go, Python, Java.

2.3.3Compiler Fundamentals

Position: lexical/syntactic/semantic analysis → IR → code generation. LLVM as the main thread.

The Classic Three-Stage Compiler

Front end: source code → AST → intermediate representation (IR)

  • - Lexical analysis (lexing): character stream → token stream. Tools: flex, or hand-written.
  • - Syntactic analysis (parsing): token stream → AST. Tools: bison, ANTLR, or hand-written recursive descent.
  • - Semantic analysis: type checking, scope resolution, generating a type-annotated AST.
  • - IR generation: AST → IR (e.g., LLVM IR).

Middle end: optimizations on the IR (next section).

Back end: IR → target assembly. Includes instruction selection, register allocation, and instruction scheduling.

LLVM and SSA

LLVM's dominance rests on two things: its modular design (front end and back end decoupled) and its SSA (Static Single Assignment) IR.

SSA requires that every variable be assigned exactly once. Traditional code:

x = 1;
x = x + 2;
y = x * 3;

In SSA form:

x1 = 1
x2 = x1 + 2
y1 = x2 * 3

Each variable has exactly one definition, making def-use analysis trivial — "which definition does this use refer to?" is immediately obvious. At control-flow merge points, φ functions (phi) select the value from the appropriate path:

if (cond) x1 = 1;
else      x2 = 2;
x3 = φ(x1, x2);
y  = x3 + 1;

SSA lets all dataflow optimizations (constant propagation, dead-code elimination, common subexpression elimination) be implemented with simple algorithms — that is LLVM's strength.

LLVM IR Example

// C source: int add(int a, int b) { return a + b; }

// LLVM IR (SSA form)
define i32 @add(i32 %a, i32 %b) {
entry:
  %sum = add i32 %a, %b
  ret i32 %sum
}

LLVM IR is platform-independent — this IR can be compiled by different back ends (x86, ARM, RISC-V, GPU). This is the architectural foundation of cross-platform compilers.

Back End

Three main tasks for the back end:

  1. Instruction selection: IR instructions → target architecture instructions. One IR instruction may map to multiple assembly instructions (or vice versa). Common algorithms: tree pattern matching, BURS.
  2. Register allocation: IR is written assuming an unlimited number of registers; the back end must map them to a finite set of physical registers. The classic algorithm is graph coloring — color the register interference graph so variables with different colors can share a physical register. LLVM uses a more sophisticated linear-scan variant.
  3. Instruction scheduling: On out-of-order CPUs, reordering instructions at compile time reduces pipeline stalls.

In AI

AI compilers (TVM, MLIR, XLA) apply the ideas of general-purpose compilers to tensor computation. MLIR (Multi-Level IR) is an extension of the LLVM project that provides multiple levels of intermediate representation — from PyTorch's ATen operator level, to the HLO tensor level, to the affine loop level, to the LLVM level — all unified in a single framework. PyTorch 2.0's TorchDynamo + Inductor and Triton's MLIR path are both products of this trend.

2.3.4Compiler Optimizations

Position: dozens of optimizations working together can make code several times faster — with direct implications for AI kernel optimization.

Local Optimizations

Constant propagation: x = 5; y = x + 3; → y = 8;

Constant folding: Evaluate at compile time what can be evaluated at compile time. 86400 * 30 is not left to be multiplied at runtime.

Common subexpression elimination (CSE):

a = x * y + 1;
b = x * y + 2;
// →
tmp = x * y;
a = tmp + 1;
b = tmp + 2;

Dead code elimination: Remove computations whose results are never used.

Strength reduction: Replace expensive operations with cheaper ones. x * 8 → x << 3; replace i*c in loops with an accumulating variable.

Intraprocedural Optimizations

Inlining: Replace a function call with the function body. This eliminates call/return overhead, and more importantly, exposes the inlined code and its call-site context to other optimizations — "inlining opens the optimization window."

Cost: code bloat. The compiler uses heuristics to decide whether to inline: always inline small functions, inline hot functions, inline functions called only once. The inline keyword is merely a hint; the final decision belongs to the compiler.

Loop optimizations (important):

  • - Loop unrolling: for (i=0;i<4;i++) a[i]=0; → 4 store instructions. Reduces branch overhead and exposes parallelism.
  • - Loop fusion: Merge two adjacent independent loops into one. Improves cache locality.
  • - Loop fission: Split one large loop into several smaller ones. Enables vectorization and exposes parallelism.
  • - Loop interchange: Swap the order of nested loops so the inner loop accesses memory contiguously.
  • - Loop-invariant code motion (LICM): Hoist computations that don't change inside the loop to outside it.
  • - Vectorization: Transform scalar loops into SIMD instructions. for (i=0;i under AVX2 becomes a vector instruction that processes 8 floats at a time.

Interprocedural Optimizations

Optimizations that cross function boundaries: inlining, interprocedural constant propagation, dead function elimination. Require whole-program visibility — LTO (Link Time Optimization) lets the linker see all IR before optimizing, which is more powerful than the traditional approach of compiling each .o independently.

PGO

Profile-Guided Optimization: first compile a version with profiling enabled, run it under a real workload to collect a profile, then recompile — letting the compiler use the profile to drive optimization decisions (which branches are more likely taken, which functions are hot). Chromium and MySQL both use PGO; it typically yields a 5–15% performance improvement.

Limits of Auto-Vectorization

The compiler's auto-vectorization is powerful but not unlimited. It gives up in the following situations:

  • - Data dependencies (loop body contains a[i] = a[i-1] + 1)
  • - Uncertain pointer aliasing (the restrict keyword can help the compiler)
  • - Function calls blocking it (the pure attribute can help)
  • - Complex branches

So for performance-critical paths, inspecting the compiler-generated assembly (gcc -S or Compiler Explorer) is an essential skill for experts.

In AI

Hand-tuning AI kernels is applying these compiler optimization ideas to tensor computation:

  • - Operator fusion = loop fusion + inlining, reducing memory round-trips through intermediate tensors.
  • - Tiling = loop blocking, keeping data resident in cache/SRAM.
  • - Vectorization to SIMD/Tensor Cores = the tensor version of auto-vectorization.
  • - Auto-tuning (TVM, Triton AutoTune) = the PGO idea — empirically search different parameters for the best configuration.

Once you understand general compiler optimizations, AI compilers are just the same ideas applied to a specific domain.

PART FOUR

2.4 Databases

An increasingly important piece of AI infrastructure — from relational databases to vector databases.

2.4.1Database Fundamentals

Position: storage engines, transactions, query optimization, logging — the four pillars of relational databases.

Storage Engines: B+Tree vs LSM-Tree

Two mainstream index architectures.

B+Tree (MySQL InnoDB, PostgreSQL): a balanced multi-way tree whose leaf nodes hold data pages linked in a list (convenient for range scans). Reads are fast ($\log_B n$ per lookup); updates are in-place. Downside: frequent random writes are unfriendly to SSDs.

LSM-Tree (RocksDB, Cassandra, LevelDB): Log-Structured Merge Tree. Writes first land in memory (memtable); when full, the memtable is flushed as an immutable SSTable to disk. Background compaction merges old SSTables. Reads may need to check multiple levels and consult bloom filters. Writes are extremely fast (purely sequential I/O); reads are somewhat slower; write amplification is the main concern.

Choosing: read-heavy OLTP → B+Tree; write-intensive (logs, time-series) → LSM.

Row Storage vs Column Storage

Row storage (OLTP): a row's data is physically contiguous. Fast for reading or writing whole rows. MySQL, PostgreSQL.

Column storage (OLAP): a column's data is physically contiguous. OLAP queries typically read only a few columns (SELECT avg(salary) FROM emp) and have high filter selectivity; columnar storage dramatically reduces I/O. Achieves high compression ratios (values in the same column are similar). ClickHouse, Parquet, DuckDB.

ACID Transactions

A (Atomicity): a transaction either fully completes or has no effect. C (Consistency): transactions preserve data constraints. I (Isolation): concurrent transactions do not interfere with each other. D (Durability): once committed, data is not lost.

Implementation relies on two mechanisms: locking (enforcing I) and logging (enforcing A and D).

Locking and Isolation Levels

The SQL standard defines four isolation levels, each preventing a stricter set of anomalies:

Read CommittedNoPossiblePossible

In practice, implementations use 2PL (Two-Phase Locking) or MVCC (Multi-Version Concurrency Control). MVCC is the modern mainstream: each write creates a new version; readers see the snapshot from when their transaction began, so reads and writes do not block each other. PostgreSQL, MySQL InnoDB, and Oracle all use MVCC.

WAL and Recovery

WAL (Write-Ahead Log): write to the log before modifying data pages (sequential log I/O is faster than random page I/O). After a crash, redo committed transactions and undo uncommitted ones using the log. The ARIES algorithm is the industry-standard recovery protocol: Analysis → Redo → Undo.

Query Optimizer

SQL is declarative — it says "what," not "how." The optimizer translates SQL into an execution plan.

The process: SQL parsing → logical plan (relational algebra) → logical optimization (predicate pushdown, subquery rewriting) → physical plan (choosing concrete operators: hash join vs nested loop join, index scan vs full table scan) → cost estimation (based on statistics: table size, column distributions) → selecting the lowest-cost plan.

The optimizer is the "brain" of a database; a good one can outperform a bad one by an order of magnitude.

Vectorized Execution

The traditional execution engine uses the row-at-a-time Volcano model: each operator's next() returns one row. Simple, but per-row overhead is high (virtual dispatch, cache misses).

Vectorized execution: each next() returns a batch of rows (e.g., 1024). Overhead is amortized across the batch; a single SIMD instruction processes multiple rows; cache utilization improves. ClickHouse, DuckDB, and Apache Arrow are all vectorized — roughly 10× faster than the Volcano model.

In AI

In AI infrastructure, relational databases remain the primary store for metadata — users, model versions, training job records, and experiment tracking. PostgreSQL + MLflow is a common combination. But the databases most relevant to AI are the three covered in the following sections: graph, time-series, and vector.

2.4.2Graph Databases

Position: native storage of nodes and edges — the foundation for knowledge graphs, social networks, and reasoning chains.

Graph Storage

A relational database can store a graph (two tables: a node table and an edge table), but querying multi-hop relationships requires repeated joins and performs poorly. Graph databases use native storage for adjacency information — each node directly holds pointers to its outgoing edge list, so traversing k hops is k pointer dereferences.

The Property Graph Model

The mainstream graph model is the property graph: nodes have labels (types) and properties (key-value pairs); edges have direction, type, and properties. For example, a social graph: (Alice :Person {age: 30})-[:FRIEND {since: 2020}]->(Bob :Person).

Query languages: Cypher (Neo4j, most popular), Gremlin (TinkerPop, cross-database), SPARQL (RDF graphs, semantic web).

Cypher Example

// Find friends-of-friends of Alice who are not already Alice's direct friends
MATCH (alice:Person {name: 'Alice'})-[:FRIEND]->(friend)-[:FRIEND]->(fof)
WHERE NOT (alice)-[:FRIEND]->(fof) AND fof <> alice
RETURN DISTINCT fof.name

The equivalent SQL query requires self-joins and NOT EXISTS subqueries — worse in both readability and performance.

Common Graph Algorithms

  • BFS/DFS: the most fundamental traversals
  • Shortest path: Dijkstra (no negative weights), Bellman-Ford (handles negative weights), A* (with heuristic)
  • PageRank: importance of pages/nodes
  • Community detection: Louvain, Label Propagation
  • Connected components: Union-Find
  • Centrality: degree, betweenness, closeness centrality

2.4.3Time-Series Databases

Position: data streams ordered by timestamp — purpose-built for monitoring, IoT, and financial market data.

Characteristics of Time-Series Data

Time-series data differs from ordinary relational data in several ways:

  • - Ordered by time: the time dimension is a natural index
  • - Append-heavy: old data is rarely modified
  • - High write rate: monitoring may generate millions of points per second
  • - Fixed query patterns: range aggregation, downsampling, time windows
  • - Data has a lifecycle: old data is downsampled or deleted

Key Technology: Compression

Adjacent values in time-series data vary little, enabling very high compression ratios. Two classic techniques:

Delta-of-Delta (timestamp compression): store Δ(Δt). For regularly sampled data, Δ is nearly constant, so ΔΔ is mostly zero; variable-length encoding reduces it to a few bits.

XOR encoding (value compression): XOR adjacent floating-point values; high-order bits are mostly zero. The compression algorithm from the Gorilla (Facebook) paper achieves 1.37 bytes per point in practice — a 10×+ compression over the raw 16 bytes (8 bytes each for timestamp and value).

Storage Structure

Most time-series databases are based on LSM-tree variants. Write path:

  1. Form a series by (metric, tags)
  2. Each series is appended to independently in time order
  3. Partition by time window (e.g., one chunk per 2 hours)
  4. Compress chunks to disk as they age out

Downsampling

Raw 1-second-granularity data is retained for 7 days, then downsampled to 1-minute granularity for 30 days, then to 1-hour granularity for 1 year. Automatic continuous aggregates materialize the rollups, avoiding the cost of recomputing them at query time.

2.4.4Vector Databases

Position: nearest-neighbor search over high-dimensional vectors — the foundation of RAG, semantic retrieval, and recommendation.

Problem Definition

Given $N$ vectors of dimension $d$ and a query vector $\mathbf{q}$, find the $k$ vectors closest to $\mathbf{q}$. Distance is typically measured by L2 distance or cosine similarity.

Brute-force search is $O(Nd)$. With $N = 10^8$ and $d = 768$, each query takes several seconds — unacceptable. So the core of a vector database is approximate nearest neighbor (ANN) search: accept a small probability of returning a non-exact nearest neighbor in exchange for sub-millisecond query latency.

The ANN Algorithm Family

LSH (Locality-Sensitive Hashing): use random hyperplanes to partition the space into buckets; similar vectors fall into the same bucket with high probability. Simple, but recall is low.

IVF (Inverted File): cluster all vectors with k-means first; at query time, search only the nearest few clusters. Typical usage: IVF_PQ (vectors within clusters are stored using PQ quantization).

PQ (Product Quantization): split a $d$-dimensional vector into $m$ segments of $d/m$ dimensions each; independently quantize each segment with k-means into an 8-bit codebook index. For $d=128$, 8 segments of 256 classes each compress the original 512 bytes down to 8 bytes — a 64× reduction. Distance computation is approximated via lookup tables.

HNSW (Hierarchical Navigable Small World): a multi-layer graph index; the top layer is sparse, the bottom layer is dense. Queries start at the top layer and greedily "hop" toward the target. Recall and speed are best-in-class in most scenarios; it is currently the mainstream choice.

Intuition Behind HNSW

Think of all vectors as nodes in a graph, with each node connected to several nearby neighbors. A query starts at an entry point and at each step jumps to the neighbor closest to the query vector, until no neighbor is closer than the current node — a local optimum.

HNSW adds hierarchy: multiple graph layers, where each layer is a sparse subset of the layer below; edges at higher layers span greater distances, while edges at lower layers are short. A query first takes large steps at the top layer to approach the target region, then descends layer by layer for refinement. Think of it as a two-dimensional generalization of a skip list.

Pure vector retrieval underperforms keyword search on technical terms and exact matches. Hybrid retrieval fuses vector (semantic similarity) and BM25 (keyword) results using RRF (Reciprocal Rank Fusion) or other reranking methods. Industrial RAG systems almost universally use hybrid retrieval.

Filtering and Metadata

Real-world use cases require not just "similar," but also "only search within this user's documents" or "only results after 2024." This requires the vector database to support ANN with filters. Two approaches:

  • Pre-filter: filter by metadata first, then run ANN. If very few results survive the filter, brute-force search is used.
  • Post-filter: run ANN first, then filter. But this may return fewer than k results.

In AI

Semantic search, recommendation systems, image-text retrieval, and deduplication are all direct applications of vector databases. Long-term memory: an agent's conversation history is stored as vectors and retrieved on demand — an increasingly important pattern in LLM applications.

2.5.1High-Performance Message Bus

Position: The foundation of all inter-process/service communication patterns — RPC, Pub/Sub, message queues.

Classification of Communication Patterns

The underlying abstractions of middleware all revolve around a few fundamental communication semantics:

Core Technical Points of RPC

gRPC is the de facto standard for modern RPC. Key elements:

  • IDL (Interface Definition Language): Protobuf describes messages and services, automatically generating stubs for each language
  • Serialization: Protobuf is binary, compact, and cross-language. 3–10x smaller than JSON with faster parsing
  • Transport: HTTP/2 foundation — multiplexing, streaming, header compression
  • Streaming modes: Unidirectional/bidirectional streams
  • Interceptors: Authentication, rate limiting, tracing

The Loose Coupling of Pub/Sub

The design philosophy of Pub/Sub: producers and consumers are unaware of each other. Publishers deliver messages to a topic; subscribers register interest in topics to receive messages. Scaling the system means adding consumers or partitions.

Kafka is the canonical example of "persistent-log Pub/Sub." Each topic is a partitioned append-only log; consumers read by offset and can replay history. A log retention period controls storage.

Zero-Copy and High Performance

All high-performance message middleware employs zero-copy optimizations:

  • - User-space ring buffer with lock-free multi-producer multi-consumer support
  • - sendfile/splice (kernel-level zero-copy)
  • - mmap shared memory (single-machine IPC)
  • - RDMA (cross-machine)

In single-machine scenarios, a shared-memory ring queue with atomic pointers can achieve 20M+ msg/s. In cross-machine scenarios, RDMA can reach line rate.

Service Discovery

The mandatory question in distributed service communication: "What IP and port is the service I need to call?" — that's service discovery. Several approaches:

  • DNS: The simplest approach; TTL governs update speed
  • Client-side registries: Consul, Zookeeper, etcd — services register on startup, clients query
  • Service Mesh: Istio, Linkerd — a proxy intercepts traffic, transparently injecting discovery and governance
  • Kubernetes Service: Built-in DNS + kube-proxy

Reliability Guarantees

Three tiers of message delivery semantics:

  • at-most-once: Delivered at most once; may be lost — simple and high-performance
  • at-least-once: Delivered at least once; may be duplicated — requires idempotent consumers
  • exactly-once: Delivered exactly once — expensive, requires transactions or deduplication

Kafka's exactly-once holds within Kafka itself (via transactional messages); end-to-end exactly-once involving external systems remains difficult.

In AI

Message middleware sees considerable use in AI systems:

  • - Training job scheduling: Producers (users) submit jobs; consumers (training workers) process them
  • - Pre-inference queuing: A queue in front of high-concurrency LLM services smooths traffic spikes
  • - Data ingestion → feature engineering pipeline: Kafka as the intermediary
  • - Distributed RL: Actors generate experience; learners consume from the queue
  • - Agent tool-call bus: LLM → tool router → individual tool processes

2.6 Network Stack: DPDK & VPP

Two main paths to high-performance packet forwarding — DPDK for kernel bypass, VPP for vectorized processing.

2.6.1Bottlenecks of the Kernel Network Stack

Position: Why bypass the kernel — understanding the reason DPDK/VPP exists.

The Linux Network Stack Path

A TCP packet traveling from the NIC to the application must pass through:

flowchart TB NIC[Network Card] --> IRQ[Hardware IRQ] IRQ --> SI[Softirq NET_RX] SI --> NP[netif_receive_skb] NP --> IP[IP layer] IP --> TCP[TCP layer] TCP --> SKB[socket buffer] SKB --> SYS[syscall read] SYS --> APP[Application]

Every layer has a cost:

  • - Hardware interrupt overhead
  • - sk_buff struct allocation/deallocation (once per packet, heavy pressure on the memory allocator)
  • - Multi-layer protocol stack (Ethernet, IP, TCP) parsed layer by layer
  • - User/kernel space data copies
  • - System call overhead
  • - Scheduler invocation per packet

This architecture is superb for generality, but its performance ceiling is roughly 1–2 Mpps per core. To handle 10G small-packet line rate (14.88 Mpps) or 100G line rate (148.8 Mpps), the kernel stack simply cannot keep up.

Root Causes of the Bottleneck

Interrupt storms: At high packet rates, millions of hardware interrupts per second force the CPU to spend the majority of its time on interrupt handling. The NAPI mechanism partially mitigates this with an "interrupt fires → disable interrupts → poll" model, but is still limited.

Memory allocation: Receiving each packet requires a kmalloc for an sk_buff and a kfree when done. The allocator itself becomes the bottleneck.

Locks: The protocol stack contains global locks in many places (routing table, connection table). Multi-core scalability is poor.

Cache pollution: sk_buff migration across cores and the large protocol stack codebase cause frequent I-cache misses.

Two Paths to a Breakthrough

Bypass the kernel: User space directly manipulates the NIC. DPDK, Netmap, Snabb.

Optimize the kernel: XDP (eBPF filtering/forwarding at the driver layer), io_uring, AF_XDP.

The latter is maturing rapidly (XDP can reach 15 Mpps+ per core), but DPDK remains the preferred choice for maximum performance and is covered in detail in the next section.

2.6.2DPDK Architecture

Position: The industry-standard framework for user-space networking that bypasses the kernel.

DPDK's Design Philosophy

DPDK (Data Plane Development Kit) is a user-space networking framework open-sourced by Intel in 2013 (internal development began around 2010). Four core decisions:

  1. User-space driver: UIO/VFIO is used to unbind the NIC from the kernel; a user-space process directly mmaps the device registers.
  2. Polling instead of interrupts: The application main loop polls the receive ring continuously — no interrupt overhead.
  3. Pre-allocated huge-page memory: All required memory is allocated once at startup; no malloc at runtime.
  4. Per-core isolation: One thread per core, pinned to a fixed NIC queue, with minimal shared data.

Result: 10 Mpps+ per core with ease; single-server 100G line-rate forwarding is achievable.

Architecture Diagram

Key Components

EAL (Environment Abstraction Layer): Abstracts the OS and hardware, providing a unified interface (memory, CPU, devices). rte_eal_init() is the entry point of every DPDK program.

PMD (Poll Mode Driver): User-space NIC driver. One PMD per NIC type (Intel ixgbe, Mellanox mlx5, virtio, etc.).

mbuf: DPDK's equivalent of sk_buff, but much simpler — a fixed-size header plus data, no linked list. Pre-allocated in a mempool.

rte_ring: Lock-free MPMC (multi-producer multi-consumer) ring queue — the primary primitive for user-space IPC.

mempool: Object pool. Allocation/deallocation in O(1); per-core cache reduces contention.

    struct rte_mbuf *bufs[BURST_SIZE];

        uint16_t nb_rx = rte_eth_rx_burst(port, 0, bufs, BURST_SIZE);
        if (unlikely(nb_rx == 0)) continue;

        // Process packets... (omitted here)

        uint16_t nb_tx = rte_eth_tx_burst(port, 0, bufs, nb_rx);

        for (uint16_t i = nb_tx; i < nb_rx; i++)
            rte_pktmbuf_free(bufs[i]);

This code runs on a dedicated CPU core to which the NIC is bound and can easily reach several Mpps of throughput — 10–50x faster than a kernel-stack socket program.

The Cost

DPDK is not a free lunch:

  • - Dedicated CPU: The polling core runs at 100% utilization even when there are no packets
  • - No kernel protocol stack: TCP/UDP/ARP must be implemented yourself or via user-space stacks like mTCP or F-Stack
  • - Development complexity: Standard socket libraries cannot be used
  • - Hardware dependency: The NIC must have DPDK support

DPDK is therefore suited to dedicated forwarding appliances (firewalls, load balancers, NFV) and not to general-purpose applications.

2.6.3Key DPDK Mechanisms

Position: The secrets behind DPDK's performance — HugePages, lock-free rings, NUMA, and RSS.

HugePages

DPDK mandates the use of HugePages (2 MB or 1 GB). The reason was covered in Section 2.2.3: TLB coverage expands dramatically, eliminating TLB misses. A typical DPDK startup command specifies how many 2 MB huge pages to reserve on each NUMA node.

Lock-Free Ring Queue

rte_ring is one of DPDK's most elegant components. It is an MPMC ring queue based on two pairs of producer/consumer head/tail pointers:

// Simplified rte_ring core variables
struct rte_ring {
    uint32_t size;           // Ring capacity (power of 2 for efficient & masking)
    struct {
        uint32_t head;       // Next position to be written
        uint32_t tail;       // Last committed position
    } prod, cons;
    void *ring[size];
};

Enqueue flow: atomic CAS increments prod.head by n (reserving slots) → copy data → wait for prod.tail to catch up to the original head → advance prod.tail. This two-phase approach allows multiple producers to reserve concurrently and commit in order, guaranteeing that data in the ring is complete when seen by a consumer.

NUMA Awareness

Modern servers have multiple sockets; cross-socket memory access doubles latency (remote DRAM vs. local DRAM). DPDK's rte_socket_id() lets you allocate mempools and queues on a specific NUMA node. Production code must NUMA-align all resources — NIC, CPU core, and memory pool must reside on the same socket.

RSS and Multi-Queue

A single queue on a single core has a throughput ceiling. DPDK works with the NIC's RSS to distribute packets from different flows across multiple receive queues, each bound to a core, enabling true parallelism.

RSS hashes on the 5-tuple — packets belonging to the same flow always go to the same queue (preserving intra-flow ordering), while different flows are distributed evenly across cores. This design underlies all modern high-speed forwarding.

CPU Affinity

DPDK threads must be pinned to cores (the --lcores argument), preventing scheduler migrations that cause cache misses. Combined with the isolcpus kernel parameter to remove these cores from the Linux scheduler, they become fully dedicated.

Batch Processing

All DPDK APIs operate in batches — rte_eth_rx_burst receives 32 packets at a time, rte_eth_tx_burst transmits 32 at a time — amortizing per-packet overhead to near zero. Application logic should also batch: parse 32 Ethernet headers, look up 32 flow table entries, and so on, which is friendly to both the cache and the CPU pipeline.

Zero-Copy

The mbuf data region is directly the DMA region — the NIC DMAs packets into mbufs; the application reads them directly; on transmit, the NIC DMAs directly from the mbuf. The entire data path is zero-copy.

2.6.4VPP Vectorized Forwarding

Position: A high-performance, programmable forwarding framework built on top of DPDK, based on the idea of "applying the same operation to a batch of packets consecutively."

VPP's Core Innovation: Graph Nodes + Vectorization

VPP (Vector Packet Processing) is the FD.io project (based on Cisco open-source code). Its two core concepts:

Forwarding graph: The processing pipeline is a directed graph; nodes are "processing stages" (e.g., ethernet-input, ip4-lookup, ip4-rewrite). Packets enter from source nodes, pass through a series of nodes, and reach a sink (transmit or drop). Each node does exactly one thing and is pluggable/extensible.

Vectorization: Instead of processing one packet at a time, process a batch at a time. The traditional "each packet traverses the entire forwarding pipeline" is replaced by "each node processes the entire batch before passing it to the next node."

Why Vectorization Is So Fast

Suppose a batch of 256 packets with 10 processing stages. Traditional approach:

for packet in 256:
    for stage in 10:
        do_stage(packet)

Each stage's code is fetched 256 times; each fetch may cause an I-cache miss.

Vectorized approach:

for stage in 10:
    for packet in 256:
        do_stage(packet)

The stage code is fetched only once, staying warm in the I-cache. The 256 loop iterations allow data to remain in the D-cache (prefetcher-friendly). The improvement in I-cache hit rate is the primary reason VPP is fast — not SIMD vectorization.

This is the network-forwarding equivalent of the compiler optimization known as loop interchange — swapping loop order to improve cache locality.

VPP Node Example

// Simplified sketch of the ip4-lookup node
static uword
ip4_lookup_node_fn(vlib_main_t *vm, vlib_node_runtime_t *node,
                    vlib_frame_t *frame) {
    u32 *from = vlib_frame_vector_args(frame);
    u32 n_left = frame->n_vectors;   // Number of packets in this batch

    // Process 4 at a time (use prefetch to hide cache miss latency)
    while (n_left >= 4) {
        vlib_buffer_t *b0, *b1, *b2, *b3;
        b0 = vlib_get_buffer(vm, from[0]);
        b1 = vlib_get_buffer(vm, from[1]);
        b2 = vlib_get_buffer(vm, from[2]);
        b3 = vlib_get_buffer(vm, from[3]);

        // Prefetch the cache line for the next batch
        CLIB_PREFETCH(vlib_get_buffer(vm, from[4]), CLIB_CACHE_LINE_BYTES, LOAD);
        // ...

        // Parallel route lookup for 4 packets
        ip4_fib_mtrie_lookup(...);   // for b0
        ip4_fib_mtrie_lookup(...);   // for b1
        ip4_fib_mtrie_lookup(...);   // for b2
        ip4_fib_mtrie_lookup(...);   // for b3

        from += 4; n_left -= 4;
    }

    // Handle remaining 0–3 packets individually
    while (n_left > 0) { /* ... */ }

    return frame->n_vectors;
}

This "4-at-a-time + prefetch" pattern is the hallmark of VPP code. It uses instruction-level parallelism and explicit prefetching to hide memory latency behind computation.

What VPP Can Do

  • - IPv4/IPv6 route forwarding (FIB)
  • - MPLS
  • - Tunneling: VXLAN, GRE, etc.
  • - NAT
  • - Basic firewall ACL
  • - Load balancing
  • - IPSec

Performance: 10+ Mpps per core; dual-socket servers can achieve 100 Gbps+ line rate.

Relationship with DPDK

VPP uses DPDK as its default I/O driver layer but also supports AF_XDP, Netmap, and others. VPP handles the "packet processing framework"; DPDK handles "talking to hardware." The two are complementary and are commonly used together.

2.6.5TCP / UDP / Raw IP

Position: Transport-layer protocols — you use sockets every day, but you should know what's happening underneath.

The IP Layer

IP is an unreliable packet delivery service. Each IP packet is routed independently — there is no concept of a "connection"; intermediate routers forward each packet independently based on a lookup. The IPv4 header is 20 bytes (with optional fields); the IPv6 header is 40 bytes (but with simplified fields).

Key IP mechanisms: fragmentation (when the next-hop MTU is smaller than the packet), TTL (decremented by one at each hop; discarded at zero to prevent loops), and checksum (header only).

UDP

The thinnest transport layer — an 8-byte header (source port, destination port, length, checksum). Connectionless, no retransmission, no congestion control, no ordering guarantee.

Suitable for: DNS queries, real-time audio/video, game state synchronization, custom reliability layers (QUIC is built on UDP).

TCP: The Three Core Mechanisms

Connection: Established with a three-way handshake; closed with a four-way handshake. Maintains state such as sequence numbers, acknowledgment numbers, and window size.

Reliable delivery: Retransmission based on sequence numbers and ACKs. Timeout retransmit + fast retransmit (retransmit on 3 duplicate ACKs without waiting for timeout).

Congestion control: AIMD (Additive Increase Multiplicative Decrease). The congestion window grows exponentially during slow start, then linearly after reaching the threshold, and is halved on packet loss. Algorithms include Reno, Cubic (Linux default), and BBR (Google — models bandwidth/delay rather than relying on loss feedback).

TCP Performance Tuning

Parameters to pay attention to in production deployments:

  • - Buffer sizes: net.core.rmem_max/wmem_max. Long fat networks (high bandwidth, high latency) require large buffers.
  • - TCP_NODELAY: Disables the Nagle algorithm, sending small packets immediately — must be enabled for low-latency interactive workloads.
  • - TCP_QUICKACK: Disables delayed ACKs.
  • - Congestion algorithm: net.ipv4.tcp_congestion_control. BBR generally outperforms Cubic for data center or intercontinental links.
  • - TCP Fast Open: Carries data in the SYN packet, saving one RTT.

Raw IP / Raw Socket

The application constructs IP packets directly (any protocol can be customized). Requires root privileges. Used for: ping (ICMP), routing protocols (OSPF, BGP), custom protocols. socket(AF_INET, SOCK_RAW, IPPROTO_XXX).

In AI

AI training/inference communication primarily uses:

  • - TCP: gRPC/REST APIs, control plane for distributed training
  • - RDMA (covered in Chapter 4, Section 5.7): Training data plane, completely bypassing TCP/IP
  • - UDP: Occasionally used in custom AllReduce implementations

2.6.6TLS and QUIC

Position: Two paradigms for encrypted transport — TLS over TCP vs. QUIC (UDP + encryption).

TLS Handshake

TLS 1.3 (2018) simplified the handshake to 1 RTT (TLS 1.2 required 2 RTT):

  1. Client Hello: carries supported algorithms and ECDHE public key share
  2. Server Hello + Certificate + Finished: server returns selected algorithm, certificate, its own ECDHE share, and signature
  3. Client Finished: client verifies the certificate; both sides derive the session key

TLS 1.3 0-RTT mode: A client with an existing session can send request data in the very first packet (using a cached PSK). Zero latency overhead, but vulnerable to replay attacks.

QUIC's Design Motivation

TCP + TLS has several inherent limitations:

  1. TCP handshake (1 RTT) + TLS handshake (1 RTT) = at least 2 RTT before data can flow
  2. TCP is implemented in the OS; upgrades are slow (congestion algorithm deployment cycles are measured in years)
  3. TCP's head-of-line blocking: a single lost packet stalls the entire stream
  4. TCP connections are identified by a 4-tuple; when a phone switches between Wi-Fi and 4G, the IP changes and the connection breaks

QUIC (Quick UDP Internet Connections) reinvents the transport layer on top of UDP, solving all four problems:

  • - 1 RTT handshake (first connection), 0 RTT (reconnection) — encryption and reliable transport are merged
  • - User-space implementation: A library-level component; upgrading only requires a library update
  • - Multiple streams: Many independent streams within a single connection; packet loss in one stream does not block others
  • - Connection ID: Not tied to an IP address; connections survive network changes

HTTP/3

HTTP/3 = HTTP/2 semantics + QUIC transport. Compared to HTTP/2 over TCP+TLS: faster handshake, multiplexing without head-of-line blocking, and smooth network transitions. Standardized by the IETF in 2022; already supported by major browsers and CDNs.

Implementations

QUIC libraries: Google's QUICHE, Meta's mvfst, Cloudflare's quiche, Microsoft's msquic. All are user-space implementations.

In AI

LLM inference APIs (OpenAI, Anthropic, etc.) largely support HTTP/3, reducing time-to-first-token latency. Mobile AI applications (which switch networks frequently) benefit even more noticeably from QUIC.

2.6.7Forwarding Management and the Control Plane

Position: The data plane uses DPDK/VPP to forward packets at speed; the control plane is responsible for distributing policies and synchronizing routing tables. Brief section.

Control Plane vs. Data Plane

Modern network devices separate CP and DP:

  • Control plane: Routing protocols (BGP, OSPF), policy computation, configuration management. Runs on general-purpose CPUs; Python/Go implementations are fine. Being a bit slower is acceptable.
  • Data plane: Per-packet processing. DPDK/VPP/NP/ASIC. Must be fast.

The CP computes routes and pushes them to the DP's forwarding table (FIB). The DP only performs table-lookup forwarding; it does not participate in routing computation.

SDN

Software-Defined Networking takes this separation to its logical extreme: extract the control plane from every individual device and centralize it in a single controller. The controller pushes forwarding rules via protocols such as OpenFlow, gNMI, and P4Runtime.

Advantages: global-view optimization, unified policy, simplified devices (only the data plane needs to be implemented). Disadvantage: the controller becomes a single point of failure.

Control Plane Configuration Protocols

  • CLI: Traditional — SSH into a device and type commands
  • SNMP: Legacy monitoring; read-heavy, write-light
  • NETCONF + YANG: Structured configuration with transaction support
  • gNMI: gRPC-based, streaming telemetry
  • OpenConfig: Vendor-neutral YANG models

Routing Protocols in Brief

  • BGP: The Internet backbone protocol — path-vector based, inter-AS routing. TCP port 179; rich policy support.
  • OSPF: IGP (Interior Gateway Protocol) — link-state algorithm (Dijkstra shortest path). Fast convergence; used within an AS.
  • IS-IS: Similar to OSPF; commonly used by ISPs.

In AI

The network topology of AI training clusters (fat-tree, dragonfly) has an enormous impact on performance. The control plane itself, however, has little direct bearing on AI; this section is included only for general awareness.

2.6.8Logging and Observability

Position: The trinity of Metrics + Logs + Traces — without observability there is no production operations.

The Three Pillars of Observability

Metrics: Numerical values stored as time series. QPS, CPU, latency. Low-frequency sampling, small volume, easily aggregated. Prometheus is the standard.

Logs: Textual records of discrete events. High cardinality, large volume. ELK (Elasticsearch + Logstash + Kibana) and Loki are common choices.

Traces: Distributed tracing of a single request across multiple services. Each span records start time, duration, and tags. Jaeger, Zipkin, Tempo.

Structured Logging

Traditional printf-style logs are difficult for machines to process. The modern approach is structured logging — JSON or logfmt format:

{"time":"2024-05-01T10:30:15Z","level":"INFO","service":"inference",
 "request_id":"abc123","model":"gpt-4","ttft_ms":234,"tokens":156}

Structure enables efficient search and aggregation. Go's slog, Python's structlog, and Rust's tracing all push in this direction.

OpenTelemetry

OpenTelemetry (OTel) is a CNCF graduated project that aims to unify observability data collection standards — language SDKs, data models, and transport protocol (OTLP). A single SDK simultaneously produces metrics, logs, and traces that can be routed to different backends (Prometheus, Jaeger, commercial APM).

For any new service being written today, choosing OTel is almost always the right call.

Sampling

Full-volume traces are too expensive — storing 100K requests per second would overwhelm any disk. Sampling strategies:

  • Head-based sampling: Decided at the entry point — based on probability or rules
  • Tail-based sampling: Decided after the trace completes (can target errors and slow requests specifically)
  • Adaptive sampling: Dynamically adjusts the sampling rate

eBPF Observability

Traditional observability requires intrusive instrumentation of the application. eBPF lets you capture any function call, any syscall, and any socket event in the kernel without modifying the application. Pixie, Cilium Hubble, and Parca are all eBPF-driven observability tools.

In AI

Key observability metrics specific to LLM inference:

  • - TTFT (Time to First Token): Critical for the frontend experience
  • - TPOT (Time Per Output Token): Steady-state generation speed
  • - tokens/sec throughput: System-level throughput
  • - KV cache hit rate: Effectiveness of prefix caching
  • - Batch size distribution: Scheduler health
  • - GPU utilization: Hardware efficiency

Training observability: loss curves, gradient norms, learning rate, per-rank synchronization latency, data loading delay. TensorBoard, Weights & Biases, and MLflow are purpose-built platforms.
