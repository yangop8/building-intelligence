3.1 Traditional Machine Learning

3.1.1K-means and KNN

Overview: Two of the most fundamental distance-based algorithms — an introduction to clustering and classification.

KNN: Nearest Neighbor Classification

K-Nearest Neighbors requires no training — it simply memorizes all training samples. At prediction time, it finds the $k$ training samples closest to the query point and uses their labels to vote (classification) or average (regression). The only parameters are $k$ and the distance metric.

It may seem simple, but in low-dimensional, data-dense settings it performs well. Its theoretical properties are also elegant — as $n \to \infty$, KNN's error rate is at most twice the Bayes optimal error rate. This is one of the rare "distribution-free" guarantees in machine learning.

KNN's problems are all engineering ones:

  • - Prediction time: $O(nd)$ per query. For large datasets, use KD-Tree, Ball Tree, or approximate nearest neighbors (see Chapter 2, vector databases) to speed things up.
  • - Curse of dimensionality: In high-dimensional space, all points are roughly equidistant, and KNN breaks down.
  • - Feature scale sensitivity: A feature with range [0, 1] and a feature with range [0, 10000] computed directly using Euclidean distance will be completely dominated by the latter. Normalization is mandatory.

K-means: Nearest Centroid Clustering

Given data $\{\mathbf{x}_i\}$, partition them into $k$ clusters so that the sum of squared distances from each point to its cluster center is minimized:

$\min_{\{\mathbf{c}_j\}, \{S_j\}} \sum_{j=1}^k \sum_{\mathbf{x} \in S_j} \|\mathbf{x} - \mathbf{c}_j\|^2$

This is a combinatorial optimization problem (NP-hard); K-means finds an approximate solution via alternating optimization:

  1. Initialize $k$ cluster centers (random selection or K-means++)
  2. Assignment: assign each point to the nearest cluster center
  3. Update: recompute each cluster center as the mean of its assigned points
  4. Repeat steps 2–3 until centers no longer change
import numpy as np

def kmeans(X, k, max_iter=100, tol=1e-4):
    # K-means++ initialization: spread initial centers far apart
    n = len(X)
    centers = [X[np.random.randint(n)]]
    for _ in range(k - 1):
        # For each point, compute the squared distance to the nearest existing center
        d2 = np.min([np.sum((X - c)**2, axis=1) for c in centers], axis=0)
        # Sample the next center proportional to squared distance
        idx = np.random.choice(n, p=d2/d2.sum())
        centers.append(X[idx])
    centers = np.array(centers)

    for it in range(max_iter):
        # Assign
        dists = ((X[:, None] - centers[None, :])**2).sum(axis=2)
        labels = dists.argmin(axis=1)
        # Update
        new_centers = np.array([X[labels == j].mean(axis=0) for j in range(k)])
        if np.linalg.norm(new_centers - centers) < tol:
            break
        centers = new_centers
    return centers, labels

A few pitfalls with K-means. First, sensitivity to initialization — different seeds can produce completely different clusters. K-means++ initialization (as done in the code above) significantly improves this. Second, you must specify $k$ — typically the elbow method (plot $k$ vs. SSE, find the kink) or the silhouette score helps with selection. Third, it assumes spherical clusters — elongated or ring-shaped distributions are beyond its reach; DBSCAN or spectral clustering are needed for those cases.

In the Networking Domain

KNN was widely used in early traffic classification work — extract features from a connection (packet size distribution, inter-arrival times) and do KNN matching against a labeled traffic database. The downside is that as the traffic database grows, prediction latency becomes unacceptable. Tree models or small NNs are now the common replacements.

K-means is used for anomaly detection: normal traffic clusters into a few groups, and points that don't belong to any cluster (far from the nearest center) are potentially anomalous. But real anomalies are often too sparse and scattered for K-means to handle well; Isolation Forest is more commonly used instead.

3.1.2Tree Models: Decision Trees, Random Forests, XGBoost

Overview: The king of tabular data. Deep dive: from decision trees to GBDT to XGBoost, with full derivations.

Decision Trees

Each internal node of the tree is a feature test (e.g., age < 30), and leaf nodes produce predictions. The key to building a tree is choosing split points. For classification, the goal is to select splits that maximally improve the purity of child nodes. Common purity measures are the Gini impurity and entropy:

$\text{Gini} = 1 - \sum_c p_c^2, \quad H = -\sum_c p_c \log p_c$

Here $p_c$ is the fraction of class $c$ at a node. After a split, the weighted average impurity of child nodes is compared to the parent; the difference is called the information gain. The split with the highest information gain is chosen greedily.

Decision trees are naturally interpretable ("age > 30 → high income → ..."), require no feature normalization, handle missing values, and support both numerical and categorical features. But a single tree overfits easily — grow it deep enough and each leaf holds one sample, giving perfect training-set fit and catastrophic test-set performance. Pruning (pre-pruning: limit depth/leaf count; post-pruning: grow fully then prune) is necessary.

From Single Trees to Ensembles

The intuition behind ensemble learning: many weak learners voting beats a single strong learner. Two main lines:

Bagging (Bootstrap AGGregating): each tree is trained on a bootstrap sample (sampling with replacement) of the original data, and at each split only a random subset of features is considered. Predictions from multiple trees are averaged or voted. Random Forest (RF) is bagging applied to decision trees. The primary benefit is variance reduction — a single tree has high variance and low bias; averaging many trees dramatically reduces variance.

Boosting: trees are trained sequentially, with each tree learning the residuals that the previous tree failed to capture. GBDT (Gradient Boosting Decision Tree), XGBoost, and LightGBM all follow this approach.

The Mathematics of GBDT

Take regression as an example. The model is $F(x) = \sum_{t=1}^T f_t(x)$, where $f_t$ is the $t$-th tree. At each round we fit the negative gradient — the direction of steepest descent for the current loss.

For squared loss $L(y, F(x)) = \frac{1}{2}(y - F(x))^2$, the negative gradient is exactly the residual $r_i = y_i - F_{t-1}(x_i)$ — so GBDT under squared loss is just "fitting the residuals of the previous step." For other losses (log loss, Huber), the pseudo-residuals are the corresponding gradients.

The key insight of GBDT is that "training the $t$-th tree" is reduced to an ordinary regression problem (fit the pseudo-residuals), which is then added to the cumulative prediction.

XGBoost's Innovations

XGBoost is an engineered refinement of GBDT, proposed by Tianqi Chen in 2014. It introduces several important ideas.

Second-order Taylor expansion. Traditional GBDT uses only first-order gradients; XGBoost uses both first and second order. Expanding the loss $L$ around $F_{t-1}$ to second order:

$L(y, F_{t-1} + f_t) \approx L(y, F_{t-1}) + g_i f_t(x_i) + \frac{1}{2} h_i f_t(x_i)^2$

$g_i = \partial L / \partial F$, $h_i = \partial^2 L / \partial F^2$. This yields a closed-form optimal solution for the split gain.

Closed-form leaf weights. Let the $t$-th tree map $x$ to leaf $j$ with value $w_j$. Adding L2 regularization $\Omega(f) = \gamma T + \frac{1}{2}\lambda \sum_j w_j^2$ (where $T$ is the number of leaves):

$\text{Obj}^{(t)} = \sum_j \left[ (\sum_{i \in I_j} g_i) w_j + \frac{1}{2}(\sum_{i \in I_j} h_i + \lambda) w_j^2 \right] + \gamma T$

Setting the derivative with respect to $w_j$ to zero gives the optimal leaf weight:

$w_j^* = -\frac{G_j}{H_j + \lambda}, \quad G_j = \sum_{i \in I_j} g_i, \quad H_j = \sum_{i \in I_j} h_i $

Substituting back gives the minimum loss for a fixed tree structure:

$\text{Obj}^* = -\frac{1}{2} \sum_j \frac{G_j^2}{H_j + \lambda} + \gamma T$

This yields a closed-form split gain: splitting a node into left and right children $L, R$, the gain equals the original loss minus the post-split loss:

$\text{Gain} = \frac{1}{2}\left[\frac{G_L^2}{H_L + \lambda} + \frac{G_R^2}{H_R + \lambda} - \frac{(G_L+G_R)^2}{H_L+H_R+\lambda}\right] - \gamma$

All split candidates are ranked by this gain and the maximum is selected. $\gamma$ controls whether to split at all (no split if gain is below $\gamma$); $\lambda$ prevents overfitting.

Engineering optimizations: histogram approximation (LightGBM discretizes features into 256 bins, reducing $O(n \times d)$ to $O(n \times \text{bins})$); sparsity awareness (missing values automatically pick a default direction); column-block parallelism; GPU acceleration.

When to Choose Tree Models

XGBoost/LightGBM remain top-tier baselines on tabular data (rows are samples, columns are features). Over 90% of top Kaggle solutions on tabular competitions include GBDT. Advantages:

  • - No feature normalization required
  • - Automatically handles feature interactions (deep trees can express complex nonlinearities)
  • - Fast training (millions of samples in minutes)
  • - Fast inference (bounded tree depth, hundreds of microseconds per sample)
  • - Interpretable feature importance

When does deep learning take over? For unstructured data like images, text, and audio — deep learning learns features end-to-end and nearly always outperforms hand-crafted features combined with trees. But as long as your data is in row-column tabular form, XGBoost is almost always the first choice.

3.1.3SVM

Overview: Maximum-margin classification — once dominant, now a reference baseline.

Maximum-Margin Hyperplane

The objective of binary linear SVM: find a hyperplane $\mathbf{w}^\top \mathbf{x} + b = 0$ that separates two classes while maximizing the margin (distance to the nearest point). This is the maximum-margin classifier.

Geometric intuition: many hyperplanes can separate two classes, but the "wide" one is more tolerant of noise. Maximizing the margin is equivalent to minimizing $\|\mathbf{w}\|$. Adding classification constraints $y_i(\mathbf{w}^\top \mathbf{x}_i + b) \ge 1$ yields a convex quadratic program:

$\min_{\mathbf{w}, b} \frac{1}{2}\|\mathbf{w}\|^2 \quad \text{s.t. } y_i(\mathbf{w}^\top \mathbf{x}_i + b) \ge 1, \forall i$

Dual Form and Support Vectors

Applying Lagrangian duality, the dual problem is:

$\max_\alpha \sum_i \alpha_i - \frac{1}{2}\sum_{i,j} \alpha_i \alpha_j y_i y_j \mathbf{x}_i^\top \mathbf{x}_j, \quad \alpha_i \ge 0, \sum_i \alpha_i y_i = 0$

After solving for $\alpha^*$, $\mathbf{w} = \sum_i \alpha_i y_i \mathbf{x}_i$. A key observation: the vast majority of $\alpha_i = 0$ ; only the points on the margin boundary have $\alpha_i > 0$ — these are the support vectors, which is where the name SVM comes from. Prediction only requires the support vectors:

$f(\mathbf{x}) = \text{sign}\left(\sum_{i \in \text{SV}} \alpha_i y_i \mathbf{x}_i^\top \mathbf{x} + b\right)$

The Kernel Trick

What about linearly inseparable data? Map the data to a higher-dimensional space $\phi(\mathbf{x})$ where linear separability holds. Notice that the dual only involves $\mathbf{x}_i^\top \mathbf{x}_j$ — replacing this with $\phi(\mathbf{x}_i)^\top \phi(\mathbf{x}_j)$ suffices. And this inner product can be defined as a kernel function $K(\mathbf{x}_i, \mathbf{x}_j)$ computed directly, without explicitly computing $\phi$ — this is the kernel trick.

Common kernels:

  • - Linear kernel: $K = \mathbf{x}_i^\top \mathbf{x}_j$ (equivalent to no kernel)
  • - Polynomial kernel: $K = (\mathbf{x}_i^\top \mathbf{x}_j + c)^d$
  • - RBF kernel (Gaussian kernel): $K = \exp(-\gamma \|\mathbf{x}_i - \mathbf{x}_j\|^2)$ — corresponds to an infinite-dimensional space, most commonly used

The hyperparameter $\gamma$ of the RBF kernel controls the "curvature" of the decision boundary — large $\gamma$ makes the boundary cling tightly to training points (overfitting); small $\gamma$ produces a smoother boundary. Grid search over $\gamma$ and regularization parameter $C$ is the standard approach.

Soft Margin

Real data often has noise and some inseparable samples. A hard margin (perfectly separate everything) is too strict. The soft margin introduces slack variables $\xi_i \ge 0$:

$\min \frac{1}{2}\|\mathbf{w}\|^2 + C \sum_i \xi_i, \quad y_i(\mathbf{w}^\top \mathbf{x}_i + b) \ge 1 - \xi_i$

$C$ controls how much misclassification to tolerate — large $C$ is strict, small $C$ is lenient.

Current Status

In the 2000s, SVMs were the state of the art for text and image classification. Their prominence declined in the deep learning era, but they retain value in small-sample, high-dimensional, interpretable settings — bioinformatics and certain text classification tasks still use them. The downside is training complexity of $O(n^2)$ to $O(n^3)$, making them impractical on large datasets.

3.1.4N-gram and Label Propagation

Overview: Two old but useful tools — N-gram statistical language models and LPA for graph semi-supervised learning.

N-gram Language Models

N-gram is a probabilistic model that predicts the next word from the previous $N-1$ words, based on counts:

$P(w_n | w_1, ..., w_{n-1}) \approx P(w_n | w_{n-k+1}, ..., w_{n-1}) = \frac{\text{count}(w_{n-k+1}...w_n)}{\text{count}(w_{n-k+1}...w_{n-1})}$

Typical choices are $k = 2$ (bigram) or $k = 3$ (trigram). Before neural language models, this was the mainstream approach in NLP.

The core problem is sparsity — unseen combinations in the training set have zero probability. Smoothing techniques address this:

  • Add-one smoothing (Laplace): add 1 to all counts — the simplest approach
  • Good-Turing: uses the frequency of items appearing $r+1$ times to estimate the probability of items appearing $r$ times
  • Kneser-Ney: the most sophisticated, considers "how many distinct contexts a word appears as a continuation of"

Today LLMs dominate NLP and N-grams are no longer the lead approach. But character-level N-grams remain in use: unknown language detection, spell correction, and fast approximate matching (MinHash shingles).

LPA: Label Propagation Algorithm

Semi-supervised learning on graphs: a small number of nodes have labels, most do not. LPA assumes "tightly connected nodes should have the same label" and iteratively propagates labels along edges:

  1. Set initial labels for labeled nodes; unlabeled nodes start empty or with a uniform distribution
  2. Each node sets its label distribution to the (weighted) average of all its neighbors' label distributions
  3. Fix (clamp) the labels of labeled nodes
  4. Repeat until convergence

In matrix form: let $F \in \mathbb{R}^{n \times C}$ be the class distribution for each node, and $W$ the adjacency matrix (or a normalized version). Update $F \leftarrow D^{-1} W F$, where $D$ is the degree diagonal matrix. After clamping labeled nodes, repeat. This iteration has a closed-form solution (convergence can be proved).

LPA's strengths: computationally cheap, requires no training, and especially well-suited for large-graph community detection and label diffusion. Its weakness: heavily dependent on graph structure — poor graph construction leads to poor results.

In the Networking Domain

N-gram remains active in DGA domain detection — malware-generated random domain names can be distinguished from legitimate ones by computing character-level N-gram perplexity (malicious domains have higher perplexity).

LPA has direct applications in community detection over network topologies (finding tightly connected device subgroups) and malicious IP propagation detection (known malicious IPs influence suspicion scores of connected nodes via the graph).

3.1.5Regression and Prediction

Overview: From linear regression to generalized regression — the oldest method in machine learning, still important today.

Linear Regression

$y = \mathbf{w}^\top \mathbf{x} + b + \epsilon$; solve for $\mathbf{w}, b$ via least squares. The closed-form solution was covered in Chapter 1. Three commonly overlooked points:

Assumptions: the error $\epsilon$ is i.i.d. normal (zero mean). When this assumption is violated, the estimator remains unbiased but is no longer minimum-variance; in more serious cases, conclusions may be misleading. Always inspect residual plots in regression analyses.

Multicollinearity: when features are highly correlated, $\mathbf{X}^\top \mathbf{X}$ becomes near-singular and the estimate of $\mathbf{w}$ has high variance and is unstable. Ridge (L2) or PCA dimensionality reduction are the remedies.

Outliers: squared loss is extremely sensitive to outliers (a single outlier contributes loss proportional to the square of its distance). Huber loss (squared for small residuals, linear for large ones) is more robust.

Logistic Regression

Despite the name "regression," it performs classification. The model is:

$P(y=1|\mathbf{x}) = \sigma(\mathbf{w}^\top \mathbf{x} + b), \quad \sigma(z) = \frac{1}{1+e^{-z}}$

Training uses maximum likelihood (equivalent to minimizing cross-entropy). There is no closed-form solution; gradient descent or IRLS (Iteratively Reweighted Least Squares) is used.

The log-odds of logistic regression are linear: $\log\frac{P(y=1)}{P(y=0)} = \mathbf{w}^\top \mathbf{x} + b$. This gives the coefficients $w_i$ a direct interpretation — a unit increase in feature $x_i$ increases the log-odds by $w_i$, i.e., multiplies the odds by $e^{w_i}$.

Ridge and Lasso

The two canonical regularization methods.

Ridge (L2): minimize $\|\mathbf{X}\mathbf{w} - \mathbf{y}\|^2 + \lambda \|\mathbf{w}\|_2^2$. Analytical solution: $\mathbf{w} = (\mathbf{X}^\top \mathbf{X} + \lambda \mathbf{I})^{-1} \mathbf{X}^\top \mathbf{y}$. Larger $\lambda$ shrinks coefficients more.

Lasso (L1): minimize $\|\mathbf{X}\mathbf{w} - \mathbf{y}\|^2 + \lambda \|\mathbf{w}\|_1$. No analytical solution (L1 is non-differentiable); solved via coordinate descent or ISTA. Key property: Lasso drives irrelevant feature coefficients exactly to zero — automatic feature selection.

Why does L1 produce sparse solutions? Geometric intuition: the L2 level set is a sphere, and its tangency with the loss function's level set is almost never on a coordinate axis; the L1 level set is a diamond (with corners), whose corners lie on the coordinate axes, making it more likely that the tangency point corresponds to some coefficients being exactly zero.

ElasticNet = L1 + L2 combined, balancing Lasso's sparsity with Ridge's stability under multicollinearity.

Generalized Linear Models (GLM)

An extension of linear regression to settings where "the prediction target is not a real number." The core idea: use a "link function" $g$ to map the linear predictor $\eta = \mathbf{w}^\top \mathbf{x}$ to distribution parameters.

DistributionLink FunctionTask
NormalIdentity $\mu = \eta$Real-valued regression
BernoulliLogit $\log\frac{p}{1-p} = \eta$Binary classification (logistic)
PoissonLog $\log\mu = \eta$Count prediction
GammaLog $\log\mu = \eta$Positive reals (duration, cost)

Poisson regression is useful for predicting counts like network connections per day or attacks per hour — ordinary linear regression can produce negative predictions, while Poisson regression guarantees positive outputs.

In the Networking Domain

Linear/logistic regression in the networking domain primarily serves as a baseline — any complex model should be compared against logistic regression; failing to outperform LR signals a problem with features, data, or the task itself. Ridge/Lasso are used for interpretable traffic-prediction models (coefficients indicate which features matter). Poisson regression is used for arrival-rate modeling.

3.1.6Time Series: DTW and DBA

Overview: Classic algorithms for time series similarity and clustering — still occupying a unique niche in KPI monitoring and pattern mining.

Why Euclidean Distance Fails for Time Series

Two KPI curves may be similar in shape but shifted in time — Euclidean distance, which aligns them point-by-point, will conclude they are very different. DTW (Dynamic Time Warping) measures shape similarity through nonlinear temporal alignment, allowing one segment to correspond to multiple segments of the other.

The DTW Algorithm

Given sequences $X = (x_1, ..., x_n)$ and $Y = (y_1, ..., y_m)$, DTW finds an alignment path $\{(i_k, j_k)\}$ such that:

  1. It starts at $(1, 1)$ and ends at $(n, m)$
  2. Each step goes $(i, j) \to (i+1, j)$, $(i, j+1)$, or $(i+1, j+1)$
  3. The total point-to-point distance along the path is minimized

Solved via dynamic programming. Define $D(i, j)$ as the optimal alignment cost from $X_{1:i}$ to $Y_{1:j}$:

$D(i, j) = |x_i - y_j| + \min\{D(i-1, j), D(i, j-1), D(i-1, j-1)\}$

Filling the table takes $O(nm)$ time and space. The DTW distance is $D(n, m)$.

import numpy as np

def dtw(x, y):
    n, m = len(x), len(y)
    D = np.full((n+1, m+1), np.inf)
    D[0, 0] = 0
    for i in range(1, n+1):
        for j in range(1, m+1):
            cost = abs(x[i-1] - y[j-1])
            D[i, j] = cost + min(D[i-1, j], D[i, j-1], D[i-1, j-1])
    return D[n, m]

Optimizations: the Sakoe-Chiba band restricts $|i - j| \le w$, reducing complexity from $O(nm)$ to $O(nw)$; FastDTW uses multiscale recursion to approximate in $O(n)$.

DBA: Dynamic Time Warping Barycenter Averaging

With a DTW distance defined, it is natural to want to cluster time series — for example, replacing Euclidean distance with DTW inside K-means. But K-means requires computing cluster centers (averages) at each step, and the DTW average of time series is not the pointwise average (which distorts shapes). DBA is the "time-series average under DTW semantics":

  1. Initialize a candidate average sequence (e.g., randomly select one member)
  2. Compute DTW alignment between each member and the candidate
  3. Take the pointwise average of the aligned positions to form a new candidate
  4. Repeat until convergence

DBA + K-means is the classical time series clustering approach. It is widely used in KPI pattern discovery (what curve patterns recur frequently) and building libraries of anomalous patterns.

Modern Alternatives

Deep learning for time series now offers Transformer variants (Informer, Autoformer, PatchTST) and MLP-based models (N-BEATS), which generally outperform DTW on large datasets. But DTW's strengths are no training required, works with few samples, and is fully interpretable — the alignment path between two curves can be visualized intuitively. It remains worth using for initial analysis and small-sample scenarios.

3.2.1Matrix Multiply and Fundamental Operations

Overview: GEMM is everything — 70%+ of deep learning computation is matrix multiplication. Also covered: reductions, casts, and elementwise operations.

GEMM

General Matrix Multiply: $\mathbf{C} = \alpha \mathbf{A} \mathbf{B} + \beta \mathbf{C}$, $\mathbf{A} \in \mathbb{R}^{M \times K}$, $\mathbf{B} \in \mathbb{R}^{K \times N}$, $\mathbf{C} \in \mathbb{R}^{M \times N}$. This is the core API of all linear algebra libraries (BLAS level 3).

The naive triple-loop has complexity $O(MNK)$:

for (i = 0; i < M; i++)
    for (j = 0; j < N; j++)
        for (k = 0; k < K; k++)
            C[i][j] += A[i][k] * B[k][j];

The naive implementation performs extremely poorly — each access to B[k][j] jumps across rows causing cache misses. High-performance GEMM in practice (OpenBLAS, MKL, cuBLAS) uses multi-level blocking: the inner loops operate within L1/L2 cache, and small-tile results are accumulated into larger tiles.

In AI, matrix multiplication also has batched GEMM: perform multiple independent GEMMs simultaneously (e.g., one per sample in a batch) — specifically optimized to reduce kernel launch overhead.

Reduction Operations (Reduce)

Sum, maximum, or average across a given dimension. Typical examples:

  • sum(x, dim=-1): sum along the last dimension, common in the softmax denominator
  • mean(x, dim=(2,3)): spatial average pooling
  • max(x, dim=-1): subtract the maximum for numerical stability in softmax

GPU reductions have elegant implementations — tree-structured reduction (halving the number of active threads at each step), or warp-level primitives (__shfl_down_sync). A naive loop is $O(n)$ serial; an optimized version can achieve $O(\log n)$ parallelism.

Cast and Precision Conversion

Converting between FP32, FP16, BF16, INT8, and INT4. Seemingly simple, but full of subtleties:

  • FP32 → FP16: can underflow (small values become 0) or overflow (large values become inf). Loss scaling must be used during training.
  • BF16 ↔ FP32: BF16 is just FP32 with truncated mantissa — conversion is cheap and dynamic range is preserved.
  • FP ↔ INT: requires scale and zero-point; this is quantization (see section 3.6.2).

Elementwise Operations

Unary: abs, neg, exp, log, sqrt. Binary: +, -, *, /, pow, max.

The performance bottleneck for these is memory bandwidth, not compute. Performing x * 2 on a tensor of $N$ elements reads $N$ values and writes $N$ values, but only performs $N$ multiplications — arithmetic intensity is just 0.5 FLOP/byte. A GPU Tensor Core can deliver 1000 TFLOPS, but if the operation is fully memory-bound, the actual throughput is capped by memory bandwidth.

This is why operator fusion has the greatest benefit for elementwise operators — fusing relu(bn(conv(x))) into a single kernel means the data is read once and written once, eliminating the intermediate tensor round-trips through HBM. This is the core idea behind FlashAttention and PyTorch inductor optimizations.

Comparison and Selection Operations

greater, equal, where(cond, a, b), topk. where is conditional selection, commonly used in mask implementations (the causal mask in attention is just where(mask, score, -inf)). topk is critical in inference sampling (top-k sampling) and MoE routing (selecting the top-k experts).

Broadcasting

When performing elementwise operations on two tensors with different shapes, numpy/PyTorch broadcasting rules align each dimension from the right: dimensions must either be equal or one of them must be 1 (expandable).

Typical use case: adding bias to every row. If x has shape [B, N, D] and bias has shape [D], x + bias automatically broadcasts bias to every batch position.

Engineering note: broadcasting can produce unexpectedly large intermediate tensors. [N, 1] + [1, N] yields [N, N] — memory explosions often originate here.

3.2.2Data Movement Operators

Overview: Transpose, Gather, Scatter, Concat — no computation, just data shuffling, but the cost can be surprisingly high.

Transpose

Reordering dimensions. Mathematically an identity transformation; in practice it means rearranging memory. Transposing [B, N, H, D] to [B, H, N, D] changes the physical memory address of every element.

The key optimization for transpose on GPU is shared memory tiling — read a tile of the matrix into on-chip shared memory, then write it back to global memory in the new order, avoiding strided memory accesses.

But the best transpose is no transpose at all — in many cases a "virtual transpose" suffices (changing only the shape/stride metadata without touching the data). PyTorch's .transpose() returns a view; a real copy is made only when contiguous memory is subsequently required (e.g., when passing to a CUDA kernel).

Reshape / View

Changes the shape without changing the total number of elements. If the original data is contiguous in memory, reshape is nearly free (only metadata changes). When it is not contiguous, a contiguous() call must first copy the data.

Gather / Scatter

Gather: read values from a source tensor by index. y[i] = x[idx[i]]. Embedding lookup is a gather — given an input sequence of token IDs, fetch the corresponding rows from the embedding table.

Scatter: the reverse — write into a target tensor by index. y[idx[i]] = x[i].

Both present irregular memory access challenges on GPU: adjacent threads access non-adjacent memory locations, causing severe cache misses. For large embedding tables (vocabulary sizes of 100k+), gather is frequently a bottleneck during inference. Optimization approaches include bucket-sorting by index for batch access and using TLB-friendly layouts.

Concat / Split

Concatenate or split tensors. Concat requires allocating new memory and copying; Split can often be zero-copy (if the split dimension keeps each fragment contiguous).

Typical use: in MHA, concat(head_1, ..., head_H, dim=-1) merges multi-head outputs. But modern implementations often merge Q/K/V into a single large Linear and reshape to expose the head dimension, avoiding an explicit concat and saving one operator.

Pad

Padding the edges of a tensor with zeros (or other values). Before convolution, padding preserves output shape = input shape; variable-length sequences are padded to the same length for batching.

The cost of padding: waste. In a batch where the longest sequence is 2048 tokens and the shortest is 16, both are padded to 2048 — all computation on the short sequence is wasted. This is why continuous batching (see Chapter 4, section 5.13) is so valuable for LLM inference — it avoids padding waste.

The Hidden Cost of Data Movement

Rule of thumb: in a deep learning model, data movement accounting for 20–40% of total time is common. Unlike matrix multiplications, these costs are not conspicuous — they are scattered throughout the model but accumulate significantly. When profiling reveals that transpose, reshape, and cat operations account for a high fraction, it usually means the model architecture can be simplified.

3.2.3Convolution and Pooling

Overview: The lifeblood operators of CNNs. Deep dive: im2col unfolding and Winograd acceleration.

Definition of 2D Convolution

Input $\mathbf{X}$ with shape [C_in, H, W], convolution kernel $\mathbf{W}$ with shape [C_out, C_in, kH, kW], output $\mathbf{Y}$ with shape [C_out, H', W']:

$Y_{c, h, w} = \sum_{c'} \sum_{i, j} W_{c, c', i, j} \cdot X_{c', h+i, w+j}$

(This is technically cross-correlation; in engineering it is universally called convolution.) Each output position is the inner product between the kernel and the corresponding input region.

Parameters: stride (sliding step), padding (how much to add to edges), dilation (sampling interval within the kernel), and groups (grouped convolution, where each group only sees a subset of channels).

Naive Implementation and im2col

Direct implementation requires 7 nested loops: three output dimensions (C, H, W) × three kernel dimensions (C, H, W) × summing over input C. This performs poorly on GPU.

im2col is the classic technique: convert convolution into a matrix multiplication. The idea:

  1. Unfold each $kH \times kW \times C_{in}$ local patch from the input into a row; stack patches for all output positions into a large matrix $\mathbf{X}_{col}$ with shape [C_in×kH×kW, H'×W']
  2. Reshape the convolution kernel to [C_out, C_in×kH×kW]
  3. Perform the matrix multiply: $\mathbf{Y} = \mathbf{W} \cdot \mathbf{X}_{col}$, with shape [C_out, H'×W']
  4. Reshape the result to [C_out, H', W']

This allows calling a GEMM kernel directly, achieving near-peak throughput. The cost is a memory expansion of $\mathbf{X}_{col}$ by a factor of kH × kW — each input pixel is duplicated to multiple locations.

import numpy as np

def im2col(X, kh, kw, stride=1):
    # X: [C, H, W]
    C, H, W = X.shape
    oh, ow = (H - kh) // stride + 1, (W - kw) // stride + 1
    cols = np.zeros((C * kh * kw, oh * ow))
    for y in range(oh):
        for x in range(ow):
            patch = X[:, y*stride:y*stride+kh, x*stride:x*stride+kw]
            cols[:, y*ow + x] = patch.flatten()
    return cols, (oh, ow)

def conv2d_im2col(X, W):
    # X: [C, H, W], W: [C_out, C, kh, kw]
    C_out, _, kh, kw = W.shape
    cols, (oh, ow) = im2col(X, kh, kw)       # [C*kh*kw, oh*ow]
    W_mat = W.reshape(C_out, -1)              # [C_out, C*kh*kw]
    Y = W_mat @ cols                           # [C_out, oh*ow]
    return Y.reshape(C_out, oh, ow)

Winograd Algorithm

Offers a remarkable speedup for small kernels (3×3). The core idea: through a transformation, reduce the 9 multiplications needed by direct convolution to just 4 (per 2×2 output tile).

For the F(2, 3) transform: computing 2 outputs $y_1, y_2$ directly requires 6 multiplications. Winograd achieves this via:

$\mathbf{y} = \mathbf{A}^\top[(\mathbf{G}\mathbf{w}) \odot (\mathbf{B}^\top \mathbf{x})]$

where $\mathbf{G}, \mathbf{B}, \mathbf{A}$ are fixed transformation matrices ($\odot$ is elementwise multiply). When expanded, only 4 multiplications are needed (a 33% reduction). Extending to F(2×2, 3×3): a 2×2 output tile requires 36 multiplications directly, but only 16 with Winograd (a 56% reduction).

The cost: the transforms introduce extra additions and subtractions, and the benefit only applies to small kernels (with F(6×6, 3×3), the transform overhead begins to outweigh the multiplication savings). Libraries like cuDNN and ACL automatically choose between Winograd, im2col+GEMM, and FFT based on input and kernel size.

Depthwise Separable Convolution

Standard convolution: [C_out, C_in, kH, kW] has $C_{out} C_{in} kH kW$ parameters.

MobileNet introduced depthwise separable convolution: factorize into a depthwise step (independent kH×kW convolution per channel) plus a pointwise step (1×1 convolution for cross-channel mixing). Parameters: $C_{in} kH kW + C_{in} C_{out}$. For $kH = kW = 3$ and $C = 64$, this reduces parameters by roughly 9×.

The tradeoff is a slight reduction in representational capacity, but it is worthwhile in mobile/embedded settings — the foundation of MobileNet and EfficientNet.

Pooling

A downsampling operation. Max pooling: take the maximum over each 2×2 window, halving spatial dimensions. Average pooling: take the mean. Global pooling: compress an entire feature map into a single value (used for feature aggregation before classification).

The roles of pooling:

  1. Reduce computation (H, W halved)
  2. Expand the receptive field (the next convolutional layer sees a wider range)
  3. Approximate translation invariance

Modern architectures sometimes replace pooling with stride-2 convolutions, making the downsampling learnable.

3.2.4Activation Functions

Overview: The source of "nonlinearity" in neural networks — without them, no matter how deep the network, it is equivalent to a single linear layer.

Why Nonlinearity Is Needed

Composing linear layers still yields a linear transformation: $W_2(W_1 x + b_1) + b_2 = (W_2 W_1) x + (W_2 b_1 + b_2)$. The entire network collapses to a single linear transform — insufficient expressive power. Activation functions introduce nonlinearity, allowing the network to approximate any continuous function (the universal approximation theorem).

Sigmoid

$\sigma(x) = 1/(1 + e^{-x})$. Output is in (0, 1). The oldest activation, still used in logistic regression and LSTM gates.

Problem: vanishing gradients. $\sigma'(x) = \sigma(x)(1 - \sigma(x))$, with a maximum of 0.25. During backpropagation through a deep network, gradients are multiplied at each layer and quickly vanish. Sigmoid is not used in hidden layers of deep CNNs or Transformers.

Tanh

$\tanh(x) = (e^x - e^{-x})/(e^x + e^{-x})$. Output is in (-1, 1); a centered version of sigmoid with maximum gradient value of 1, slightly better. Historically common in RNNs.

ReLU

$\text{ReLU}(x) = \max(0, x)$. AlexNet in 2012 made ReLU mainstream. Advantages: extremely simple to compute; derivative is constant 1 on the positive half (solves vanishing gradients).

Disadvantage: Dead ReLU — once a neuron's output stays at 0, its gradient is also 0 and it never updates again. Particularly prone to occurring with large learning rates.

ReLU Variants

  • LeakyReLU: $\max(\alpha x, x)$, $\alpha = 0.01$. Small slope on the negative side prevents dead neurons.
  • PReLU: $\alpha$ becomes a learnable parameter.
  • ELU: uses $e^x - 1$ on the negative side, smooth.

GELU and SiLU/Swish

Two activations popular in the Transformer era:

GELU: $x \cdot \Phi(x)$, where $\Phi$ is the standard normal CDF. Equivalent to "dropout with probability proportional to input magnitude." Used in BERT, GPT, and related models. In practice, an approximation is used: $0.5 x (1 + \tanh(\sqrt{2/\pi}(x + 0.044715 x^3)))$.

SiLU / Swish: $x \cdot \sigma(x)$. Simpler. Used in the LLaMA family (specifically SwiGLU, discussed below).

GELU and SiLU look similar in shape — both are smooth versions of ReLU with a slight dip on the negative side. Empirically they perform slightly better than ReLU on large models (around 1%).

SwiGLU / GeGLU

Used in the FFN of LLaMA and Qwen. Not a pure activation function but a "gated linear unit" combined with activation:

$\text{SwiGLU}(x) = \text{SiLU}(W_1 x) \odot (W_2 x)$

Two parallel linear transformations: one passed through an activation and one not, then elementwise multiplied. Equivalent to learning a multiplicative gate at each position. The cost is that parameters change from [d, 4d] + [4d, d] to [d, 8d/3] + [d, 8d/3] + [8d/3, d] (to keep total parameter count unchanged, the intermediate dimension is typically reduced from 4d to 8d/3).

Softmax

Strictly not an "activation function," but plays a similar role — converting a vector into a probability distribution:

$\text{softmax}(\mathbf{x})_i = \frac{e^{x_i}}{\sum_j e^{x_j}}$

Properties: outputs sum to 1, differentiable everywhere, monotone. Used at classification output layers and for attention weights.

A numerically stable implementation must subtract the maximum value (see Chapter 1, section 2.8.1).

3.2.5Normalization

Overview: BatchNorm, LayerNorm, RMSNorm — pillars of training stability.

Why Normalization Is Needed

Training deep networks suffers from internal covariate shift: small changes to earlier layer parameters cause the input distribution of later layers to shift dramatically, destabilizing training. Normalization forces each layer's input back toward a standard distribution, making training more stable and allowing larger learning rates.

BatchNorm

For each feature (along the channel dimension) across the batch, subtract the batch mean and divide by the batch standard deviation, then rescale and shift with learnable parameters $\gamma, \beta$:

$\hat{x}_i = \frac{x_i - \mu_B}{\sqrt{\sigma_B^2 + \epsilon}}, \quad y_i = \gamma \hat{x}_i + \beta$

During training, $\mu_B$ and $\sigma_B^2$ are computed from the current batch; during inference, running mean/var accumulated during training (or from the final batch) is used.

BatchNorm was revolutionary for CNN training — introduced in 2015, it instantly became the standard, making it possible to train very deep networks like ResNet with 100+ layers.

Drawbacks:

  • - Sensitive to batch size: statistics are unstable with small batches (recommended ≥ 16)
  • - Different behavior during training vs. inference: a common source of subtle bugs
  • - Not suitable for RNNs/Transformers (variable sequence lengths make batch-dimension statistics meaningless)

LayerNorm

Normalizes over the feature dimension of each individual sample (not the batch dimension):

$\mu = \frac{1}{D}\sum_i x_i, \quad \sigma^2 = \frac{1}{D}\sum_i (x_i - \mu)^2$

Each sample is normalized independently with no dependence on batch. Transformers use LayerNorm throughout.

Behavior is identical during training and inference, and it is insensitive to batch size. Has become standard in NLP.

RMSNorm

A simplified version of LayerNorm: removes mean centering (deemed unimportant) and only normalizes magnitude:

$\text{RMS}(x) = \sqrt{\frac{1}{D}\sum_i x_i^2}, \quad y_i = \frac{x_i}{\text{RMS}(x)} \cdot \gamma_i$

Eliminates the mean computation and the $\beta$ bias parameter. LLaMA, Qwen, and DeepSeek all use RMSNorm — slightly less computation, no difference in accuracy: exactly the kind of trade-off worth making in LLMs.

GroupNorm / InstanceNorm

Variants of BatchNorm. GroupNorm partitions channels into groups and normalizes within each group (no batch dependence). InstanceNorm normalizes each sample and channel independently (commonly used in image style transfer). BatchNorm, GroupNorm, and InstanceNorm form a continuum along the axis of channel-dimension granularity.

Pre-Norm vs. Post-Norm

The placement of normalization in Transformers has been debated.

Post-Norm (original Transformer): LN(x + Sublayer(x)). Less stable; training large models requires warmup.

Pre-Norm (GPT series): x + Sublayer(LN(x)). The residual connection links input and output directly, improving gradient flow and making large model training more stable.

All modern large models use Pre-Norm.

PART THREE

3.3 Classic Deep Learning Models

The dominant architectures before Transformers — each still holds independent value in its own domain.

3.3.1CNN

Role: The cornerstone of computer vision. Understanding CNN means understanding the core concept of "inductive bias."

The Inductive Bias of CNNs

The secret behind CNN's success lies in two assumptions:

  1. Locality: Relevant features are typically spatially close (the meaning of a pixel is related to its neighbors)
  2. Translation equivariance: Features are recognized the same way regardless of where they appear in the image (a cat is a cat whether it's in the top-left or bottom-right)

Convolution neatly encodes both points — the kernel is local and shared across the entire image. Parameter efficiency is orders of magnitude higher than a fully connected MLP, and the training data requirement is much lower.

Evolution of Classic Architectures

LeNet (1998): The ancestor of CNNs, designed for handwriting recognition. A few convolutional layers + pooling + fully connected layers.

AlexNet (2012): The landmark that revived deep learning. 8 layers, using ReLU, Dropout, data augmentation, and GPU training. Reduced ImageNet top-5 error from 26% to 15%, shaking the entire CV community.

VGG (2014): Deepened the network by repeatedly stacking small 3×3 convolutional kernels. Proved that "depth matters." 16–19 layers.

Inception / GoogLeNet (2014): Parallel multi-scale convolutions (1×1, 3×3, 5×5). Introduced 1×1 convolutions for dimensionality reduction (fewer parameters).

ResNet (2015): Residual connections. This is the pivotal innovation in CNN history.

Residual Connections: The Secret to Training Deep Networks

After VGG, researchers found that continuing to add depth hurt rather than helped — not due to overfitting (training accuracy also dropped), but due to optimization difficulty. ResNet proposed: instead of learning $F(x)$, learn $F(x) - x$, and output $x + F(x)$.

flowchart LR X[x] --> F[Conv-BN-ReLU-Conv-BN] X --> ADD((+)) F --> ADD ADD --> RELU[ReLU] RELU --> Y[y]

Why does it work? Two perspectives:

Gradient perspective: Gradients along the residual path are passed directly to the input, bypassing the nonlinearities and multiplications of intermediate layers — alleviating vanishing gradients.

Identity mapping perspective: If the optimal behavior for a layer is the identity mapping (no transformation), the residual version only needs $F = 0$, which is far easier than learning a full identity $F(x) = x$.

ResNet enabled training 152-layer and even 1000-layer networks, completely unlocking depth.

More Recent Architectures

DenseNet (2016): Each layer is densely connected to all previous layers.

EfficientNet (2019): Finds the optimal scaling ratio across depth, width, and resolution.

ConvNeXt (2022): Redesigns CNNs using Transformer design philosophy (large kernels, GELU, LayerNorm), demonstrating that CNNs and Transformers each have their own strengths.

Where CNNs Stand Today

In CV, Transformers (ViT) already dominate large models, but CNNs still lead for small models and edge deployment — their inductive bias wins on small data, and their regular computation graphs are more amenable to hardware optimization. In embedded systems, industrial control, and medical imaging, CNNs remain the first choice.

3.3.2RNN / LSTM / GRU

Role: The dominant sequence modeling architectures before Transformers. Deep dive into LSTM gating mechanisms.

Vanilla RNN

At each time step, the current input $\mathbf{x}_t$ and the previous hidden state $\mathbf{h}_{t-1}$ are combined into a new hidden state:

$\mathbf{h}_t = \tanh(W_{xh} \mathbf{x}_t + W_{hh} \mathbf{h}_{t-1} + \mathbf{b})$

Unrolled over a time sequence:

flowchart LR X1[x_1] --> H1[h_1] X2[x_2] --> H2[h_2] X3[x_3] --> H3[h_3] H1 --> H2 H2 --> H3 H1 --> Y1[y_1] H2 --> Y2[y_2] H3 --> Y3[y_3]

The same weight $W_{hh}$ is reused at every time step. This allows RNNs to handle sequences of arbitrary length, with parameter count that does not grow with sequence length.

The problem: vanishing/exploding gradients. During backpropagation, gradients pass through $T$ matrix multiplications by $W_{hh}$ (where $T$ is the sequence length). If the eigenvalues of $W_{hh}$ are less than 1, gradients decay exponentially; greater than 1, they grow exponentially. This causes vanilla RNNs to remember only very short histories.

LSTM Gating

Long Short-Term Memory (1997) solves this problem through explicit memory cells and gating mechanisms. The core innovation is introducing a cell state $\mathbf{c}_t$, which provides a "highway" for gradients to travel long distances.

Three gates govern how information flows:

$\mathbf{f}_t = \sigma(W_f [\mathbf{h}_{t-1}, \mathbf{x}_t] + \mathbf{b}_f) \quad (\text{forget gate})$
$\mathbf{i}_t = \sigma(W_i [\mathbf{h}_{t-1}, \mathbf{x}_t] + \mathbf{b}_i) \quad (\text{input gate})$
$\mathbf{o}_t = \sigma(W_o [\mathbf{h}_{t-1}, \mathbf{x}_t] + \mathbf{b}_o) \quad (\text{output gate})$
$\tilde{\mathbf{c}}_t = \tanh(W_c [\mathbf{h}_{t-1}, \mathbf{x}_t] + \mathbf{b}_c) \quad (\text{candidate cell})$
$\mathbf{c}_t = \mathbf{f}_t \odot \mathbf{c}_{t-1} + \mathbf{i}_t \odot \tilde{\mathbf{c}}_t$
$\mathbf{h}_t = \mathbf{o}_t \odot \tanh(\mathbf{c}_t)$

Intuition:

  • Forget gate decides how much information to retain from the old cell state (near 1: retain; near 0: forget)
  • Input gate decides how much new information to accept
  • Output gate decides how much to expose from the cell state to the outside

The key gradient channel is c_t = f_t * c_{t-1} + ... — when the forget gate stays close to 1 for a long time, c_t retains long history through addition, and gradients during backpropagation decay almost not at all. This is the meaning of "long short-term memory."

GRU

Gated Recurrent Unit (2014) simplifies LSTM: it merges the cell state and hidden state, and combines the forget and input gates into an "update gate." 25% fewer parameters, with empirical performance comparable to LSTM.

$\mathbf{z}_t = \sigma(W_z [\mathbf{h}_{t-1}, \mathbf{x}_t]) \quad (\text{update gate})$
$\mathbf{r}_t = \sigma(W_r [\mathbf{h}_{t-1}, \mathbf{x}_t]) \quad (\text{reset gate})$
$\tilde{\mathbf{h}}_t = \tanh(W [\mathbf{r}_t \odot \mathbf{h}_{t-1}, \mathbf{x}_t])$
$\mathbf{h}_t = (1 - \mathbf{z}_t) \odot \mathbf{h}_{t-1} + \mathbf{z}_t \odot \tilde{\mathbf{h}}_t$

Bi-RNN and Deep RNN

Bidirectional RNN: Runs one pass forward and one pass backward; the representation at each position is a concatenation from both directions. Well-suited for understanding tasks (each word's representation takes context from both sides into account).

Multi-layer RNN: The output of one layer serves as the input to the next, stacked several layers deep. Greater expressive power.

Why RNNs Gave Way to Transformers

The fundamental limitation of RNNs is that they cannot be parallelized — time steps must be computed sequentially. GPU throughput cannot be fully utilized. In 2017, Transformers replaced the recurrence with self-attention entirely, enabling full parallelism during training with even better results. Today LSTM/GRU in industry mainly survives in: small-model scenarios (embedded time-series forecasting), classic baselines, and research.

The Revival of Mamba

In 2023, Mamba (selective SSM) brought "RNN-like architectures" back into the spotlight — training can use parallel scanning, while inference is $O(1)$ per step. It challenges Transformers in long-context scenarios. The spirit of RNNs is not dead; it just needs cleverer design (see Chapter 4, section 5.6).

3.3.3GNN and GAT

Role: Graph Neural Networks — the mainstream paradigm for processing non-Euclidean data.

Why Graph Data Needs Specialized Models

Characteristics of graphs: variable number of nodes, variable number of neighbors, no fixed ordering. The "locality + translation equivariance" assumption of CNNs does not hold (there is no regular grid on a graph). RNNs don't work either (graphs have no natural sequential order). GNNs are designed specifically for graph data.

The Message-Passing Paradigm

Almost all mainstream GNNs operate within the "message passing" framework. Each layer performs two steps:

  1. Aggregation: Each node collects messages from its neighbors
  2. Update: Updates its representation based on the aggregated result and its current representation
$\mathbf{h}_v^{(l+1)} = \text{UPDATE}\left(\mathbf{h}_v^{(l)}, \text{AGG}\left(\{\mathbf{h}_u^{(l)}: u \in \mathcal{N}(v)\}\right)\right)$

$\mathcal{N}(v)$ is the set of neighbors of $v$. Different GNNs differ in how AGG and UPDATE are implemented.

GCN

Graph Convolutional Network (2017). The aggregation is a normalized weighted average:

$\mathbf{h}_v^{(l+1)} = \sigma\left(\sum_{u \in \mathcal{N}(v) \cup \{v\}} \frac{1}{\sqrt{\deg(u) \deg(v)}} W^{(l)} \mathbf{h}_u^{(l)}\right)$

The normalization factor $1/\sqrt{d_u d_v}$ prevents high-degree nodes from dominating the aggregation. The derivation comes from a spectral-domain approximation of the graph Laplacian — the details are complex, but the idea is simple: take a weighted average of neighbors, then apply a linear transformation.

GraphSAGE

GCN requires computing the Laplacian over the entire graph, which does not scale to large graphs. GraphSAGE instead uses neighbor sampling: each node randomly samples $k$ neighbors to aggregate from. This enables mini-batch training and supports very large graphs (billions of nodes).

GAT: Graph Attention

In GCN, neighbor weights are determined by graph structure (degree normalization), and cannot express "this neighbor is more important." GAT introduces attention:

$\alpha_{vu} = \frac{\exp(\text{LeakyReLU}(\mathbf{a}^\top [W \mathbf{h}_v \| W \mathbf{h}_u]))}{\sum_{k \in \mathcal{N}(v)} \exp(\text{LeakyReLU}(\mathbf{a}^\top [W \mathbf{h}_v \| W \mathbf{h}_k]))}$
$\mathbf{h}_v^{(l+1)} = \sigma\left(\sum_{u \in \mathcal{N}(v)} \alpha_{vu} W \mathbf{h}_u\right)$

Each node computes attention weights over its neighbors and aggregates with those weights. $\|$ denotes concatenation and $\mathbf{a}$ is a learnable vector. Multi-head attention is supported (just like Transformers).

GAT's advantages lie in heterogeneous graphs (where neighbor importance varies) and interpretability (weight visualization).

Applications

Node classification (predicting a category for each node), link prediction (predicting whether an edge should exist between two nodes), graph classification (properties of an entire graph), graph generation. Used in: social network recommendations, molecular property prediction, knowledge graph completion, network topology analysis.

Over-Smoothing

GNNs cannot be stacked as deep as CNNs — after a few layers, all node representations converge (because each layer averages over neighbors). In practice, 2–3 layers is common; going deeper requires specialized designs (residual connections, PairNorm, etc.).

3.3.4GAN and VAE

Role: The two major generative model paradigms — adversarial vs. variational. Now superseded by diffusion models, but their ideas live on.

What Generative Models Do

Given data samples $\{x_i\} \sim p_{\text{data}}$, learn a model $p_\theta$ such that samples drawn from $p_\theta$ "look like" real data. Two major challenges:

  1. The distribution of high-dimensional data (images, audio) is extremely complex
  2. Directly maximizing likelihood $\max \prod p_\theta(x_i)$ requires a normalizing constant that is intractable in high dimensions

GAN and VAE each sidestep these problems from completely different directions.

VAE: Variational Inference

Assume the data is generated by a latent variable $z$: $z \sim p(z) = \mathcal{N}(0, I)$, $x \sim p_\theta(x|z)$. We want to maximize the likelihood $\log p_\theta(x) = \log \int p_\theta(x|z) p(z) dz$, but the integral is intractable.

VAE approximates it using a variational lower bound (ELBO):

$\log p_\theta(x) \ge \mathbb{E}_{q_\phi(z|x)}[\log p_\theta(x|z)] - D_{\text{KL}}(q_\phi(z|x) \| p(z))$

Here $q_\phi(z|x)$ is the encoder (maps $x$ to a distribution over $z$), and $p_\theta(x|z)$ is the decoder (reconstructs $x$ from $z$). Both are neural networks.

Training jointly optimizes two terms: the reconstruction term (the decoder can recover $x$ from $z$) and the KL term ($q$ stays close to the prior $p(z)$). Training uses the reparameterization trick : $z = \mu_\phi(x) + \sigma_\phi(x) \odot \epsilon, \epsilon \sim \mathcal{N}(0, I)$ — making sampling differentiable.

VAE advantages: explicit latent space (supports interpolation and controllable generation), stable training. Disadvantage: generated samples tend to be blurry (reconstruction loss is pixel-level MSE, which averages across outcomes).

GAN: Adversarial Game

An entirely different approach. Two networks compete:

  • Generator $G$ : takes noise $z$ as input, outputs fake sample $G(z)$
  • Discriminator $D$: takes a sample as input, judges whether it is real or fake (outputs a 0–1 probability)

Objective function:

$\min_G \max_D \mathbb{E}_{x \sim p_{\text{data}}}[\log D(x)] + \mathbb{E}_{z \sim p_z}[\log(1 - D(G(z)))]$

$D$ wants to classify real samples as real and fake samples as fake; $G$ wants to fool $D$. The two compete until $G$ generates samples that $D$ cannot distinguish from real ones — at that point $G$ has learned $p_{\text{data}}$.

GAN advantages: sharp samples with high visual quality. Disadvantages: extremely unstable training (achieving game equilibrium is hard), mode collapse (the generator produces only a few types of samples), no explicit likelihood.

Notable improvements: WGAN replaces JS divergence with Wasserstein distance, resulting in much more stable training; StyleGAN (NVIDIA) achieves fine-grained style control and was the primary technology behind early deepfakes.

The Rise of Diffusion Models

From 2020 onward, diffusion models rapidly replaced GAN/VAE as the mainstream approach to image generation.

Core idea: progressively add noise to data until it becomes pure noise (forward process, fixed), then train a neural network to denoise (reverse process, learned). During generation, start from pure noise and iteratively denoise over $T$ steps to obtain a sample.

Advantages: extremely stable training (a pure denoising objective, no adversarial component), high sample quality, controllable (guidance, conditioning). DALL-E 2, Stable Diffusion, Imagen, and Sora all use diffusion.

Trade-off: slow sampling (requires 20–50 steps, each a full forward pass). Acceleration directions: DDIM, DPM-Solver, Consistency Models (1-step generation).

PART FOUR

3.4 Data & Feature Engineering

"Garbage in, garbage out" — data quality sets the ceiling on model performance.

3.4.1Sample Scoring, Generation, and Augmentation

Role: What to do when data is scarce — synthetic samples, augmentation, and active learning.

Data Augmentation

Multiple "new" samples are generated from a single real sample through transformations, expanding the training set. Domain-specific approaches:

DomainCommon Augmentations
ImageFlip, rotation, crop, color jitter, Cutout, Mixup, CutMix, AutoAugment
TextSynonym replacement, back-translation, random insertion/deletion, EDA, Back-translation
AudioTime stretching, pitch shifting, background noise injection, SpecAugment (spectral masking)
Time SeriesTime warping, noise injection, window slicing, DBA synthesis

Mixup is a particularly interesting augmentation: both samples and labels are interpolated as convex combinations, $x' = \lambda x_1 + (1-\lambda) x_2, y' = \lambda y_1 + (1-\lambda) y_2$. The model learns that "a linearly interpolated sample should have a linearly interpolated label" — the regularization effect is significant and broadly benefits classification tasks.

Sample Scoring

Not all samples are equally important. Scoring methods:

  • Difficulty: How large is the current model's loss on this sample (loss-based)
  • Uncertainty: The model's prediction confidence (entropy, variance)
  • Representativeness: How different this sample is from other samples
  • Influence: The effect on test-set performance when this sample is removed from training (influence function)

Curriculum learning: Train on easy samples first, then gradually introduce harder ones — similar to how humans learn. For certain tasks this can significantly accelerate convergence.

Active learning: When labeling budget is limited, use the model to select the most worthwhile samples to label (typically those with the highest uncertainty) for human annotation. Reduces labeling cost.

Synthetic Data

Two main approaches:

Rule-based generation: Generate samples using domain rules (e.g., synthesize adversarial network traffic from known malicious attack rules, or synthesize autonomous driving scenarios using a physics engine). Controllable, but may be too "clean" to match the real distribution.

Model-based generation: Use generative models (GAN, diffusion, LLM) to produce synthetic samples. Particularly popular in the LLM era:

  • Self-Instruct: Have a strong model generate instructions + responses, which serve as training data for a weaker model
  • Distillation: A small model learns from a large model; the large model's outputs serve as "synthetic labels"
  • Rejection Sampling: The large model generates multiple answers; a scoring function selects the good ones for the training set

Risk of synthetic data: model collapse — if successive generations of models are trained on data synthesized by the previous generation, the distribution drifts over generations and ultimately loses diversity. Mixing in real data is essential for safety.

Handling Class Imbalance

In extreme cases (1% positive, 99% negative), a model that simply predicts all negatives achieves 99% accuracy — but is completely useless. Remedies:

  • Resampling: Oversample the minority class, undersample the majority class
  • SMOTE: Generate new samples by interpolating between minority-class examples
  • Class weights: Assign higher loss weight to the minority class
  • Focal Loss: $-\alpha(1-p)^\gamma \log(p)$ — down-weights easy samples (high $p$) so the model focuses on hard ones

3.4.2Preprocessing and Feature Construction

Role: Cleaning, normalization, encoding — in traditional ML, feature engineering often matters more than model selection.

Data Cleaning

Real-world data is messy: missing values, outliers, type inconsistencies, duplicates. Cleaning is unglamorous work but must be done:

  • Missing values: Drop rows, fill with mean/median, predict with a model, or add a "is missing" flag feature
  • Outliers: Detect using IQR ($Q_1 - 1.5 \cdot \text{IQR}, Q_3 + 1.5 \cdot \text{IQR}$) or Z-score, then drop, clip, or flag
  • Duplicates: Exact or approximate deduplication

Numerical Feature Processing

Normalization: Scale to [0, 1] — $x' = (x - \min)/(\max - \min)$. Sensitive to outliers.

Standardization: Subtract mean and divide by standard deviation — $x' = (x - \mu)/\sigma$. More robust; preferred for normally distributed data.

Log transformation: For heavy-tailed or skewed data, apply $\log(1+x)$ before further processing. Network traffic packet sizes, file sizes, and costs are typical heavy-tailed distributions.

Binning: Discretize a continuous variable into intervals. Can be equal-width, equal-frequency, or decision-tree-based. Benefits include handling nonlinear relationships, robustness to outliers, and strong interpretability.

Categorical Feature Encoding

One-Hot: One binary column per category. Simple, but causes dimensionality explosion when there are many categories.

Label Encoding: One integer per category. Suitable for tree models (trees can handle categorical features), but not for linear models (which would incorrectly assume a numeric ordering).

Target Encoding: Map each category to the mean of the target variable for samples in that category. Information-rich but carries leakage risk — must be computed on the training set before being applied to validation/test sets.

Embedding: Map categories to low-dimensional continuous vectors (learned through a network). The standard approach for handling categorical features in deep learning.

Temporal Features

Using raw timestamps directly tends to perform poorly. Common expansions: hour of day, day of week, day of month, month, quarter, is_weekend, holiday flag, days since a specific event, etc.

Cyclic encoding: Hour is not a linear quantity (11 PM and midnight are close but numerically far apart). Use $\sin(2\pi h/24), \cos(2\pi h/24)$ to encode and preserve periodicity.

Text Features

Traditional: TF-IDF, N-grams, bag of words. Modern: use pretrained models (BERT, Sentence-BERT) to extract embeddings directly as features.

Feature Crosses

Multiply or combine two features. Tree models handle this automatically (deep trees naturally express feature interactions), but linear models require explicit feature engineering. Factorization Machines (FM) and FFM in recommender systems are automated feature crossing methods.

3.4.3Dimensionality Reduction Algorithms

Role: The curse of dimensionality, and the remedies — PCA, t-SNE, UMAP.

PCA

The math was covered in Section 2.3.4 of Chapter 1. Engineering notes:

  • - Must standardize first — PCA is variance-sensitive; without standardization, features with larger ranges dominate
  • - Choosing how many components: look at cumulative explained variance ratio (typically retain 95% of variance)
  • - PCA is linear dimensionality reduction — it cannot handle nonlinear structure (e.g., a Swiss roll)

Variants: Kernel PCA uses the kernel trick for nonlinear structure; Incremental PCA supports streaming data; Sparse PCA produces sparse principal components for better interpretability.

t-SNE

t-distributed Stochastic Neighbor Embedding (2008). Dimensionality reduction designed specifically for visualization to 2D/3D.

Idea: compute pairwise "similarity" in high-dimensional space (Gaussian kernel), compute similarities in low-dimensional space using a t-distribution (heavy-tailed), and minimize the KL divergence between the two. High-dimensional neighbors remain neighbors in low dimensions.

t-SNE produces stunning results — the 10 digit classes of MNIST separate cleanly in 2D. But there are pitfalls:

  • - The perplexity hyperparameter has a large effect (typical range 5–50); different values yield different results
  • - Global structure is not preserved (distances between clusters are meaningless)
  • - Results differ across runs (random initialization)
  • - Only for visualization; cannot be used as features for downstream models

UMAP

Uniform Manifold Approximation and Projection (2018). Faster than t-SNE with comparable or better quality, and better global structure preservation. Based on Riemannian geometry and algebraic topology (sounds intimidating, but in practice it's just a library call).

The modern first choice for visualization. Also supports transform (projecting new data into an already-trained space), which t-SNE strictly does not.

Autoencoder Dimensionality Reduction

The "encoder" part of a neural network: input $x$ → low-dimensional latent vector $z$ → decode back to $\hat{x}$. Minimize reconstruction loss $\|x - \hat{x}\|^2$. $z$ is the reduced representation.

Stronger than PCA for nonlinear structure. VAE is the probabilistic version of an autoencoder. But training is time-consuming; for small to medium scale, PCA is often sufficient.

When to Use Which

  • PCA: As features for downstream models, large datasets, speed required, interpretability needed
  • t-SNE / UMAP: Data exploration, visualizing clusters
  • Autoencoder: Nonlinear structure, end-to-end scenarios

3.4.4Feature Selection

Role: Picking the truly useful features from many — reduce overfitting, improve interpretability, lower inference cost.

Three Categories of Methods

Filter methods: Evaluate each feature independently, without involving the model. Fast.

  • - Variance threshold: Drop features with very low variance (nearly constant)
  • - Correlation coefficient: Drop features with low correlation to the target (linear relationship)
  • - Mutual information: Captures "correlation" for nonlinear relationships
  • - Chi-squared: Categorical features vs. categorical targets

Wrapper methods: Use a model to evaluate the performance of different feature subsets, searching greedily.

  • - Forward selection: Start from empty; each round add the feature that gives the largest performance gain
  • - Backward elimination: Start from all features; each round remove the feature with the smallest impact on performance
  • - RFE (Recursive Feature Elimination): Train a model, inspect coefficients/importance, remove the least important feature, retrain, and repeat

Embedded methods: Models with built-in feature selection capability.

  • - Lasso regression: L1 regularization drives irrelevant feature coefficients to zero
  • - Tree model feature importance

Pitfalls of Feature Importance

Tree model feature importance has multiple definitions:

  • Gain: Total information gain from all splits using this feature
  • Split count: Number of times this feature is used to split
  • Permutation importance: How much model performance drops after shuffling this feature

Different definitions give different rankings. More importantly: feature importance ≠ causality. Highly correlated features may share importance (they are predictive jointly); removing one causes its importance to be redistributed to others. A feature may also be purely a leakage signal (e.g., "hospital ID" predicting disease — not because it has a causal relationship, but because different hospitals see different disease distributions).

SHAP

SHapley Additive exPlanations (2017). Uses Shapley values from game theory to give a fair attribution of each feature's contribution to a single prediction. More fine-grained than global feature importance — you can see "for this sample, how much did each feature contribute."

import shap, xgboost

model = xgboost.XGBClassifier().fit(X_train, y_train)
explainer = shap.TreeExplainer(model)
shap_values = explainer.shap_values(X_test)

# Single-sample explanation
shap.force_plot(explainer.expected_value, shap_values[0], X_test.iloc[0])

# Global feature importance (by SHAP values)
shap.summary_plot(shap_values, X_test)

Explainable AI in industry is largely built on SHAP. It is standard in scenarios with legal explainability requirements, such as financial risk control and medical diagnosis.

PART FIVE

3.5 Training Optimization Algorithms

From supervised to self-supervised to alignment — the core techniques driving the evolution of model capabilities.

3.5.1Supervised Learning

Overview: The most classical paradigm — given labeled data, train a model to fit it. The details hold a wealth of knowledge.

The Basic Loop

Supervised learning is the foundational paradigm of deep learning. Given data $(x_i, y_i)$, a model $f_\theta$, a loss $\ell$, and an optimizer update:

$\theta_{t+1} = \theta_t - \eta \nabla_\theta \sum_i \ell(f_\theta(x_i), y_i)$

Choosing a Loss Function

TaskLoss
RegressionMSE, MAE, Huber
Binary classificationBinary cross-entropy
Multi-class classificationCategorical cross-entropy
Multi-label classificationBinary CE per label
Sequence generationToken-level cross-entropy
Object detectionClassification loss + bbox regression + IoU
SegmentationPixel-wise CE + Dice loss

Overfitting and Regularization

The primary enemy of supervised learning is overfitting — perfect on the training set, poor on the test set. Countermeasures stack up in layers:

  • L2 regularization (weight decay): $\ell + \lambda \|\theta\|^2$. Keeps parameters small and decision boundaries smooth.
  • Dropout: Randomly drops a fraction of neurons (e.g., 50%) during training, forcing the model not to rely on any single neuron. At inference time all neurons are kept but scaled by the dropout probability.
  • Data augmentation: Increases data diversity.
  • Early stopping: Stop training when validation performance stops improving.
  • Label smoothing: Replace hard labels [0, 1, 0] with [0.05, 0.9, 0.05] to prevent the model from becoming overconfident.

Train / Validation / Test Split

The classic 70/15/15 or 80/10/10 split. Key points:

  • - Look at the test set only once at the very end — repeatedly tuning on the test set constitutes data leakage
  • - Stratified sampling: preserve class proportions
  • - Time-series splitting: split by time, not randomly (using future data for training is cheating)
  • - Group-based splitting: data from the same user or device must not appear in both train and test sets (information leakage)

Cross-Validation

When data is scarce, use K-fold cross-validation: divide the data into K folds, train on K-1 folds and validate on the remaining one, rotate K times and average the results. This eliminates the randomness of a single split.

For time series, use time series split: progressively expand the training window.

Training Diagnostics

The loss curve reveals a great deal:

  • - Training loss not decreasing: learning rate too small, vanishing gradients, insufficient model capacity, or data issues
  • - Training loss oscillating or exploding: learning rate too large or numerical instability
  • - Training loss decreasing but validation loss not: overfitting
  • - Both decreasing but with a large gap: overfitting, but manageable
  • - Neither decreasing: fundamental problem with the model or data

3.5.2Self-Supervised Pre-Training

Overview: The foundational paradigm for LLMs and large foundation models — "creating" labels from unlabeled data.

Core Idea

Labeled data is expensive — but unlabeled data (web pages, code, video) is virtually unlimited. Self-supervised learning designs an "auxiliary task" whose labels can be automatically generated from the data itself. The representations learned on this auxiliary task are often highly useful for downstream tasks.

Pre-Training Objectives for NLP

MLM (Masked Language Modeling, BERT): Randomly mask 15% of tokens and have the model predict the masked ones. The label is the original word. Encoder-only architecture.

CLM (Causal Language Modeling, GPT): Given the first $t$ tokens, predict the $(t+1)$-th. The label is the next word. Decoder-only architecture.

Span Corruption (T5): Mask a contiguous span of tokens and have the model generate the masked span.

The success of the GPT family demonstrates that CLM scales best — more data and larger models yield stronger capabilities, with zero-shot/few-shot transfer to a wide range of tasks.

Pre-Training for CV

Contrastive learning (SimCLR, MoCo): Apply two different augmentations (random crop, color jitter, etc.) to the same image, and train the representations of these two views to be similar while being distant from representations of other images.

MAE (Masked Autoencoders): Inspired by MLM, mask 75% of image patches and have the model reconstruct them. Simple yet remarkably effective.

DINO (self-distillation): A student network mimics the output of a teacher network without requiring explicit negative samples.

Why It Works

A central finding of self-supervised pre-training: representation learning can emerge from "prediction" tasks. In order to predict masked words or reconstruct images, the model must learn grammar, semantics, and object concepts — this knowledge is a byproduct of "solving the task," but it also happens to be exactly what downstream tasks need.

Scaling Laws

OpenAI and DeepMind observed that model capability improves as a power law of data volume × model size × compute, without saturating for a very long time. This is the scaling law:

$L(N, D) \approx \left(\frac{N_c}{N}\right)^{\alpha_N} + \left(\frac{D_c}{D}\right)^{\alpha_D}$

Loss as a function of parameter count $N$ and data volume $D$ follows power-law decay. This turns large model training into an "engineering problem" (just add compute to improve) rather than a "research problem" (requiring algorithmic breakthroughs).

Chinchilla (2022) refined this: for a given compute budget, the optimal allocation is to scale data and parameter counts in roughly equal proportion (rather than simply stacking more parameters as was previously done).

3.5.3Fine-Tuning and Alignment

Overview: From pre-trained model to usable product — the complete pipeline of SFT, DPO, and RLHF.

This section goes deep; these are the most critical training techniques of the LLM era.

Three-Stage Training Pipeline

Modern LLMs go through three steps from pre-training to release:

flowchart LR PT[Pretrain
Massive text CLM] --> SFT[SFT
Instruction-response pairs] SFT --> RM[Reward Model
Preference data] RM --> RLHF[RLHF / DPO
Align to human preferences] RLHF --> REL[Released Model]

Pre-training falls under Section 3.5.2 and produces the base model — it holds vast world knowledge and can complete text, but does not know how to "converse." The subsequent three steps transform it into a conversational assistant.

SFT: Supervised Fine-Tuning

Train on human-annotated "instruction → ideal response" pairs. The loss is CLM — the model learns to "generate the response following the instruction" from this data.

# SFT data format (simplified)
{
    "instruction": "Explain what gradient descent is",
    "response": "Gradient descent is an optimization algorithm that computes the gradient of the loss function..."
}

# Construct training sequence (with system prompt and role markers)
prompt = f"<|system|>You are a helpful assistant<|user|>{instruction}<|assistant|>"
full   = prompt + response + "<|end|>"

# Loss is computed only on the response portion (prompt portion is loss-masked)
labels = [-100] * len(tokenize(prompt)) + tokenize(response + "<|end|>")

The amount of SFT data is crucial but does not need to be enormous — tens of thousands of high-quality instruction samples are enough to turn a base model into a usable conversational model. OpenAI's InstructGPT paper revealed the approximate scale.

The limitation of SFT: it only teaches the model "this kind of answer is correct," not "this kind of answer is wrong." What the model learns is imitation, not preference ranking. This is where RLHF or DPO comes in.

RLHF: The Classic Approach

Reinforcement Learning from Human Feedback. Three steps:

Step 1: Collect preference data. For the same prompt, have the SFT model generate multiple responses and have humans annotate which is better — producing $(x, y_{\text{win}}, y_{\text{lose}})$ triples.

Step 2: Train a Reward Model. A neural network $r_\phi$ (typically initialized from the SFT model) learns to "score good responses high and bad responses low." The loss uses the Bradley-Terry preference model:

$\mathcal{L}_{\text{RM}} = -\log \sigma(r_\phi(x, y_{\text{win}}) - r_\phi(x, y_{\text{lose}}))$

Step 3: Optimize the policy with PPO. Use the reward model's scores as reinforcement signals. The policy $\pi_\theta$ generates responses, receives rewards, and updates $\theta$ to increase the probability of high-reward responses.

To prevent the policy from drifting too far from the original SFT model (maintaining fluency and avoiding reward-hacking tricks), a KL penalty is added:

$\mathcal{L}_{\text{PPO}} = -\mathbb{E}[r_\phi(x, y) - \beta \cdot D_{\text{KL}}(\pi_\theta \| \pi_{\text{SFT}})]$

In practice, the clipping variant of the PPO algorithm is used.

Engineering Complexity of RLHF

A single RLHF training run requires four models to be loaded simultaneously:

  • - Policy (the model being trained)
  • - Reference (frozen SFT model, for computing KL)
  • - Reward Model (frozen, for scoring)
  • - Critic (value function estimate for PPO)

GPU memory is tight, training is unstable, hyperparameter tuning is difficult, and reward hacking risk is high. This motivated the development of DPO.

DPO: Bypassing RL

Direct Preference Optimization (2023). The key insight: the combination of PPO + KL penalty + RM is mathematically equivalent to a closed-form classification loss. Specifically, under the Bradley-Terry preference model, the optimal policy and the reference policy have an explicit relationship:

$\pi^*(y|x) \propto \pi_{\text{ref}}(y|x) \exp\left(\frac{1}{\beta} r(x, y)\right)$

Solving for $r$:

$r(x, y) = \beta \log \frac{\pi^*(y|x)}{\pi_{\text{ref}}(y|x)}$

Substituting back into the RM preference loss:

$\mathcal{L}_{\text{DPO}} = -\log \sigma\left(\beta \log \frac{\pi_\theta(y_w|x)}{\pi_{\text{ref}}(y_w|x)} - \beta \log \frac{\pi_\theta(y_l|x)}{\pi_{\text{ref}}(y_l|x)}\right)$

No reward model, no RL, no critic — only the policy and reference models are needed, with a standard supervised loss. Training is stable, simple, and achieves results comparable to or better than PPO.

This has been the most important simplification in alignment training over the past two years. Today a large number of open-source models (Zephyr, Tulu, Nous-Hermes) use DPO rather than PPO.

Newer Variants

  • IPO: Fixes a problem with DPO under noisy preference data
  • KTO: Replaces pairwise comparisons with single-point binary feedback (good/not good), making data easier to collect
  • ORPO: Combines SFT and preference alignment into a single training pass
  • RLAIF: Uses AI instead of humans to label preference data, enabling scale

The Goal of Alignment

At the technical level: make model outputs conform to "the distribution of human preferences." But preferences themselves are pluralistic — the three goals of helpful, harmless, and honest sometimes conflict (being too helpful might assist bad actors; being too harmless might refuse legitimate requests). In practice, this is balanced through diverse RLHF data.

Constitutional AI (Anthropic) and rule-based preferences represent directions for improving on "pure human annotation" — using explicit rules to generate preference comparisons, reducing annotation inconsistency.

Parameter-Efficient Fine-Tuning (PEFT)

Full fine-tuning requires updating all model parameters, which is extremely costly in the LLM era. Parameter-Efficient Fine-Tuning trains only a tiny fraction of parameters while achieving performance close to full fine-tuning:

  • - LoRA (Low-Rank Adaptation): Freeze the original weights $W_0$ and inject a low-rank bypass matrix $BA$ ($r \ll d$). The number of trainable parameters can be reduced 100–1000×. At inference time, $BA$ can be merged back into $W_0$ with zero additional latency.
  • - QLoRA: Builds on LoRA by quantizing the frozen weights to NF4 (4-bit), combined with double quantization and a paged optimizer, enabling fine-tuning of a 65B model on a single consumer-grade GPU.
  • - Adapter: Insert small bottleneck modules into each Transformer layer and train only the Adapter parameters.
  • - Prefix Tuning / P-Tuning: Prepend trainable virtual tokens to the input.

Detailed mathematical derivations and code implementations are available in Chapter 5, Paper Deep-Dive P11 (LoRA / QLoRA).

3.5.4Reinforcement Learning

Overview: From games to alignment — core RL concepts and several key algorithms.

The MDP Framework

The mathematical backbone of RL is the Markov Decision Process (MDP): state $s$, action $a$, transition $P(s'|s,a)$, reward $R(s,a)$, discount factor $\gamma$. The objective is to learn a policy $\pi(a|s)$ that maximizes the expected cumulative return:

$J(\pi) = \mathbb{E}_\pi\left[\sum_{t=0}^\infty \gamma^t R(s_t, a_t)\right]$

Two Main Lines

Value-based: Learn the state value function $V(s)$ or the state-action value $Q(s, a)$. Representatives: Q-learning, DQN.

Policy-based: Directly learn the policy $\pi_\theta$. Representatives: REINFORCE, PPO.

Actor-Critic: A combination of both — the policy (actor) generates actions while the value function (critic) evaluates their quality. All modern mainstream methods (A3C, PPO, SAC) use the AC architecture.

Q-learning and DQN

The Bellman optimality equation for Q-learning:

$Q^*(s, a) = R(s, a) + \gamma \mathbb{E}_{s'}[\max_{a'} Q^*(s', a')]$

$Q^*$ can be approximated via iterative updates. When the state space is large, the Q-table becomes intractable, so DQN (2015) uses a neural network to approximate $Q$. Key techniques:

  • Experience replay: Store $(s, a, r, s')$ tuples in a buffer and sample randomly during training — breaks temporal correlations
  • Target network: Use lagged weights to compute the target $y = r + \gamma \max_{a'} Q_{\text{target}}(s', a')$, stabilizing training

DQN was the first to achieve superhuman performance on Atari games, marking RL's emergence from the ivory tower.

Policy Gradient and PPO

The policy gradient theorem:

$\nabla_\theta J(\pi_\theta) = \mathbb{E}_\pi[\nabla_\theta \log \pi_\theta(a|s) \cdot A(s, a)]$

$A(s, a) = Q(s, a) - V(s)$ is the advantage function (how much better this action is compared to average).

Naive policy gradient has enormous variance and unstable updates. TRPO (2015) and PPO (2017) stabilize training by limiting the magnitude of each update. PPO uses clipping instead of TRPO's hard constraint — simple, efficient, and has become the default RL algorithm:

$\mathcal{L}^{\text{CLIP}} = \mathbb{E}\left[\min(r_t A_t, \text{clip}(r_t, 1-\epsilon, 1+\epsilon) A_t)\right]$

$r_t = \pi_\theta / \pi_{\theta_{\text{old}}}$ is the probability ratio of new to old policy. This prevents $r$ from deviating too far from 1 (typical $\epsilon = 0.2$).

Applications

RL has achieved success in games (AlphaGo, AlphaStar, OpenAI Five), robotics, autonomous driving, and LLM alignment (RLHF). However, RL training is notoriously difficult — sample efficiency is low, reward engineering is tricky, and hyperparameter tuning is an art.

In the networking domain, RL is used for congestion control (variants of PCC and BBR), routing optimization, resource scheduling, and ad bidding, but industrial deployment remains relatively rare — supervised learning and rule-based systems are the default in industry, with RL as a supplement.

3.5.5Distillation, Transfer, and Multi-Task Learning

Overview: Using one model to accelerate another, using one task to help another — three parameter-efficient and data-efficient techniques.

Knowledge Distillation

A large model (teacher) "teaches" its knowledge to a small model (student). Proposed by Hinton in 2015: the small model learns not only from hard labels but also from the teacher's soft labels (the full probability distribution).

$\mathcal{L} = \alpha \cdot \text{CE}(y_{\text{true}}, p_s) + (1-\alpha) \cdot T^2 \cdot \text{KL}(p_t^T \| p_s^T)$

$T$ is the temperature, used to "soften" the distribution ($T > 1$ flattens it). At high temperature, the teacher's "runner-up class" information is preserved — for example, "this image looks most like a cat but also somewhat like a dog."

DistilBERT, TinyBERT, and TinyLLaMA are all distillation products. Compression ratios are typically 2–5×, with only a few percentage points of performance loss.

Transfer Learning

Transfer a model trained on a source domain to a target domain. Typical workflow:

  1. Load a pre-trained backbone
  2. Replace the final layer with an output head for the target task
  3. Freeze the backbone and train only the head (linear probing); or train all parameters together but with a smaller learning rate (fine-tuning)

The key decision is what to freeze and what to train. With little data: freeze most layers. With more data: train everything, but use a smaller learning rate for the early layers (which capture general features).

Domain Adaptation goes a step further — the target domain may be unlabeled, and feature distribution alignment methods are used for transfer.

Multi-Task Learning

A single model learns multiple tasks simultaneously. Shares a backbone with one head per task.

Advantages:

  • - Shared parameters, less memory
  • - Implicit regularization; one task may help another generalize
  • - Low-resource tasks benefit from high-resource tasks

Challenge: task conflict — gradients from different tasks may point in opposite directions, pulling against each other. Mitigations:

  • Loss weighting: Assign higher weight to more important tasks
  • Gradient projection (PCGrad): Project conflicting gradients to remove the conflicting component
  • MoE: Different tasks activate different experts

T5 and FLAN are canonical examples of multi-task training — hundreds of NLP tasks trained together, producing general-purpose models.

Continual Learning

A model has already learned task A and now needs to learn task B. Naive fine-tuning causes catastrophic forgetting — learning B overwrites knowledge of A. Mitigations include: EWC (add larger regularization on parameters important for A), Replay (mix in a portion of task A data), and Adapter (add small modules for each task, freezing the main trunk).

3.5.6Federated Learning

Overview: Distributed training where data never leaves the local device — the solution for privacy-compliant scenarios.

Motivation

Hospitals have patient data but cannot share it, phones have user inputs but cannot upload them, banks have transaction data subject to compliance restrictions. Centralized training is not an option — federated learning keeps data on-device and uploads only model updates.

FedAvg

The most basic federated algorithm (Google, 2017):

  1. The server distributes the current global model $\theta_t$ to participating clients
  2. Each client trains locally for a few rounds, obtaining a local update $\theta_t^k$
  3. Clients upload $\theta_t^k - \theta_t$ (or directly $\theta_t^k$) to the server
  4. The server computes a weighted average: $\theta_{t+1} = \sum_k \frac{n_k}{n} \theta_t^k$

Challenges

Non-IID data: Different clients have different data distributions (one client's photos are all cats, another's are all dogs), causing FedAvg to converge slowly or not at all. Improvements: FedProx adds a proximal term; SCAFFOLD uses control variates to correct bias.

Communication cost: Uploading and downloading large models is expensive. Compression (gradient quantization, Top-K sparsification) is the main optimization direction.

Heterogeneity: Wide variation in client device performance. Asynchronous FedAsync and hierarchical FL address this.

Privacy: Even uploading only updates can leak information (model inversion attacks can recover inputs from gradients). Differential privacy (adding noise) and secure aggregation (aggregating over encrypted values so the server sees only the sum, not individual contributions) are the mainstream defenses.

Applications

Google's Gboard (mobile keyboard) uses federated learning to train language models on-device. In the medical domain, multi-institution collaborative training (FeTS challenge) trains tumor segmentation models across multiple hospitals. In networking: collaborative training of multi-site attack detection models.

Industrial deployment remains limited — engineering complexity is high and performance lags behind centralized training. This may become more important as privacy regulations tighten (GDPR, CCPA).

3.5.7AutoML

Overview: Automating the machine learning pipeline — hyperparameter search, architecture search, and feature engineering.

Hyperparameter Search

Traditional methods:

  • Grid search: Enumerate all combinations. Explodes in high dimensions.
  • Random search: Sample each hyperparameter independently at random. Bergstra (2012) demonstrated this outperforms grid search in most scenarios — most hyperparameters are actually insensitive and don't need a fine grid.
  • Bayesian optimization: Use a Gaussian process or TPE to model the "hyperparameter → validation loss" function, and select the next point to try using an acquisition function (EI, UCB). Optuna, Hyperopt, and Ax are the mainstream libraries.
  • Hyperband / BOHB: Early-stop poor configurations and allocate more budget to promising ones.

NAS: Neural Architecture Search

Automatically find the best network structure. Google's NASNet (2017) used RL to search and achieved state-of-the-art on ImageNet, but at enormous compute cost (thousands of GPU-days).

Subsequent improvements:

  • DARTS: Relax discrete architecture choices into continuous weights and optimize with gradient descent — runs in a few GPU-days
  • EfficientNet: A representative architecture family found by NAS
  • One-shot NAS: Train a single supernet; all candidate architectures are subgraphs of it

NAS was prominent in CV's early days. In the era of large models, Transformers dominate and NAS is less relevant for mainstream architecture selection. It remains useful for small models and hardware-targeted scenarios.

Automated Feature Engineering

Libraries like Featuretools and TSFresh automatically generate large numbers of features from tabular or time-series data (aggregations, cross-features, time windows), followed by feature selection to keep the useful ones. This can significantly reduce manual effort in traditional ML pipelines.

End-to-End AutoML Systems

H2O AutoML, AutoGluon, Auto-sklearn, Google Vertex AI AutoML — upload your data and the system automatically handles preprocessing, feature engineering, model selection, hyperparameter tuning, and ensembling. Suitable for teams without ML experts who need to get started quickly.

That said, experienced ML engineers typically outperform AutoML — domain knowledge provides priors that AutoML cannot know (e.g., knowing that certain feature combinations are meaningless). AutoML is an accelerator, not a replacement.

3.6 AI Performance Engineering

Taking models from demo to production — comprehensive optimization of inference latency, throughput, and power consumption.

3.6.1NPU Performance Modeling

Focus: Estimating theoretical peak performance and identifying real bottlenecks — the Roofline model.

The Roofline Model

Given a hardware platform (compute $P$ FLOPS, bandwidth $B$ bytes/s) and a kernel (arithmetic intensity $I$ = FLOPs / bytes accessed), the performance upper bound is:

$\text{Perf} \le \min(P, I \cdot B)$

Plotted on a log-log graph: the x-axis shows arithmetic intensity, the y-axis shows performance. $I \cdot B$ forms the sloped "bandwidth roof" and $P$ forms the horizontal "compute roof." The ridge point $I^* = P / B$ is the hardware balance point.

  • - $I \lt I^*$: memory-bound. Spending effort to increase compute is pointless; the goal is to reduce memory accesses.
  • - $I > I^*$: compute-bound. The hardware's compute is saturated; the only options are upgrading hardware or reducing FLOPs through algorithmic improvements.

Computing Arithmetic Intensity for Typical Operators

GEMM: $M \times K \times N$ multiplication reads $MK + KN + MN$ elements. Arithmetic intensity is approximately $O(\min(M, N, K))$. High for large matrices — compute-bound.

Elementwise: 1 FLOP per element, reading and writing 2 elements. Arithmetic intensity of 0.5 FLOP/byte. Severely memory-bound.

Attention (standard implementation): Arithmetic intensity varies with sequence length $n$. Short sequences are memory-bound; long sequences gradually become compute-bound. FlashAttention completely rewrites attention into a compute-bound form.

The Balance Point on H100

NVIDIA H100 SXM5: FP16 Tensor Core compute is approximately 989 TFLOPS (dense) / 1979 TFLOPS (2:4 sparse), with HBM3 bandwidth of 3.35 TB/s. Using dense compute, the balance point is $I^* = 989000 / 3350 \approx 295$ FLOP/byte; using 2:4 structured sparsity gives $I^* \approx 591$ FLOP/byte.

This means 295 floating-point operations must be performed for every byte read from HBM to fully saturate compute. Most kernels fall short — which is exactly why tiling, operator fusion, and on-chip SRAM utilization are central to optimization.

The Unique Characteristics of NPUs

NPUs typically feature:

  • - Tensor core compute >> general-purpose compute (most compute capacity is inaccessible without GEMM workloads)
  • - On-chip SRAM significantly larger than GPU shared memory (on the order of several MB)
  • - Slightly lower HBM bandwidth but better energy efficiency

This means the Roofline ridge point on NPUs may sit at a higher arithmetic intensity — making efficient use of on-chip SRAM even more critical.

3.6.2Model Architecture Optimization, Quantization, Pruning, and Lookup Tables

Focus: Reducing computation at the algorithm level — the three pillars of model compression.

Architecture Optimization

Designing for efficiency from the start:

  • Depthwise separable convolutions (MobileNet): 1/9 the parameters and computation
  • Grouped convolutions (ShuffleNet, ResNeXt): reduced computation
  • GQA (Grouped Query Attention): multiple Q heads share a single set of K and V. The mainstream approach since LLaMA2, significantly reducing KV Cache size
  • MLA (Multi-head Latent Attention, DeepSeek): compresses KV into a latent space, saving even more
  • MoE: sparse activation — large total parameter count but only a small fraction is activated per forward pass

Quantization

This topic has come up several times before; here it is treated systematically.

PTQ (Post-Training Quantization): quantize after training, without retraining. Calibrate on a few hundred samples to estimate activation ranges, then compute scale/zero-point.

  • - Weight-only INT8: the most conservative option, nearly lossless
  • - Weight-only INT4: GPTQ / AWQ / AutoGPTQ, the mainstream approach for LLM inference
  • - W8A8 (both weights and activations in INT8): accelerates inference but with accuracy loss

QAT (Quantization-Aware Training): simulates quantization during training (using fake-quantization nodes), allowing the model to learn to function under quantized conditions. Higher accuracy than PTQ, at the cost of requiring retraining.

The key idea behind AWQ: observations show that "important weights" correspond to "channels with large activation magnitudes." Direct quantization causes the quantization error on these important weights to dominate the output. AWQ first scales up the weights of important channels (reducing their relative quantization error), then inversely scales the activations — mathematically equivalent, but with quantization error redistributed.

The key idea behind GPTQ: treats quantization as column-wise optimization, using Hessian information to absorb the error from already-quantized columns into the remaining columns. Algorithmically complex but achieves the best results at INT4.

Pruning

Unstructured pruning: sets individual weights with small absolute values to zero. High compression ratio, but sparse matrix acceleration requires special hardware support.

Structured pruning: removes entire channels, entire heads, or entire layers. Directly accelerates inference without special hardware support.

2:4 sparsity: forces exactly 2 out of every 4 consecutive weights to be zero. Natively supported by NVIDIA Ampere and later Tensor Core hardware, delivering a direct 2× speedup. Slightly inferior to unstructured pruning in terms of accuracy, but an engineering win overall.

Pruning typically requires fine-tuning to recover accuracy — accuracy always drops after pruning, but fine-tuning for a few epochs recovers most of it. Iterative pruning (alternating rounds of pruning and fine-tuning) yields the best results.

Lookup Tables Instead of Computation

In some cases, results can be precomputed offline and stored in a lookup table, replacing runtime computation with a table lookup for significant speedups. Typical examples:

  • Softmax LUT: precomputed $\exp(x)$ table for discrete value domains
  • Embedding Cache: prefetched embeddings for high-frequency tokens
  • Activation LUT: nonlinear activations approximated with piecewise linear functions plus table lookups
  • PQ in retrieval: precomputed codebook distance tables for product-quantized vectors replacing real-time distance computation

Well-suited for embedded scenarios — table lookups are faster than computation and consume less power.

Distillation

Covered in section 3.5.5 above. A small model learns from a large one, achieving the highest compression ratios (10× or more), but requiring training from scratch. Typically combined with quantization and pruning — distill to a small model, then quantize to INT4, for a final speedup of 20–50×.

3.6.3Operator Optimization

Focus: Squeezing hardware performance at the operator level — tiling, fusion, and FlashAttention in practice.

This section is a technically dense deep dive.

Tiling: Moving the Problem into On-Chip SRAM

The standard optimization for GEMM. In a naive implementation, $\mathbf{A}, \mathbf{B}, \mathbf{C}$ all reside in HBM and every element is read repeatedly. Tiling divides the matrices into small blocks, loading each block entirely into SRAM to be computed there.

// Three-level tiling: block (HBM→SRAM) / warp / thread
#define BM 128
#define BN 128
#define BK 16

for (int bi = 0; bi < M; bi += BM) {
    for (int bj = 0; bj < N; bj += BN) {
        // Result tile: BM × BN, held in SRAM
        float C_tile[BM][BN] = {0};

        for (int bk = 0; bk < K; bk += BK) {
            // Load A[bi:bi+BM, bk:bk+BK] and B[bk:bk+BK, bj:bj+BN] from HBM to SRAM
            load_to_sram(A_tile, A + bi*K + bk, BM, BK);
            load_to_sram(B_tile, B + bk*N + bj, BK, BN);

            // Perform small matrix multiply within SRAM
            for (int i = 0; i < BM; i++)
                for (int j = 0; j < BN; j++)
                    for (int k = 0; k < BK; k++)
                        C_tile[i][j] += A_tile[i][k] * B_tile[k][j];
        }
        // Write C_tile back to HBM
        store_from_sram(C + bi*N + bj, C_tile, BM, BN);
    }
}

The number of times each element is read from HBM drops from $O(N)$ in the naive implementation to $O(N/BN)$ — reducing memory bandwidth pressure by a factor of $BN$. In practice, cuBLAS and CUTLASS add further levels (block / warp / thread fragment) but the principle is the same.

The Engineering Elegance of FlashAttention

Standard attention: $Q, K, V \in \mathbb{R}^{n \times d}$, computing $S = QK^\top$ (shape $n \times n$), $P = \text{softmax}(S)$, $O = PV$. The memory bottleneck lies in $S$ and $P$ — with $n = 4096, d = 64$, $S$ alone occupies 64MB, and a single round trip through HBM costs 128MB.

The core idea of FlashAttention: fuse attention into a single kernel, never materializing $S$ or $P$. The procedure:

  1. Partition $Q$ into blocks $Q_1, Q_2, \ldots$, each fitting in one SRAM tile
  2. Partition $K, V$ into blocks $K_j, V_j$
  3. Outer loop over $Q_i$, inner loop over $K_j, V_j$
  4. For each pair $(Q_i, K_j)$, compute partial $S_{ij}$, partial softmax, and partial output $O_i += P_{ij} V_j$
  5. Use an online softmax algorithm to merge partial results across different $j$

The online softmax uses a running max and running sum:

$m_i^{\text{new}} = \max(m_i, \max(S_{ij}))$
$l_i^{\text{new}} = e^{m_i - m_i^{\text{new}}} \cdot l_i + \sum_k e^{S_{ij,k} - m_i^{\text{new}}}$
$O_i^{\text{new}} = \frac{l_i}{l_i^{\text{new}}} e^{m_i - m_i^{\text{new}}} O_i + \frac{1}{l_i^{\text{new}}} \sum_k e^{S_{ij,k} - m_i^{\text{new}}} V_{j,k}$

Each new block $K_j, V_j$ updates the running max, running sum, and running output. The final result is equivalent to a full softmax, but the $n \times n$ matrices $S$ and $P$ are never materialized.

Benefits:

  • - HBM accesses reduced from $O(n^2)$ to $O(n^2 / \text{tile\_size})$
  • - Memory footprint reduced from $O(n^2)$ to $O(n)$ (only intermediate m, l, O are stored)
  • - 2–4× speedup, with greater gains for longer sequences

FlashAttention-2 further optimizes warp scheduling; FlashAttention-3 is designed specifically for the Hopper architecture, using TMA (Tensor Memory Accelerator) and WGMMA to overlap asynchronous computation with data movement, achieving 840 TFLOPS/s (85% utilization) in BF16 and 1.3 PFLOPS/s in FP8; FlashAttention-4 is written with CuTeDSL and optimizes both Hopper and Blackwell. Today it is a mandatory component of every major inference engine.

Operator Fusion

What torch.compile in PyTorch 2.0 does under the hood is automatic operator fusion. Classic fusion patterns:

  • Conv + BN + ReLU: BN parameters can be folded into Conv — three-in-one
  • GEMM + Bias + Activation: handled in a single CUTLASS epilogue
  • LayerNorm: inherently a three-step fusion of "subtract mean, divide by std, then scale"
  • RMSNorm + SwiGLU: a common fusion pattern in LLMs

Evaluate performance gains for manual fusion. The general rule: adjacent elementwise, broadcast, and reduction operators should always be fused.

Operator Autotuning

For the same operator, the optimal tile size and thread layout differ across shapes and hardware. Triton AutoTuner, TVM Ansor, and cuDNN's algorithm selection all follow this approach — for each target shape, they try multiple configurations and record the best one. This is done once at compile time or on first run, with results cached for subsequent use.

3.6.4Software Stack Optimization

Focus: Frameworks, runtimes, and scheduling — system-level optimization above the operator layer.

Graph-Level Optimization

The computation graph passes through a series of optimization passes:

  • Constant folding, dead code elimination
  • Operator fusion
  • Memory planning (reusing tensor buffers)
  • Layout optimization (NCHW ↔ NHWC)
  • Operator selection (choosing the fastest among multiple operator implementations)

ONNX Runtime, TensorRT, OpenVINO, and CoreML are all graph compilers of this kind. They typically deliver 2–5× speedups for inference deployment.

Scheduling Optimization

Concurrent kernel execution, CPU-GPU pipelining, and asynchronous data copies are all important for both training and inference. CUDA Streams allow multiple kernels to run simultaneously on the GPU, as long as they are independent.

A classic optimization: overlapping communication and computation. In distributed training, gradients produced by each layer's backward pass immediately begin AllReduce while the previous layer continues its backward pass. With proper configuration, communication is entirely hidden behind computation, and scaling efficiency approaches linear.

Memory Optimization

Gradient Checkpointing: the forward pass stores only a subset of intermediate activations; the rest are recomputed during the backward pass. Trades roughly 30% extra computation for 2–5× memory savings. Essential for training large models.

ZeRO (see Chapter 4, section 5.10): shards optimizer states, gradients, and parameters across devices so each device holds only a fraction.

CPU Offload: moves infrequently accessed state (e.g., optimizer states) to CPU memory, transferring back when needed. Saves GPU memory but introduces latency.

PagedAttention (Chapter 4, section 5.13): virtual memory paging for KV Cache during inference.

I/O Optimization

Training bottlenecks often lie in data loading:

  • - Multi-process DataLoader + pinned memory
  • - Prefetching
  • - Compressed formats (WebDataset, TFRecord)
  • - Distributed file systems (Lustre, GlusterFS) or object storage (S3)

Inference bottlenecks:

  • - Model loading: use mmap + safetensors instead of pickle
  • - First-request cold start: warm up the model, enable keep-alive

Kernel Runtimes

Triton, CUDA Graphs, and CUTLASS each have their place:

  • Triton: the preferred tool for writing new kernels, with Python-like syntax
  • CUDA Graphs: records a series of kernels as a graph and launches the entire graph in a single call. Reduces launch overhead (critical for small-batch inference)
  • CUTLASS: a high-performance GEMM and related template library

3.7 AI in Networking

AI is more than ChatGPT — it has mature applications in network operations, security, and performance optimization.

3.7.1Traffic Classification: Requirements and Challenges

Focus: Traffic identification in the age of encryption — traditional methods have failed, making AI the only viable option.

Why Traffic Classification Is Needed

Network devices need to know "what kind of traffic this is" in order to apply QoS policies, security rules, billing, and compliance auditing. Traditional methods:

  • Port-based: 80 for HTTP, 443 for HTTPS, 53 for DNS...
  • Deep Packet Inspection (DPI): matching characteristic strings in the payload (User-Agent, SNI, protocol signatures)

Both approaches worked in the early 2010s but are no longer viable today —

  1. Port obfuscation: applications no longer use fixed ports; P2P and VPN traffic uses random port assignments
  2. TLS ubiquity: HTTPS accounts for the vast majority of traffic; payloads are encrypted, making DPI blind
  3. QUIC: even the handshake is encrypted (once TLS 1.3 Encrypted ClientHello becomes widespread, SNI is no longer visible)
  4. CDN/cloud: a single IP address can carry hundreds or thousands of applications

As a result, industry today relies almost exclusively on AI methods — identifying traffic from statistical features and behavioral patterns, without depending on payload content.

Classification Granularity

Traffic classification is a multi-level problem:

  • Traffic type: Web / Video / VoIP / Gaming / File Transfer
  • Application identification: Netflix / YouTube / TikTok / Zoom
  • Behavioral identification: casual browsing / video conferencing / file download / malicious scanning
  • User/device type: IoT device / mobile / desktop

Finer granularity demands more complex features and models.

Challenges in Data Acquisition

Where does training data come from? Several approaches:

  • Controlled packet capture: capturing pcap traces from known applications; labels are precise but volume is limited
  • Public datasets: ISCX, USTC, CICIDS, MAWI — labels are not always accurate
  • Weak supervision: using traditional methods to produce coarse-grained labels, with AI refining them
  • Self-supervised: exploiting the natural structure of traffic (forward/backward flows, connection establishment sequences)

Deployment Location and Constraints

Where the classifier is deployed determines the available resources:

  • Edge devices (routers, CPE): compute < 10 GFLOPS; must be extremely lightweight
  • Core devices (datacenter firewalls): GPU/NPU available; can run larger models
  • Cloud-side analysis: offline deep analysis; model size is unconstrained

Edge deployment is the most demanding — XGBoost with a few dozen hand-crafted features is the typical solution; in recent years, compact CNN/Transformer models have also become viable in this context.

3.7.2Traffic Classification: Features and Models

Focus: Feature engineering in practice — which features actually matter and which models to use.

Three Categories of Features

Statistical features (per-flow): statistics computed over a single flow (a sequence of packets sharing the same 5-tuple).

  • - Total packet count, total byte count, duration
  • - Packet size statistics: min / max / mean / std / median / percentiles
  • - Inter-packet interval statistics: same as above
  • - Directionality: forward/backward packet count ratio, byte ratio
  • - TCP flag occurrence counts (SYN, ACK, PSH, RST...)
  • - Window size mean and variance

Temporal features: the temporal patterns of packets, without aggregation.

  • - Size sequence of the first N packets (N = 20–100)
  • - Arrival time sequence of the first N packets
  • - Direction sequence of the first N packets (binary: forward = 0, backward = 1)

Header features (when available): TLS ClientHello cipher suite list, SNI (when unencrypted), TCP options, QUIC version, TLS ALPN, etc.

Hand-Crafted Features + Tree Models

The most common industrial approach. A few dozen features combined with XGBoost / LightGBM. Fast to train, fast to infer, interpretable, and accurate enough (90%+) for common tasks.

import numpy as np
import xgboost as xgb

def extract_flow_features(packets):
    """Extract features from a sequence of packets in a single flow"""
    sizes    = np.array([p.size for p in packets])
    times    = np.array([p.timestamp for p in packets])
    fwd      = np.array([p.direction == 'fwd' for p in packets])

    feat = {}
    # Basic statistics
    feat['n_packets']      = len(packets)
    feat['total_bytes']    = sizes.sum()
    feat['duration']       = times[-1] - times[0] if len(packets) > 1 else 0
    feat['bytes_per_sec']  = feat['total_bytes'] / max(feat['duration'], 1e-6)

    # Packet size distribution
    for stat, fn in [('mean', np.mean), ('std', np.std),
                     ('min', np.min), ('max', np.max),
                     ('p25', lambda x: np.percentile(x, 25)),
                     ('p50', lambda x: np.percentile(x, 50)),
                     ('p75', lambda x: np.percentile(x, 75))]:
        feat[f'size_{stat}'] = fn(sizes)

    # Directional distribution
    feat['fwd_ratio']      = fwd.mean()
    feat['fwd_bytes']      = sizes[fwd].sum()
    feat['bwd_bytes']      = sizes[~fwd].sum()
    feat['fwd_bwd_ratio']  = feat['fwd_bytes'] / max(feat['bwd_bytes'], 1)

    # Inter-arrival time (iat)
    if len(times) > 1:
        iat = np.diff(times)
        feat['iat_mean'] = iat.mean()
        feat['iat_std']  = iat.std()
        feat['iat_max']  = iat.max()

    # Size sequence of the first 20 packets (as individual features)
    for i in range(20):
        feat[f'pkt_size_{i}'] = sizes[i] if i < len(sizes) else 0

    return feat

Deep Learning Approaches

Treating a flow as a "sequence" and processing it directly with 1D-CNN or LSTM:

import torch.nn as nn

class FlowCNN(nn.Module):
    """Input: sequence of (size, iat, direction) tuples for the first N packets"""
    def __init__(self, num_classes, seq_len=50):
        super().__init__()
        # 3 channels: size (normalized), iat (normalized), direction (0/1)
        self.conv = nn.Sequential(
            nn.Conv1d(3, 32, kernel_size=5, padding=2), nn.ReLU(),
            nn.Conv1d(32, 64, kernel_size=5, padding=2), nn.ReLU(),
            nn.MaxPool1d(2),
            nn.Conv1d(64, 128, kernel_size=3, padding=1), nn.ReLU(),
            nn.AdaptiveAvgPool1d(1),
        )
        self.fc = nn.Linear(128, num_classes)

    def forward(self, x):
        # x: [B, 3, seq_len]
        h = self.conv(x).squeeze(-1)
        return self.fc(h)

Advantages: end-to-end learning without hand-crafted features. Disadvantages: large data requirements, high computational cost, black-box nature. Difficult to deploy at the edge.

A recent trend is FlowTransformer-style work — treating packet sequences as token sequences and processing them with small Transformers (3–6 layers). Strong performance but heavier.

Practical Deployment Considerations

Early classification: waiting until a flow ends before classifying is not an option — by then it has already passed through. Classification must be completed within the first N packets (N = 5–20). Feature extraction must be done incrementally, updating as each packet arrives.

Concept drift: applications update frequently (Netflix changes video encoding, Zoom changes transport protocols), causing traffic patterns to shift and model accuracy to degrade. Periodic retraining or online learning is required.

Class imbalance: dominant applications (YouTube, Netflix) account for 90% of traffic while niche applications are extremely rare. Use focal loss or oversampling.

Out-of-distribution samples in production: how to handle new applications not seen during training? Introduce an "Other" class combined with a confidence threshold.

3.7.3Log Anomaly Detection: Requirements and Challenges

Focus: Finding "something's wrong" in massive log streams — one of the foundational problems in automated operations.

The Scenario

Network devices, servers, and applications generate logs: system logs, protocol logs (BGP state changes), security logs (failed logins), application logs (error / warn). Hundreds of devices produce hundreds of millions of entries per day — manual inspection is infeasible.

The goal is to detect "anomalous patterns" in real time and alert operators. Anomaly definitions are often fuzzy — there are known patterns that directly indicate failures (supervised learning), but more often the challenge is "something never seen before" (anomaly detection, unsupervised).

Two Approaches

Rule-based: regex matching against specific patterns triggers alerts. Simple, interpretable, effective with zero examples. But rule maintenance becomes a nightmare — novel failure types always slip through. Mainstream SIEM (Security Information and Event Management) systems still rely heavily on rules.

Machine learning-based: learns "what is normal" from historical logs and flags deviations as anomalies. Not a replacement for rules but a complement — rules catch known issues, ML catches unknown ones.

Challenges with Log Data

Log data presents several unique challenges:

Templated structure: the same event type appears with different parameters. "Connection from 1.2.3.4 failed" and "Connection from 5.6.7.8 failed" represent the same event type with different parameters. Analysis requires log parsing — extracting logs into "template + parameters" (algorithms such as Drain and Spell).

Mixture of unstructured and semi-structured data: syslog is unstructured; JSON logs are structured. Modern systems are gradually moving toward structured logging, but legacy systems remain chaotic.

Temporal context dependency: individual log entries may appear normal, but "1000 consecutive login failures followed by a success" is anomalous — context must be considered.

Extreme class imbalance: anomalies account for less than 0.1% of logs.

Scarce ground truth: who is going to label millions of log entries?

3.7.4Log Anomaly Detection: Methods

Focus: From template extraction to sequence modeling, to LLM-based anomaly explanation.

The Pipeline

flowchart LR LOG[Raw Logs] --> PARSE[Log Parsing
Drain / Spell] PARSE --> TEMP[Template ID Sequence] TEMP --> FEAT[Feature Extraction] FEAT --> DETECT[Detection Model] DETECT --> ALERT[Anomaly Alert] DETECT --> EXPLAIN[LLM Explanation]

Log Parsing: The Drain Algorithm

Drain (2017) is the most widely used log parsing algorithm. The core idea: log templates have fixed prefixes while variable parts are primarily parameters. It uses a fixed-depth parse tree — classifying logs by length, branching level by level on the first few tokens, and performing fine-grained matching at the leaves.

The result is a "template library": each log entry is mapped to a template ID along with its parameter values. Subsequent analysis operates on template ID sequences rather than raw text.

Classical Methods: PCA / Isolation Forest

Represent each time window as a vector of template occurrence counts, then apply anomaly detection algorithms:

  • PCA: normal samples cluster tightly in the low-dimensional principal component space; anomalies lie far away. High reconstruction error indicates an anomaly.
  • Isolation Forest: randomly partitions the data; anomalous points are isolated with fewer splits on average. No distributional assumptions required.
  • One-Class SVM: finds the tightest boundary enclosing normal samples.

Simple, fast, and unsupervised. At baseline levels, capable of detecting coarse-grained anomalies such as traffic spikes and unusual error template patterns.

Deep Learning: DeepLog / LogAnomaly

DeepLog (2017) was the first deep learning log anomaly detection system with significant industrial impact. The idea: treat log template IDs as "words" and a complete log segment as a "sequence." Use an LSTM to learn "given the previous k templates, predict the next one" — a wrong prediction signals an anomaly.

import torch.nn as nn

class DeepLog(nn.Module):
    def __init__(self, num_templates, hidden_size=128, num_layers=2):
        super().__init__()
        self.emb  = nn.Embedding(num_templates, 64)
        self.lstm = nn.LSTM(64, hidden_size, num_layers, batch_first=True)
        self.fc   = nn.Linear(hidden_size, num_templates)

    def forward(self, x):
        # x: [B, seq_len] template IDs
        e = self.emb(x)
        h, _ = self.lstm(e)
        return self.fc(h[:, -1])   # predict distribution over next template

# At inference time: if the actual next template is not in the top-k predictions, flag as anomaly
def detect(model, seq, k=9):
    logits = model(seq)
    topk = logits.topk(k).indices.squeeze()
    return seq[-1] not in topk

LogAnomaly further accounts for anomalies in parameter values. More recent work (LogBERT, NeuralLog) replaces the LSTM with a Transformer.

Log Analysis in the LLM Era

LLMs have changed the game — feeding log snippets directly to an LLM and asking "what anomalies are here, and what might be causing them?" Advantages:

  • - No log parsing required; natural language is processed directly
  • - Joint reasoning across log patterns
  • - Output is a human-readable explanation, not merely an "anomalous/normal" verdict

Disadvantages:

  • - Context window limitations restrict how many logs can be examined at once
  • - High inference cost makes real-time processing difficult
  • - Hallucinations

Industrial practice: a two-stage approach. Classical models perform coarse-grained filtering, then an LLM provides explanation and root-cause analysis for suspicious segments — combining the strengths of both.

The Evaluation Challenge for Anomaly Detection

How do you evaluate without ground truth? Several compromise approaches:

  • - Manual review over a period of time, annotating recall
  • - Injecting known failures (chaos engineering) and checking whether they are detected
  • - Comparing against the alerting system — checking whether the model also flags events at alert timestamps

An important post-deployment metric: alert suppression rate — how much the total alert count is reduced without missing critical failures. The greatest pain point for operations teams is not missed alerts but "alert fatigue."

3.7.5KPI Time-Series Forecasting: Requirements and Challenges

Focus: Predicting QPS, CPU, and bandwidth — the foundation of capacity planning and proactive operations.

The Scenario

Core KPI time series in network operations: bandwidth utilization, CPU/memory, connection count, packet loss rate, latency, and error rate. Typical forecasting needs:

  • Capacity planning: predicting traffic trends over the next three months to inform expansion decisions
  • Auto-scaling: predicting load for the next hour to scale up proactively and avoid instability
  • Anomaly alerting: predicting the "normal" trajectory and alerting when actual values deviate
  • SLA management: predicting potential SLA violations in advance to enable early intervention

Properties of KPI Time Series

KPI time series data exhibits characteristic patterns:

  • Trend: long-term upward or downward movement
  • Seasonality: daily cycles (daytime/nighttime), weekly cycles (weekends), annual cycles
  • Residual: noise and anomalies

A typical QPS curve is a superposition of all three components.

Change points are particularly tricky — deployments, promotions, and failure recoveries can cause abrupt shifts in statistical properties. Forecasting models must either be able to detect change points or adapt quickly after one occurs.

Challenges

Multivariate: CPU, memory, and QPS are correlated with one another; joint forecasting outperforms independent forecasting.

Multi-scale: forecasts are needed at the second, minute, hour, and day level; different scales require different models.

Cold start: insufficient historical data when a new service launches.

Anomaly contamination: training data that includes past failure periods can cause the model to treat those anomalies as "normal." Anomalous periods must be removed in advance.

Uncertainty quantification: point forecasts are insufficient; prediction intervals are needed — "QPS over the next hour will likely be 5000–7000" is more useful for decision-making than "approximately 6000."

3.7.6KPI Time-Series Forecasting: Methods

Focus: From ARIMA to Prophet to Transformer — the evolution and when to use each.

Classical Statistical Methods

ARIMA (AutoRegressive Integrated Moving Average):

$\phi(L)(1-L)^d y_t = \theta(L) \epsilon_t$

Three parameters $(p, d, q)$: $p$ is the autoregressive order, $d$ is the number of differencing steps (to achieve stationarity), and $q$ is the moving average order. Parameters are selected using ACF/PACF plots or information criteria (AIC, BIC).

ARIMA performs well on stationary linear series, is computationally inexpensive, and is interpretable. For strong seasonality, SARIMA (seasonal ARIMA) is used. It remains the preferred baseline in many settings today.

Exponential smoothing: Holt-Winters triple exponential smoothing separates trend and seasonality. The kind built into Excel — simple, adequate, and old-fashioned.

Prophet

Open-sourced by Facebook in 2017. Decomposes the time series into trend + seasonality + holidays + noise, modeling each component independently:

$y(t) = g(t) + s(t) + h(t) + \epsilon$
  • - Trend $g$: piecewise linear or logistic
  • - Seasonality $s$: Fourier series
  • - Holidays $h$: user-specified list of dates

Prophet's strength is being analyst-friendly — business knowledge can be incorporated manually (next week's major sale event, a new feature launch, annual subscription renewals). Parameters are interpretable, and the model can be tuned when things go wrong.

from prophet import Prophet
import pandas as pd

# Data must have two columns: (ds, y)
df = pd.DataFrame({'ds': timestamps, 'y': qps_values})

m = Prophet(
    yearly_seasonality=False,
    weekly_seasonality=True,
    daily_seasonality=True,
    changepoint_prior_scale=0.05,  # higher values allow more trend changes
)
m.add_country_holidays(country_name='CN')
m.fit(df)

# Forecast the next 24 hours with a 5-minute granularity
future = m.make_future_dataframe(periods=24*12, freq='5min')
forecast = m.predict(future)
# forecast contains yhat (point forecast), yhat_lower, yhat_upper (80% confidence interval)

Deep Learning Methods

LSTM / GRU: the most straightforward deep learning approach for time series. Takes a historical window as input and outputs the next step (or multiple steps). Multi-step forecasting commonly uses seq2seq or iterative prediction.

N-BEATS (2019): fully MLP-based, using a "backcast/forecast" architecture. Outperformed all RNN and Transformer methods in the M4 time series competition. Demonstrates that MLPs remain competitive for time series.

Temporal Fusion Transformer (2020): a Transformer variant designed specifically for time series. Supports multivariate inputs, static covariates, and future-known covariates (such as pre-scheduled promotion dates).

PatchTST (2023): divides the time series into patches (analogous to ViT's treatment of images), then encodes them with a Transformer. Simple yet powerful.

Informer / Autoformer: Transformer variants optimized for long-sequence forecasting.

Foundation Models

Since 2024, time series foundation models have emerged: Chronos (Amazon), TimesFM (Google), Moirai. Pretrained on large and heterogeneous time series corpora, they enable zero-shot forecasting — new series can be used directly without task-specific training.

The practical picture is mixed: Prophet/ARIMA is sufficient for simple series; deep models outperform on complex multivariate and long-horizon tasks; foundation models are the only option when no historical data is available.

Evaluation

Common metrics:

  • MAE: Mean Absolute Error. Robust to outliers.
  • RMSE: Root Mean Squared Error. Penalizes large deviations heavily.
  • MAPE: Mean Absolute Percentage Error. Relative error, scale-independent, but unstable when true values are near zero.
  • sMAPE: Symmetric MAPE, addressing the issue above.
  • Quantile Loss: evaluates interval forecasts.

In practice, directional accuracy often matters more — whether the model correctly predicted "up or down" is more actionable than the precise value. Directional accuracy is a useful supplementary metric.
