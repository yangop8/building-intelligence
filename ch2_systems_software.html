系统软件

从 CPU 到 DPDK/VPP —— AI 工程师需要的系统视角
Chapter 2 · 内部学习材料 · 共 28 节

系统软件对 AI 工程师意味着什么？三件事。第一，训练和推理都跑在系统之上——GPU 再快，数据从磁盘到显存要过 PCIe、经内核、走文件系统缓存，每一层都可能成为瓶颈。第二，AI 基础设施本身就是系统软件——CUDA/CANN 是驱动层，vLLM 调度器借鉴了 OS 进程调度，PagedAttention 照搬虚拟内存分页。第三，在 AI 辅助编程时代，对底层系统的"判断力"比"动手力"更值钱——代码可以让 AI 写，但架构决策、性能分析、故障定位仍然靠人。

这章按"芯片 → OS → 语言/编译 → 数据存储 → 通信"的栈式顺序铺开 28 节。和第 1 章一样，深度不对称：内存管理、并发同步、Rust 所有权、编译优化、DPDK/VPP 几节会挖得深；设备驱动、传统关系数据库、某些偏传输协议会点到为止。内部专有技术栈（如某些内部中间件、私有协议）全部改用开源通用等价物讲解——讲清楚 DPDK，理解任何内部等价框架都只是再学一层包装。

PART ONE

2.1 CPU 与核心架构

AI 训练离不开 GPU/NPU,但理解 CPU 才是理解一切的前提。

2.1.1CPU 总体架构

定位:ISA、流水线、缓存、多核——现代处理器的骨架。

ISA:CPU 的对外契约

指令集架构(ISA)是 CPU 对软件暴露的编程模型,定义了有哪些指令、有多少寄存器、内存如何寻址。三大主流 ISA:

ISA风格代表场景
x86 / x86-64CISCIntel / AMDPC、服务器
ARMRISCApple M 系列、Ampere移动、云、嵌入式
RISC-VRISC (开放)SiFive、阿里平头哥新兴,IoT 到 HPC

CISC(复杂指令集)vs RISC(精简指令集)的对立已经没当年那么尖锐——现代 x86 在前端把复杂指令拆成微操作(μops),后端执行单元和 RISC 无异。但从软件视角,ISA 还是决定了你能用什么汇编指令、向量扩展是 AVX-512 还是 SVE、有没有 TSO 内存模型。

RISC-V 的特殊地位是开放——任何人可以免费实现,加自己的扩展。它正以罕见的速度改写嵌入式和加速器芯片的版图。

微架构:ISA 之下的现实

同样的 ISA,不同微架构性能差几倍。现代高性能 CPU 核心有这些关键部件:

流水线:把"取指 → 译码 → 执行 → 访存 → 写回"五个阶段重叠起来,每周期完成一条指令。但分支会打断流水,所以有分支预测器——用历史统计猜下一条指令在哪。预测准确率动辄 95%+。预测错了罚 15–20 周期冲刷流水。

乱序执行:硬件发现指令间没有依赖就并行发射。程序员写的顺序代码,硬件按数据流重排执行。Spectre/Meltdown 漏洞就是利用了乱序执行的副作用。

SIMD:单指令多数据。AVX-512 能一条指令算 8 个 double。深度学习没有 GPU 的年代,CPU 推理主力就靠它。

缓存层次

内存比 CPU 慢两个数量级(100+ ns vs <1 ns)。解决办法是三级缓存:

层级大小延迟范围
L1 (data + instr)32KB + 32KB~4 cyclesper-core
L2256KB–1MB~12 cyclesper-core
L3 (LLC)数 MB–数十 MB~40 cycles共享
DRAMGB~200 cycles全局

理解缓存是理解性能的钥匙。顺序访问数组缓存预取器能准确猜下一行,近乎零成本;随机访问链表几乎每次都 cache miss,慢几十倍。这就是为什么 AI 里张量基本都用连续内存布局。

// 缓存对性能的戏剧性影响
#define N 16384
int a[N][N];

// 版本1: 行序访问, cache 友好
for (int i = 0; i < N; i++)
    for (int j = 0; j < N; j++) sum += a[i][j];

// 版本2: 列序访问, 每次跨 N*sizeof(int) 字节
for (int j = 0; j < N; j++)
    for (int i = 0; i < N; i++) sum += a[i][j];

// 版本2 通常比版本1 慢 5–10 倍——纯粹因为 cache miss

多核与一致性

多核共享内存时必须保证"我写的你能看到"。硬件通过 MESI 协议(Modified/Exclusive/Shared/Invalid)维护缓存行状态。一行数据在一个核修改,其他核的副本会被标记 Invalid,下次访问要重新加载。

这带来一个隐蔽陷阱——False Sharing(伪共享)。两个不相关的变量恰好在同一缓存行(64 字节),两个核各自频繁写各自的变量,都会互相触发 cache invalidation。解决办法是缓存行对齐——给关键变量加 alignas(64),或 padding 到 64 字节。

在 AI 里

AI 训练的数据预处理、tokenization、采样逻辑都跑在 CPU 上。一个常被忽略的细节:DataLoader 的 CPU 性能经常是训练瓶颈。GPU 算得飞快,CPU 来不及供给数据。解决办法是多进程 DataLoader、绑定 NUMA 节点、用 pinned memory 减少拷贝。所有这些优化都建立在对 CPU 架构的基本理解上。

2.1.2网络处理单元

定位:专用硬件加速网络包处理——从 NP 芯片到智能网卡。

为什么需要专用单元

线速处理 100Gbps 网络流量,按 64 字节小包算是 148.8 Mpps(million packets per second)。每包预算 6.7 ns——一次 DRAM 访问都做不完。仅靠通用 CPU 软件难以实现,必须依赖硬件。

几种硬件加速路线

NP (Network Processor):专为包处理设计的多核处理器,指令集包含报头解析、查表、校验等操作。每个核跑一段微码,流水线处理报文。代表:Broadcom XLP、Intel IXP。

ASIC (交换芯片):针对固定协议完全硬化。以太网交换机里的 Broadcom Tomahawk、Marvell Teralynx 都是 ASIC,查表、转发、VLAN 处理全在硬件。延迟可做到百纳秒级,但功能写死。

智能网卡/DPU:NVIDIA BlueField、Intel IPU、AWS Nitro。本质是网卡上加了一颗 ARM SoC,可以跑完整 OS,把网络/存储/安全功能从主机 CPU 卸载到网卡。数据中心级加速趋势。

P4 可编程管线:介于 ASIC 和 NP 之间。硬件是固定流水线,但流水线各阶段的行为用 P4 语言定义,上电加载。代表:Barefoot Tofino。

常见硬件加速功能

  • RSS (Receive Side Scaling):网卡对报文按 5-tuple(源 IP、目的 IP、协议、源端口、目的端口)哈希,分发到不同 CPU 核的接收队列。让多核并行处理互不冲突。
  • 硬件 LRO/GRO:把同一流的多个小包合并成大包再交给协议栈。省 syscall 次数。
  • TSO/GSO:反方向——协议栈把大段数据交给网卡,网卡自己分片成 MTU 大小。
  • Checksum/Crypto 卸载:TCP 校验、TLS 加解密直接由网卡完成。
  • ECMP 硬件实现:等价多路径路由,硬件按 hash 分发。

在 AI 里

大模型训练集群(GPU 通过 RDMA over InfiniBand/RoCE 通信)依赖网卡硬件能力。GPUDirect RDMA 让网卡直接读写显存,绕过 CPU 内存。一次 AllReduce 的延迟主要取决于网卡的 RDMA 性能而不是 GPU 算力。

2.1.3GPU 与 NPU 加速

定位:AI 算力的载体——并行计算与张量加速的硬件。简短,详细在第 4 章。

这节和第 4 章 4.2 有重叠,这里只建立基本坐标。

GPU 的并行模型

GPU 的 DNA 是图形渲染——海量独立像素的着色天然可并行。现代 GPU 把这个能力转用于通用计算(GPGPU)。一张 NVIDIA H100 有约 16896 个 CUDA 核心 + 528 个 Tensor Core,FP16 Tensor Core 算力约 989 TFLOPS（密集）/ 1979 TFLOPS（2:4 稀疏）。

GPU 编程模型叫 SIMT(单指令多线程)——32 个线程组成一个 warp,一条指令同时作用于 32 个数据。这和 SIMD 精神一致,区别是 warp 内如果有分支(if),不同线程走不同路径,GPU 会串行执行两边——即 warp divergence,性能崩。

NPU 的张量优先设计

NPU(神经网络处理单元)从一开始就为张量计算设计。核心架构是脉动阵列(systolic array):一个二维网格的 MAC(乘加)单元,数据像脉搏一样流过,每个单元在固定时序做乘加。Google TPU 是最著名的脉动阵列。

华为昇腾走的是达芬奇架构:Cube 单元做矩阵乘、Vector 单元做逐元素、Scalar 单元做控制流。算子开发者必须显式把计算分派到合适单元。

存储结构

GPU/NPU 的内存层次更极端:

层级容量带宽
寄存器每线程 64KB极快
共享内存 / SRAM~100KB per block~几 TB/s
HBM80GB (H100)~3 TB/s

HBM 带宽虽大,但比片上 SRAM 还是慢 10×。FlashAttention 这类算子优化的本质就是尽量让计算留在 SRAM,减少 HBM 往返。

2.1.4SoC 异构计算

定位:一颗芯片整合 CPU/GPU/NPU/加速器——现代嵌入式和边缘 AI 的范式。简短。

SoC(System on Chip)不是新概念。手机处理器从 iPhone 时代起就是 SoC——一颗芯片里集成 CPU 大小核、GPU、ISP(图像处理)、神经网络加速、调制解调器。数据中心 SoC 趋势更明显:AWS Graviton、Ampere Altra 都把 CPU、内存控制器、PCIe、网络子系统一起 die 上集成。

异构计算的挑战

多种算力单元同一颗芯片共享内存/总线,挑战是:

  1. 数据搬运:CPU 内存和 GPU/NPU 内存之间的拷贝是主要开销。统一内存(CUDA Unified Memory、Apple Silicon 的 Unified Memory Architecture)试图解决,但真正零拷贝需要硬件一致性支持。
  2. 调度:哪段代码跑在 CPU、哪段跑在加速器?框架层(PyTorch、TensorFlow)自动决定,但边界情况多。
  3. 功耗墙:所有单元共享芯片功率预算。AI 推理跑满 NPU 时 CPU 必须降频。

常见卸载引擎

网络设备 SoC 里除了 CPU 和 NP,常见还有:

  • - 压缩引擎(DEFLATE/LZ4 硬件加速)
  • - 加解密引擎(AES、SHA、国密 SM4)
  • - 正则匹配引擎(DFA 执行,用于 IPS 入侵检测)
  • - 查表引擎(TCAM,用于路由查找)

这些在 AI 场景里不直接相关,但理解"可卸载计算"的思路对理解 DPU、智能网卡大有帮助——它们本质都是 SoC 思想在网络领域的延伸。

PART TWO

2.2 操作系统

OS 是所有上层软件的隐形基础设施——AI 框架调度、容器、网络,无一不经过它。

2.2.1OS 概述

定位:OS 是什么、做什么、几种主流 OS 的差异。

操作系统做两件事:管理硬件资源(CPU、内存、磁盘、网络),给应用提供抽象(进程、文件、socket)。Dijkstra 的老话:"抽象不是隐藏细节,而是创造一个新的精确的层次"。OS 正是这样的层次。

主流 OS 谱系

系列内核代表场景
Unix-like宏内核Linux、FreeBSD服务器、容器、超算
macOS / iOSXNU (混合)Darwin桌面、移动
WindowsNT (混合)Windows 10/11桌面、企业服务器
AndroidLinux 定制AOSP移动
RTOS实时微内核FreeRTOS、VxWorks、Zephyr嵌入式、工控

AI 基础设施几乎 100% 跑在 Linux 上。Linux 内核源码 3000 万行,但你日常打交道的核心子系统只有五六个:进程调度、内存管理、VFS、网络栈、块设备、字符设备。

宏内核 vs 微内核

宏内核(Linux):文件系统、网络栈、驱动全部在内核态,函数调用直接互通。快但任何子系统 bug 都能 crash 整机。

微内核(seL4、Mach):内核只保留最核心的 IPC、调度、内存保护,其它服务在用户态跑,通过消息传递通信。稳健但 IPC 开销大。

现实是大多数现代 OS 都是混合——Linux 也发展出了可加载内核模块(LKM)、eBPF 这些动态机制,往微内核方向靠拢。

实时 OS

RTOS 的核心指标不是吞吐而是最坏情况响应时延。调度必须可预测——高优先级任务就绪必须在固定时间内得到 CPU。Linux 默认不是 RTOS,但 PREEMPT_RT 补丁能让它接近实时能力。工业控制、汽车电子、机器人必须用 RTOS。

2.2.2进程与调度

定位:进程是 CPU 资源分配的基本单位,调度器决定谁跑谁等。重点讲 CFS 和现代调度挑战。

进程与线程

Linux 里进程和线程在内核看来是同一个东西——都是 task_struct。创建方式决定它是进程(fork)还是线程(clone 带 CLONE_VM 等标志)。核心区别:进程有独立地址空间、线程共享地址空间。

一个 task_struct 包含:

  • - PID、父进程、用户/组信息
  • - 运行状态(RUNNING、INTERRUPTIBLE、UNINTERRUPTIBLE、ZOMBIE...)
  • - 调度策略和优先级
  • - 打开的文件表、信号处理表
  • - 内存映射(mm_struct)
  • - 上下文(寄存器、栈)

fork 与 copy-on-write

fork() 创建子进程,传统语义是复制整个地址空间。朴素实现会巨慢——因为绝大多数子进程立刻 exec 新程序,复制的内存用不上就扔了。

Copy-on-write (COW) 是优雅的解决:fork 时不复制,只标记所有页为只读、引用计数 +1。任一方写某页时触发缺页,内核这时才真复制那一页。结果:fork 几乎和启动新线程一样快。

孤儿与僵尸

父进程先死、子进程还活着——子进程成为孤儿进程,被 init (PID 1)接管。

子进程先死、父进程没 wait——子进程成为僵尸进程,内核保留 PID 和退出状态等父进程读取。僵尸不占内存但占 PID 表项。孤儿 vs 僵尸是 Unix 面试经典题。

CFS:Linux 的默认调度器

CFS(Completely Fair Scheduler)的核心思想是"理想多任务 CPU"——如果 CPU 无限快,$N$ 个任务应该每个同时得到 $1/N$ 的算力。现实 CPU 是时分的,CFS 近似这个理想。

实现方式:每个任务维护一个虚拟运行时(vruntime),每次被调度运行后 vruntime += 实际运行时间 × 权重。调度时选 vruntime 最小的任务。用红黑树组织,$O(\log n)$ 选出下一个。

nice 值([-20, 19])通过权重影响 vruntime 推进速度:nice 值小权重大、vruntime 涨得慢、被选中的机会多。这是对"优先级"的平滑实现。

注意：自 Linux 6.6（2023 年末）起,默认公平调度器已由 CFS 切换为 EEVDF（Earliest Eligible Virtual Deadline First）。EEVDF 在保留虚拟时间公平性的同时引入了虚拟截止期限,改善了延迟尾部表现,减少了交互任务的调度抖动。核心数据结构仍是红黑树,$O(\log n)$ 复杂度不变。

实时调度

CFS 之外 Linux 还有实时调度类(SCHED_FIFO、SCHED_RR)。FIFO 没有时间片,高优先级任务会一直跑到主动让出或阻塞——适合硬实时场景。RR 是 FIFO + 时间片轮转。

另外还有 DEADLINE 调度(EDF 算法):每个任务声明 (runtime, deadline, period),内核保证每周期内完成 runtime 且不超 deadline。是目前 Linux 上最接近硬实时的调度类。

优先级反转

经典故障模式:低优先级任务 L 持有锁,中优先级任务 M 抢占 L(但 M 不需要那把锁),高优先级任务 H 等锁——H 间接被 M 阻塞,优先级倒置。1997 火星探路者号就栽在这上面。

解决:优先级继承——L 持锁期间临时提升到 H 的优先级,不怕被 M 抢。优先级天花板——静态分配更高。这些机制在 RT-mutex 里实现。

上下文切换的成本

一次上下文切换大约 1–5 μs,包含:保存寄存器、切换页表(重装 CR3,TLB 部分失效)、切换内核栈、调度决策。看起来快,但 1000 Hz 定时中断 + 频繁抢占下累积开销可观。

AI 训练里通常用CPU 亲和(taskset 或 sched_setaffinity)把 DataLoader worker 绑到固定核,减少切换和 cache miss。

2.2.3内存管理

定位:虚拟内存、页表、分配器、HugePages、OOM——全章最值得挖深的一节。

虚拟内存的契约

每个进程看到一个独立的、连续的、仿佛全是自己的地址空间(x86-64 上 128TB 用户空间)。这是虚拟内存给的幻觉。硬件(MMU)和 OS(内核页表管理)合作维持这个幻觉。

4 级页表(x86-64):把 48 位虚拟地址拆成 9+9+9+9+12 位,前四组 9 位分别索引四级页表,最后 12 位是页内偏移。为啥这么设计?单级页表全覆盖要 $2^{36} \times 8$ B = 512GB,每进程一份不现实。多级页表按需分配——只有被用到的虚存范围才真的有页表项。

TLB

每次内存访问都查 4 级页表意味着 4 次额外内存访问,太慢。硬件用 TLB(Translation Lookaside Buffer)缓存最近用过的虚拟 → 物理映射。TLB 命中时翻译 ~0 开销,miss 时走完 4 级叫 page walk,成本 ~100 cycles。

TLB 只有几十到几百个条目——随机访问大数组会让 TLB miss 率飙升。

HugePages

一般页 4KB,TLB 覆盖 4KB × 512 entries = 2MB 数据。HugePages 2MB(THP)或 1GB(hugetlbfs),一个 TLB 条目覆盖 2MB 或 1GB。对大内存应用(数据库、AI 训练)效果明显。

DPDK、数据库、JVM 都会强烈推荐 HugePages——不是为了省内存,是为了减少 TLB miss。

物理内存分配

内核的物理页分配器是Buddy System:把物理内存按 2 的幂次划分(4KB、8KB、16KB...4MB),每个尺寸维护一个空闲链表。分配时找大于等于请求的最小块,如果过大就对半分(buddy),其中一半给请求者、另一半入小一级链表。释放时检查 buddy 是否也空闲,空闲就合并。

这套算法在避免外部碎片和快速分配之间找到了平衡。但 4KB 粒度对小对象(几十字节)太粗,再加一层:

Slab/Slub:内核对象专用分配器。预分配一些页作为"slab",切分成固定大小的对象(比如 task_struct 2KB),维护对象粒度的空闲链表。分配/释放 O(1),cache 友好。

用户态 malloc

用户程序的 malloc 不直接走内核。glibc 的 ptmalloc、jemalloc、tcmalloc 都在 brk() 或 mmap() 拿大块内存后自己管理小对象分配。jemalloc(Redis、Rust 默认)和 tcmalloc(Google)的核心思想是per-thread cache——线程本地有一份小对象缓存,分配无锁,只在耗尽时才访问全局池。多线程下比 ptmalloc 快得多。

页面回收

物理内存不够时要回收。Linux 维护两条 LRU 链表:active 和 inactive。页访问会提升到 active 头,长期不访问降到 inactive 尾。回收从 inactive 尾开始。

kswapd 内核线程在后台做回收,保持一定量空闲页。回收路径:

  1. 干净 page cache 页:直接丢掉(下次再从文件读)
  2. 脏 page cache 页:先写回磁盘再释放
  3. 匿名内存(堆、栈):换出到 swap 分区(如果开了)

swap 在服务器场景常被禁用——宁可 OOM 也别让程序悄悄变慢。AI 训练机器必关 swap。

OOM Killer

当 swap 也无法缓解时,内核启动 OOM Killer 选一个进程杀掉。评分算法综合考虑内存占用、运行时间、oom_score_adj(可调)。实践中:关键服务应设负的 oom_score_adj 降低被杀概率。

在 AI 里

AI 训练的内存问题大多在 GPU 显存,CPU 内存相对宽裕。但以下场景容易出问题:

  • - 多 DataLoader worker 的 fork:由于 COW,主进程大对象(如数据集索引)被子进程"访问"时逐页复制,实际复制所有页后内存翻倍。Python 多进程训练 OOM 常因此。
  • - page cache 压垮小内存机器:读大数据集时 page cache 把内存占满,某些内核版本下释放不及时。
  • - zero-copy 诉求:训练数据从磁盘到 GPU,希望避免"磁盘 → 内核 → 用户态 → GPU"的四次拷贝。mmap + O_DIRECT + GPU pinned memory 组合能减少拷贝。
# PyTorch pinned memory 示例: 让 host→device 拷贝用 DMA
train_loader = DataLoader(
    dataset,
    batch_size=32,
    num_workers=8,
    pin_memory=True,   # 分配页锁定内存, GPU 可直接 DMA
)
# 训练循环里配合 non_blocking=True 做异步拷贝
for x, y in train_loader:
    x = x.cuda(non_blocking=True)
    y = y.cuda(non_blocking=True)

2.2.4存储与文件系统

定位:VFS 抽象、i-node、page cache、I/O 调度——大数据集加载的底座。

VFS:一切皆文件

Unix 的核心哲学之一。文件、目录、管道、socket、设备、proc 信息——所有都通过 open/read/write/close 操作。VFS(Virtual File System)是内核提供的统一接口层,下面挂不同具体文件系统:ext4、xfs、btrfs、overlayfs(容器)、nfs(网络)。

i-node

i-node 是文件的元数据对象,包含:大小、owner、权限、时间戳、指向数据块的指针。文件名不在 i-node 里——文件名存在目录里(目录也是文件,内容是 name→inode 映射)。

硬链接:两个目录项指向同一 i-node,i-node 里的引用计数 +1。删除文件实际是删目录项 + 引用计数 -1,计数到 0 才真释放。

软链接(symlink):特殊类型的文件,内容是目标路径。解析时递归。

Page Cache

读文件时内核把数据缓存在 page cache(物理内存中)。第二次读同一块数据直接命中,无磁盘 I/O。写文件默认也是先写 page cache(write-back),内核后台刷盘。

free 命令看到的 "buff/cache" 就是 page cache。它不是"浪费",是内存的高效使用——随时可回收。

I/O 调度

机械盘时代,I/O 调度器(CFQ、deadline)关注合并邻近请求、减少寻道。SSD 时代寻道成本没了,Linux 默认改用 mq-deadline 或 none(NVMe 多队列直接下发)。

同步 vs 异步 I/O

标准 read/write 是同步阻塞的。高性能场景用:

  • - O_DIRECT:绕过 page cache,直接 DMA 到用户缓冲区。数据库常用。
  • - aio(Linux AIO):传统异步 I/O,只在 O_DIRECT 下真正异步,功能弱。
  • - io_uring(2019+):现代异步 I/O,共享环形队列的 submission/completion queue,零 syscall 路径下能做到 syscall-less I/O。AI 训练的数据加载、数据库、高性能 web 服务都在转向它。
// io_uring 的核心环形队列结构 (简化)
struct io_uring_sqe {  // Submission Queue Entry
    __u8  opcode;      // IORING_OP_READ, _WRITE, ...
    __s32 fd;
    __u64 off;
    __u64 addr;
    __u32 len;
    // ...
};
struct io_uring_cqe {  // Completion Queue Entry
    __u64 user_data;
    __s32 res;
    __u32 flags;
};
// 用户态直接填 SQE、读 CQE, 无需 syscall (轮询模式下)

在 AI 里

训练数据集动辄 TB,加载性能关键。几个实践:

  • - WebDataset、TFRecord 这类打包格式:把千万小文件打包成少量大文件,避免 i-node/目录遍历开销。
  • - 预读 + 多 worker:DataLoader 的 prefetch_factor 控制每个 worker 预读多少 batch。
  • - POSIX 语义之外:分布式训练里 NFS、Lustre、S3 等后端,性能模型和本地盘迥异,要测才知道瓶颈在哪。

2.2.5异常与中断

定位:CPU 与外部世界通信的机制——中断、软中断、tasklet、workqueue。

中断分类

硬中断(IRQ):硬件触发,CPU 立即停下当前工作跳到中断处理程序。中断向量表给每个中断号对应一个处理函数。

软中断/异常:CPU 自己触发——除零、缺页、系统调用(syscall 本质是陷入)。

中断处理的两段式

中断处理有一条铁律:必须快——中断上下文里不能睡眠、不能获取可能阻塞的锁。但网络包到达这种事有时候要做复杂处理(解析协议、查路由、复制数据)。

Linux 的解决是"上半部 + 下半部":

  • 上半部(top half):中断处理程序本身。只做最必要的——应答硬件、把数据搬到内存缓冲、调度下半部。禁中断状态,越短越好。
  • 下半部(bottom half):允许中断、可调度。三种机制:
机制特点适用
softirq静态类型、per-CPU、不能睡眠网络收发、定时器、块 I/O
taskletsoftirq 之上,动态创建,同一 tasklet 同时只在一核运行中等开销驱动
workqueue内核线程上下文、可睡眠需要睡眠的延迟工作

网络收包路径是 softirq 的典型用户——硬中断来了之后把 skb 入队,NET_RX_SOFTIRQ 在下半部把 skb 交给协议栈。这也是为什么高速网络下 ksoftirqd 经常占满一个核。

中断亲和与负载均衡

默认所有中断可能都被一个核处理(尤其廉价网卡)。/proc/irq/<n>/smp_affinity 可以绑定中断到指定核。多队列网卡配合 RSS,把不同队列的中断分到不同核,实现真并行。

在 AI 里

AI 相关不多。但 GPU 驱动就是中断密集——CUDA kernel 完成时 GPU 发中断,驱动在下半部通知用户态。中断处理开销在高 QPS 推理服务(每秒几千请求)下不可忽略。

2.2.6并发、互斥与同步

定位:多核并发的基本工具——原子、锁、RCU、内存模型。重点挖深。

竞争的本质

多线程读写共享数据,如果没同步,会出问题。最简单的反例:

int counter = 0;
// 两个线程各自:
for (int i = 0; i < 1000000; i++) counter++;
// 最终 counter 远小于 2000000

counter++ 不是原子的——实际是 load/add/store 三步,两个线程交错执行时会丢更新。

原子操作

硬件提供原子指令:x86 的 LOCK 前缀、ARM 的 LDREX/STREX。C11 的 _Atomic、C++11 的 std::atomic、Rust 的 AtomicUsize 都是对这些指令的语言封装。

最关键的原子原语是 CAS(Compare-And-Swap):

// 原子地: 如果 *ptr == expected 则写 new, 返回 true; 否则读到 *ptr 更新 expected, 返回 false
bool cas(int *ptr, int *expected, int new);

CAS 是所有无锁数据结构的基础。典型用法是"循环 CAS 直到成功":

int old, new;
do {
    old = *counter;
    new = old + 1;
} while (!cas(counter, &old, new));

锁的种类

自旋锁(Spinlock):获取失败就忙等。适合持锁时间极短、多核场景。内核中断处理里必须用自旋锁(不能睡)。

互斥量(Mutex):获取失败就睡眠,被唤醒时再试。适合持锁时间长。开销大于自旋锁(上下文切换),但不浪费 CPU。

读写锁:多读共享,写独占。读多写少场景有优势。缺点:写饥饿问题(持续读流会让写永远等不到)。

信号量(Semaphore):带计数的锁,允许 N 个并发持有。资源池常用。

条件变量:不是锁,是"等待某个条件成立"的原语。典型配合 mutex 使用。

RCU

Read-Copy-Update 是 Linux 内核里最精妙的并发原语。适合读极多写极少的场景。核心思路:

  1. 读者完全不加锁——直接读旧版本指针
  2. 写者复制一份数据,改完后原子地替换指针
  3. 旧版本数据延迟释放——等所有已开始的读者都结束(grace period)后再 free

RCU 的读路径开销近乎零,几乎和单线程一样。代价是写复杂、内存短期翻倍、释放延迟。Linux 的路由表、namespace、模块列表都用 RCU。

// RCU 读者: 完全无锁
rcu_read_lock();
struct foo *p = rcu_dereference(global_ptr);
// 使用 p...
rcu_read_unlock();

// RCU 写者
struct foo *new = kmalloc(...);
*new = (struct foo){ .x = 42, ... };
struct foo *old = rcu_dereference(global_ptr);
rcu_assign_pointer(global_ptr, new);   // 原子指针替换
synchronize_rcu();                      // 等所有读者离开
kfree(old);                             // 安全释放

内存模型

程序员以为自己写的代码按顺序执行,但 CPU 和编译器都会重排指令。单线程下感知不到(满足"数据依赖"即可),多线程下重排可能让另一个核看到"不可能"的状态。

例子:

// Thread A:                Thread B:
x = 1;                       if (flag) {
flag = 1;                        assert(x == 1);  // 可能失败!
                             }

CPU 或编译器可能把 flag = 1 提前到 x = 1 之前——Thread B 看到 flag=1 但 x 还是 0。

解决:内存屏障(memory barrier)。x86 是 TSO(Total Store Order)内存模型,相对宽松——写操作保序、读可能提前。ARM 是 weakly-ordered,啥都可能重排,必须显式用屏障。

高层语言抽象:C++11 的 std::memory_order、Rust 的 Ordering,有 Relaxed、Acquire、Release、AcqRel、SeqCst 五档。SeqCst 最强(顺序一致),开销最大;Relaxed 最弱(只保证原子性),最快。真正懂得用是高水平系统工程师的分界线。

MESI 与缓存一致性

多核缓存一致性由硬件 MESI 协议保证。Modified/Exclusive/Shared/Invalid 四态,缓存行在核间迁移。

一个常被忽略的观察:锁的开销不是"锁原语本身",而是 cache line 迁移。A 核持锁后又被 B 核抢,锁变量的 cache line 从 A 迁到 B,每次迁移 ~100 cycles。这解释了为什么锁竞争一多性能就崩。

无锁数据结构

无锁(lock-free)队列、栈、hash 表存在,但难写——稍不注意就 ABA 问题、内存回收问题。工业级实现(Folly、Crossbeam)经过多年调优,但自己写无锁结构不值得冒险。

在 AI 里

PyTorch 内部大量无锁原子(引用计数 Tensor、参数服务器更新)。C10 库的 c10::intrusive_ptr 比 std::shared_ptr 快,因为内嵌引用计数省了额外分配。

训练集群的参数服务器、异步 SGD 全是并发问题。单机多卡的 NCCL 同步也隐含内存屏障——保证 AllReduce 结果被所有 rank 看到。

2.2.7设备管理与系统调用

定位:syscall 是用户态与内核的唯一接口——io_uring、eBPF 在重新塑造它。

设备驱动模型

Linux 设备分三类:

  • 字符设备:流式读写(串口、键盘、/dev/random)
  • 块设备:块随机访问(磁盘、SSD)
  • 网络设备:不走 read/write,走 socket API

驱动向内核注册操作表(file_operations)——open、read、write、ioctl、mmap 等函数指针。应用通过文件系统(/dev/xxx)找到设备,调用走到驱动。

DMA 与零拷贝

DMA(Direct Memory Access)让设备直接读写内存,不占用 CPU。传统网络收包流程:网卡 DMA 到内核缓冲区 → 内核复制到 socket 缓冲 → 用户态 read 再复制到用户 buffer。三次拷贝。

零拷贝技术:sendfile(文件→socket 直接走内核)、splice(管道+pipe 实现零拷贝)、mmap+write(映射文件到用户空间)。Nginx、Kafka 大量用 sendfile。

系统调用

syscall 是用户态唯一进内核的方式。x86-64 用 syscall 指令(旧的 int 0x80 已淘汰),传参用特定寄存器。一次 syscall 开销 ~100 ns——不慢,但千万次循环里累积显著。

vDSO(virtual Dynamic Shared Object):把几个高频轻量 syscall(如 gettimeofday)实现成用户态函数,避免陷入内核。透明地通过动态链接提供。

eBPF

eBPF(extended Berkeley Packet Filter)是近年 Linux 最重要的新机制。允许用户把经过验证的小程序注入内核热路径(网络、跟踪、安全),不改内核、不加模块。

应用:tcpdump 用 BPF 做 packet filter;bpftrace 做动态追踪;Cilium 做容器网络(替代 iptables);Katran 做四层负载均衡(Facebook)。AI 可观测性平台(训练任务的细粒度监控)也越来越多用 eBPF。

eBPF 程序通过验证器(verifier)保证:无循环、无越界访问、有限执行时间。这让它可以安全在内核态跑。

容器

容器不是虚拟机——它就是 Linux 进程,只是利用了 namespace(隔离视图)和 cgroup(资源限制)。

  • namespace:PID、net、mount、IPC、UTS、user、cgroup（4.6+）、time（5.6+）共八种,每种隔离一种资源视图。容器看到自己是 PID 1 就是 PID namespace 的效果。
  • cgroup:限制 CPU、内存、块 I/O、网络带宽的配额。Kubernetes 的资源限额就是下发到 cgroup。

Docker、containerd、Podman 都是这两个原语的上层封装。AI 训练基本跑在容器里(便于环境复现、调度),理解 namespace/cgroup 在排查性能和权限问题时必不可少。

PART THREE

2.3 编程语言与编译器

语言决定你能思考什么,编译器决定你能跑多快。

2.3.1Rust 语言

定位:所有权模型 + 无 GC + 零成本抽象——系统编程的新基线。

所有权:Rust 的核心

Rust 最独特的是所有权(ownership)系统。三条规则:

  1. 每个值有且仅有一个所有者
  2. 所有者离开作用域时,值被自动 drop(释放资源)
  3. 赋值/传参默认是"移动"所有权(除非类型实现 Copy)

这条规则从根上消除了 C/C++ 最危险的内存安全 bug:use-after-free 和 double-free(内存泄漏在 Rc/Arc 循环引用下仍可能发生,但远不如前两者危险)。编译器在编译期静态检查,无运行时开销。

fn main() {
    let s1 = String::from("hello");
    let s2 = s1;           // 所有权移动, s1 不再有效
    // println!("{}", s1); // 编译错误!
    println!("{}", s2);    // OK
} // s2 离开作用域, 字符串自动释放

借用:不移动的临时访问

但凡事都移动所有权太麻烦——借用(borrowing)就是"临时拿去用不拥有":

fn len(s: &String) -> usize { s.len() }

fn main() {
    let s = String::from("hello");
    let n = len(&s);       // 传不可变借用
    println!("{}, len {}", s, n);  // s 仍有效
}

借用分两种:不可变借用 &T(多个并存)和可变借用 &mut T(独占)。核心规则:可变借用与任何其它借用互斥。这条规则编译期检查,直接消除数据竞争——并发代码编译通过就(在所有权+借用层面)没有竞争。

生命周期

借用必须不超过被借对象的寿命。大多数时候编译器自动推断,复杂场景要显式标注:

// 'a 是生命周期参数: 返回的借用不能超过两个输入中较短的那个
fn longer<'a>(s1: &'a str, s2: &'a str) -> &'a str {
    if s1.len() > s2.len() { s1 } else { s2 }
}

生命周期是 Rust 新手的第一个心智障碍。通过了这一关,你就会意识到 C/C++ 里"这个指针还有效吗"这种问题是多么靠直觉和约定——Rust 把它变成了编译器的工作。

零成本抽象

Rust 的 trait、迭代器、泛型都是零成本抽象:编译后和手写 C 一样快。迭代器链 v.iter().filter(...).map(...).sum() 经过 LLVM 优化后和手写 for 循环生成的汇编基本一致——没有虚函数开销、没有装箱。这和 Java/Python 的抽象有本质区别。

无 GC 但也不手动 malloc/free

Rust 没有垃圾回收——所有权模型让内存管理编译期决定。每个值的 drop 点编译器知道,Drop trait 里实现资源释放(类似 C++ RAII)。Box<T>、Rc<T>、Arc<T> 分别对应独占堆分配、单线程引用计数、多线程引用计数。

异步

Rust 的 async/await 是零成本 Future——每个 async 函数编译成状态机。运行时(tokio、async-std)负责调度状态机推进。这让 Rust 异步可以压榨出接近 C 的性能,但生态目前还在成熟。

在 AI 里

PyTorch 是 Python + C++,没 Rust。但 AI 周边基础设施大量用 Rust:tokenizers(HuggingFace)、burn(纯 Rust 深度学习框架)、candle(Rust 推理框架)、vLLM 的某些组件、各种向量数据库(Qdrant、LanceDB)。

Rust 和 Python 的互操作通过 PyO3 桥接,写 Python 扩展的新首选——比写 C 扩展安全得多。

2.3.2C 语言的现代演进

定位:C 仍是底层通用语言——看看它在演化什么,以及 C++ 的系统编程替代路径。简短。

这节讲广义的"C 系演进"。C 还没死——内核、驱动、DPDK、Redis 都还是 C。但 C 的工具链和周边在持续改进:

  • Sanitizer 系列:AddressSanitizer(ASan)检测越界、UAF;ThreadSanitizer(TSan)检测数据竞争;UBSan 检测未定义行为。编译时加 -fsanitize=address 就开,运行时开销 2–5x,能找出绝大多数内存 bug。
  • 静态分析:Clang Static Analyzer、Coverity、PVS-Studio。找出编译通过但运行时出错的代码。
  • Fuzz 测试:libFuzzer、AFL++。自动生成输入探测崩溃。OpenSSL、SQLite 都通过 fuzz 发现了大量 bug。

C++ 在系统编程的位置

C++ 表达力远超 C,但复杂度也爆炸。现代 C++(C++17/20/23)通过 std::unique_ptr、std::optional、range 等让写现代代码舒服很多。主要问题是没法禁用危险特性——C++ 的子集是安全的但必须靠纪律。

性能上 C++ 和 Rust 接近(都零成本抽象、都无 GC)。但所有权检查 Rust 在编译期做、C++ 靠程序员——这是根本差异。数据库、浏览器、游戏引擎仍重度用 C++(LLVM、Chromium、Unreal)。

语言选择的考量

如果今天要从零写一个网络设备固件、操作系统、数据库,语言选型的一般建议是:

  • - 内核/驱动:C(Linux 主流)或 Rust（Rust for Linux 已合入主线,多个子系统驱动已用 Rust 重写,2025 年后生态趋于成熟）。
  • - 中间件/服务:Rust 优先,Go 其次(gc 可接受)、C++ 在团队有经验时考虑。
  • - 上层应用:Go、Python、Java。

2.3.3编译基础原理

定位:词法/语法/语义分析 → IR → 代码生成。以 LLVM 为主线。

传统编译器的三段式

前端:源码 → AST → 中间表示(IR)

  • - 词法分析(lexing):字符流 → token 流。工具:flex、自己手写。
  • - 语法分析(parsing):token 流 → AST。工具:bison、ANTLR、递归下降手写。
  • - 语义分析:类型检查、作用域解析、生成类型标注的 AST。
  • - IR 生成:AST → IR(如 LLVM IR)。

中端:IR 上的优化(下一节)。

后端:IR → 目标汇编。包含指令选择、寄存器分配、指令调度。

LLVM 与 SSA

LLVM 的统治地位建立在两件事上:模块化设计(前端后端解耦)和 SSA(Static Single Assignment)IR。

SSA 要求每个变量只赋值一次。传统代码:

x = 1;
x = x + 2;
y = x * 3;

SSA 形式:

x1 = 1
x2 = x1 + 2
y1 = x2 * 3

每个变量一个定义,分析 def-use 关系变得简单——"这个用的是哪个定义"一眼看穿。控制流合并处用 φ 函数(phi)选择来自不同路径的值:

if (cond) x1 = 1;
else      x2 = 2;
x3 = φ(x1, x2);
y  = x3 + 1;

SSA 让所有数据流优化(常量传播、死代码消除、公共子表达式消除)都用简单算法就能做——这就是 LLVM 牛的地方。

LLVM IR 示例

// C 源码: int add(int a, int b) { return a + b; }

// LLVM IR (SSA 形式)
define i32 @add(i32 %a, i32 %b) {
entry:
  %sum = add i32 %a, %b
  ret i32 %sum
}

LLVM IR 是平台无关的——这段 IR 可以被不同后端(x86、ARM、RISC-V、GPU)分别编译。这就是跨平台编译器的架构基础。

后端

后端三件大事:

  1. 指令选择:IR 指令 → 目标架构指令。一条 IR 指令可能对应多条汇编(或反之)。常用算法:tree pattern matching、BURS。
  2. 寄存器分配:大多数 IR 是无限寄存器假设,后端要把它们映射到有限物理寄存器。经典算法是图染色——寄存器冲突图染色,不同色的变量可共用一个寄存器。LLVM 用更复杂的 linear scan 变种。
  3. 指令调度:乱序 CPU 下,编译期调度指令顺序减少流水线停顿。

在 AI 里

AI 编译器(TVM、MLIR、XLA)是通用编译器思想在张量计算上的延伸。MLIR(Multi-Level IR)是 LLVM 项目的扩展,提供多级中间表示——从 PyTorch 的 ATen 算子级、到 HLO 张量级、到 affine 循环级、到 LLVM 级,统一在一个框架。PyTorch 2.0 的 TorchDynamo + Inductor、Triton 的 MLIR 路径都是这个趋势的产物。

2.3.4编译优化

定位:几十种优化组合起来,能让代码快几倍——对 AI 算子优化有直接启发。

局部优化

常量传播:x = 5; y = x + 3; → y = 8;

常量折叠:编译期能算的结果就算出来。86400 * 30 不会等到运行时乘。

公共子表达式消除(CSE):

a = x * y + 1;
b = x * y + 2;
// → 
tmp = x * y;
a = tmp + 1;
b = tmp + 2;

死代码消除:结果没被使用的计算删掉。

强度削弱:把昂贵操作换成便宜的。x * 8 → x << 3;循环里的 i*c 替换成累加变量。

过程内优化

内联(inlining):把函数调用替换成函数体。省掉调用/返回开销,更重要的是让被内联代码和调用点上下文暴露给其它优化——"内联打开优化窗口"。

成本:代码膨胀。编译器启发式决定是否内联:小函数总内联、热点函数内联、不被多次调用的函数内联。inline 关键字只是建议,最终决定权在编译器。

循环优化(重点):

  • - 循环展开(unrolling):for (i=0;i<4;i++) a[i]=0; → 4 条 store。减少分支开销、暴露并行。
  • - 循环合并(fusion):两个相邻独立循环合并成一个。提高 cache 局部性。
  • - 循环分裂(fission):一个大循环拆成几个小循环。方便向量化、暴露并行。
  • - 循环交换(interchange):调整嵌套循环顺序,让内层循环访存连续。
  • - 循环不变代码外提(LICM):循环里不变的计算搬到循环外。
  • - 向量化:把标量循环改成 SIMD 指令。for (i=0;i 在 AVX2 下变成每次处理 8 个 float 的向量指令。

过程间优化

跨函数边界的优化:内联、过程间常量传播、死函数消除。需要全程序可见——LTO(Link Time Optimization)让链接器阶段看到所有 IR 再做优化,比传统"每个 .o 独立编译"更强。

PGO

Profile-Guided Optimization:先编译一个带 profiling 的版本,用真实负载跑一遍收集 profile,再重新编译让编译器基于 profile 决定优化(哪些分支更可能走、哪些函数热)。Chromium、MySQL 都用 PGO,通常能获 5–15% 性能提升。

自动向量化的限制

编译器的自动向量化很强但不是万能的。以下情况会放弃:

  • - 数据依赖(循环体内有 a[i] = a[i-1] + 1)
  • - 指针别名不确定(restrict 关键字可以帮编译器)
  • - 函数调用阻挡(pure 属性可以帮助)
  • - 复杂分支

所以对性能关键路径,查看编译器生成的汇编(gcc -S 或 Compiler Explorer)是专家必备技能。

在 AI 里

AI 算子的手工优化就是把这些编译优化思想用到张量计算:

  • - 算子融合 = 循环合并 + 内联,减少中间张量的内存往返。
  • - tiling = 循环分块,让数据停留在 cache/SRAM。
  • - 向量化到 SIMD/Tensor Core = 自动向量化的张量版本。
  • - auto-tuning(TVM、Triton AutoTune)= PGO 思路——实测不同参数找最佳。

理解了通用编译器优化,AI 编译器就是同一套思想的领域特化。

2.4 数据库

AI 基础设施里越来越重要的一环——从关系库到向量库。

2.4.1DB 基本原理

定位:存储引擎、事务、查询优化、日志——关系数据库的四个支柱。

存储引擎:B+Tree vs LSM-Tree

两种主流索引架构。

B+Tree(MySQL InnoDB、PostgreSQL):多路平衡树,叶子节点是数据页并用链表串联(方便范围扫描)。读快(一次 $\log_B n$),更新就地修改。缺点:频繁随机写对 SSD 不友好。

LSM-Tree(RocksDB、Cassandra、LevelDB):Log-Structured Merge Tree。写入先进内存(memtable),满了 flush 成不可变的 SSTable 到磁盘。后台合并(compaction)旧 SSTable。读可能需要查多层、查 bloom filter。写极快(纯顺序 I/O),读稍慢,写放大是主要问题。

选择:OLTP 读多 → B+Tree;写密集(日志、时序)→ LSM。

行存 vs 列存

行存(OLTP):一行数据物理连续。查整行/写整行快。MySQL、PostgreSQL。

列存(OLAP):同一列数据物理连续。OLAP 查询常只读几列(SELECT avg(salary) FROM emp)、过滤选择率高,列存能大幅减少 I/O。可以高压缩比(同列数据相似性高)。ClickHouse、Parquet、DuckDB。

ACID 事务

A(Atomicity):事务要么全做要么全不做。C(Consistency):事务保持数据约束。I(Isolation):并发事务互不干扰。D(Durability):提交后不丢。

实现靠两个机制:锁(保证 I)和日志(保证 A 和 D)。

锁与隔离级别

SQL 标准定义四个隔离级别,对应越来越严的异常避免:

级别脏读不可重复读幻读
Read Uncommitted可能可能可能
Read Committed不会可能可能
Repeatable Read不会不会可能
Serializable不会不会不会

实际实现用 2PL(Two-Phase Locking)或 MVCC(Multi-Version Concurrency Control)。MVCC 是现代主流:每次写产生新版本、读者看事务开始时的快照,读写不互相阻塞。PostgreSQL、MySQL InnoDB、Oracle 都用 MVCC。

WAL 与恢复

WAL(Write-Ahead Log):改数据页前先写日志(log 顺序 I/O 比随机 I/O 快得多)。崩溃后通过日志 redo 已提交事务、undo 未提交事务。ARIES 算法是工业标准恢复协议:Analysis → Redo → Undo。

查询优化器

SQL 是声明式——"要什么"而非"怎么做"。优化器把 SQL 翻译成执行计划。

过程:SQL 解析 → 逻辑计划(关系代数)→ 逻辑优化(谓词下推、子查询改写)→ 物理计划(选具体算子:hash join vs nested loop join、索引扫描 vs 全表扫描)→ 代价估计(基于统计信息:表大小、列分布)→ 选最小代价计划。

优化器是数据库的"大脑",好坏差一个数量级。

向量化执行

传统执行引擎是逐行火山模型(Volcano):每个算子 next() 返回一行。简单但每行开销大(虚函数调用、cache miss)。

向量化执行:每次 next() 返回一批行(如 1024)。开销摊薄到批,SIMD 指令一条处理多行,cache 利用率提升。ClickHouse、DuckDB、Apache Arrow 都是向量化的。比火山模型快 10× 级别。

在 AI 里

AI 基础设施里关系数据库仍是管理元数据的主力——用户、模型版本、训练任务记录、实验跟踪。PostgreSQL + MLflow 是常见组合。但 AI 最相关的 DB 是后三节要讲的:图、时序、向量。

2.4.2图数据库

定位:节点 + 边的原生存储——知识图谱、社交网络、推理链路的底层。

图的存储

关系库也能存图(两张表:节点表、边表),但查询多跳关系时要反复 join,性能差。图数据库原生存储邻接信息——每个节点直接持有"出边列表"指针,跳 k 步就是 k 次指针跟随。

属性图模型

主流图模型是属性图:节点有标签(类型)和属性(key-value);边有方向、类型、属性。例如社交图:(Alice :Person {age: 30})-[:FRIEND {since: 2020}]->(Bob :Person)。

查询语言:Cypher(Neo4j,最流行)、Gremlin(TinkerPop,跨库)、SPARQL(RDF 图,语义网)。

Cypher 示例

// 找 Alice 的朋友的朋友, 且不是 Alice 自己的朋友
MATCH (alice:Person {name: 'Alice'})-[:FRIEND]->(friend)-[:FRIEND]->(fof)
WHERE NOT (alice)-[:FRIEND]->(fof) AND fof <> alice
RETURN DISTINCT fof.name

同样查询用 SQL 需要自连接 + NOT EXISTS 子查询,可读性和性能都差。

常见图算法

  • BFS/DFS:最基础的遍历
  • 最短路径:Dijkstra(无负权)、Bellman-Ford(可负权)、A*(带启发)
  • PageRank:网页/节点重要性
  • 社区发现:Louvain、Label Propagation
  • 连通分量:Union-Find
  • 中心性:度、介数、接近度中心性

2.4.3时序数据库

定位:按时间戳排序的数据流——监控、IoT、金融行情的专属。

时序数据的特征

时序数据和普通关系数据几个不同:

  • - 按时间有序:天然有时间维度索引
  • - 近乎只写:老数据很少修改
  • - 高速率写入:监控可能每秒百万点
  • - 查询模式固定:范围聚合、降采样、时间窗口
  • - 数据有生命周期:老数据降精度或删除

关键技术:压缩

时序数据相邻值差异小,压缩比超高。两个经典技术:

Delta-of-Delta(时间戳压缩):记录 Δ(Δt)。定时采样的数据,Δ 基本恒定,ΔΔ 大多为 0,可变长编码到几比特。

XOR 编码(值压缩):相邻浮点数异或,高位多为 0。Gorilla(Facebook)论文的压缩算法,实测 1.37 字节/点——比原始 16 字节(time+value 各 8)压缩 10×+。

存储结构

时序库多数基于 LSM 变种。写入路径:

  1. 按 (metric, tags) 组成 series
  2. 每个 series 独立时间有序追加
  3. 按时间窗口分片(如 2 小时一个 chunk)
  4. chunk 老化后压缩存盘

降采样

原始 1 秒粒度数据保留 7 天,之后降到 1 分钟粒度保留 30 天,再降到 1 小时保留 1 年。自动物化聚合(continuous aggregate)省查询时聚合代价。

2.4.4向量数据库

定位:高维向量的最近邻搜索——RAG、语义检索、推荐的底座。

问题定义

给定 $N$ 个 $d$ 维向量和一个查询向量 $\mathbf{q}$,找出距离 $\mathbf{q}$ 最近的 $k$ 个。距离度量通常是 L2 或余弦相似度。

朴素暴搜 $O(Nd)$。$N = 10^8$、$d = 768$ 时每次查询几秒——不可接受。所以向量数据库的核心是近似最近邻(ANN):允许小概率返回非真最近邻,换来亚毫秒级查询。

ANN 算法家族

LSH(Locality-Sensitive Hashing):用随机超平面把空间分桶,相似向量以高概率落同桶。简单但召回率低。

IVF(Inverted File):先 k-means 聚类所有向量,查询时只搜最近的几个簇。典型用法 IVF_PQ(簇内用 PQ 量化存储向量)。

PQ(Product Quantization):把 $d$ 维向量切成 $m$ 段每段 $d/m$ 维,每段独立做 k-means 量化成 8 位 codebook 索引。$d=128$ 切 8 段每段 256 类,原本 512 字节压缩到 8 字节,64×。距离计算用查表近似。

HNSW（Hierarchical Navigable Small World）:多层图索引,最上层稀疏、最下层稠密。查询从最上层开始贪心"跳"向目标。召回率和速度在大多数场景最好,是目前主流选择。

HNSW 的直觉

把所有向量看作图的节点,每个节点连接若干近邻。查询过程从一个入口出发,每步跳到离查询向量更近的邻居,直到没有邻居比当前更近——就是局部最优点。

HNSW 加了层次:多层图,每层是下层的稀疏子集,高层边长、低层边短。查询先在高层大步跳接近目标区域,再逐层下降精化。类比跳表(skiplist)的二维推广。

混合检索

纯向量检索在专业术语、精确匹配上不如关键词。混合检索:向量(语义相似)+ BM25(关键词)结果融合,用 RRF(Reciprocal Rank Fusion)或其它 rerank。工业 RAG 系统几乎都是混合检索。

过滤与元数据

真实场景除了"相似",还要"只在这个用户的文档里搜"、"只要 2024 年之后的"。这要求向量数据库支持带过滤的 ANN。两种做法:

  • pre-filter:先按元数据过滤再 ANN。如果过滤后剩很少,直接暴搜。
  • post-filter:先 ANN 再过滤。但可能返回不足 k 个结果。

Qdrant、Milvus 做了filterable HNSW——在 HNSW 图上走时同时检查过滤条件,兼顾速度和正确性。

在 AI 里

RAG（Retrieval-Augmented Generation）:文档 → chunk → embedding → 存向量库;查询 → embedding → ANN 取 top-k → 拼进 prompt → LLM 回答。当下最主流的 LLM 应用模式。

语义搜索、推荐系统、图文检索、去重都是向量数据库的直接应用。长期记忆:Agent 的历史对话存成向量,按需检索——这是 LLM 应用中越来越重要的模式。

2.5.1高性能消息总线

定位:各种进程/服务间通信模式的基本盘——RPC、Pub/Sub、消息队列。

通信模式的分类

RPC 的核心技术点

gRPC 是现代 RPC 的事实标准。关键要素:

  • - sendfile/splice(内核级 zero-copy)
  • - mmap 共享内存(单机 IPC)

  • Kubernetes Service:内置 DNS + kube-proxy

可靠性保障

  • at-least-once:至少一次,可能重——需消费者幂等

PART SIX

2.6 网络栈:DPDK 与 VPP

高性能网络转发的两条主线——绕过内核的 DPDK,向量化处理的 VPP。

2.6.1内核协议栈的瓶颈

定位:为什么要绕过内核——理解 DPDK/VPP 存在的理由。

Linux 网络栈的路径

一个 TCP 包从网卡到应用要经过:

flowchart TB NIC[Network Card] --> IRQ[Hardware IRQ] IRQ --> SI[Softirq NET_RX] SI --> NP[netif_receive_skb] NP --> IP[IP layer] IP --> TCP[TCP layer] TCP --> SKB[socket buffer] SKB --> SYS[syscall read] SYS --> APP[Application]

每一层都有成本:

  • - 硬中断开销
  • - sk_buff 结构分配/释放(每包一次,内存分配器压力大)
  • - 多层协议栈(以太、IP、TCP)逐层解析
  • - 用户/内核态拷贝
  • - 系统调用开销
  • - 每包触发调度

这套架构为通用性设计得极好,但性能极限大约是单核 1–2 Mpps。要处理 10G 小包线速(14.88 Mpps)或 100G 线速(148.8 Mpps),内核栈做不到。

瓶颈的根源

中断风暴:高速率下每秒百万硬中断,CPU 大量时间在中断处理。NAPI 机制用"中断触发 → 禁中断 → 轮询"部分缓解,但还是受限。

内存分配:每收一个包要 kmalloc 一个 sk_buff,处理完 kfree。分配器本身是瓶颈。

锁:协议栈多处全局锁(路由表、连接表)。多核扩展性差。

缓存污染:sk_buff 跨核迁移、协议栈代码大量不命中 I-cache。

两条突破路线

绕过内核:用户态直接操作网卡。DPDK、Netmap、Snabb。

优化内核:XDP(eBPF 在驱动层过滤转发)、io_uring、AF_XDP。

后者越来越成熟(XDP 可做到单核 15 Mpps+),但 DPDK 仍是最大化性能的首选,下一节详讲。

2.6.2DPDK 架构

定位:绕过内核做用户态网络的工业标准框架。

DPDK 的设计哲学

DPDK(Data Plane Development Kit)是 Intel 于 2013 年开源的用户态网络框架（内部开发始于 2010 年前后）。四个核心决定:

  1. 用户态驱动:用 UIO/VFIO 把网卡从内核 unbind,用户态进程直接 mmap 设备寄存器。
  2. 轮询替代中断:应用主循环不停 poll 接收环,无中断开销。
  3. 预分配大页内存:启动时一次分配所有需要的内存,运行时无 malloc。
  4. 每核独立:每核一个线程,绑定固定网卡队列,共享数据极少。

结果:单核轻松 10 Mpps+,单服务器 100G 线速转发可达。

架构图

关键组件

EAL(Environment Abstraction Layer):抽象 OS 和硬件,统一接口(内存、CPU、设备)。rte_eal_init() 是每个 DPDK 程序的入口。

PMD(Poll Mode Driver):用户态网卡驱动。每种网卡一个 PMD(Intel ixgbe、Mellanox mlx5、virtio 等)。

mbuf:DPDK 的 sk_buff 等价物,但简化得多——固定大小头部 + 数据,无链表。预分配在 mempool 里。

rte_ring:无锁 MPMC(多生产多消费)环形队列,用户空间 IPC 的主要原语。

mempool:对象池。分配/释放 O(1),每核有 cache 减少竞争。

代价

DPDK 不是免费午餐:

  • - CPU 独占:轮询核 100% 占用,即使没包也在烧
  • - 无内核协议栈:TCP/UDP/ARP 都要自己实现或集成 mTCP、F-Stack 等用户态栈
  • - 开发复杂:标准 socket 库不能用
  • - 硬件绑定:网卡必须 DPDK 支持

所以 DPDK 适合专用转发设备(防火墙、负载均衡、NFV),不适合普通应用。

2.6.3DPDK 关键机制

定位:DPDK 性能的秘密——HugePages、无锁环、NUMA、RSS。

HugePages

DPDK 强制使用 HugePages(2MB 或 1GB)。原因 2.2.3 节提过:TLB 覆盖大幅扩大,避免 TLB miss。DPDK 典型启动参数指定每个 NUMA 节点预留多少 2MB 大页。

无锁环形队列

rte_ring 是 DPDK 最精巧的组件之一。它是 MPMC 环形队列,基于两对生产者/消费者 head/tail 指针:

// 简化版 rte_ring 核心变量
struct rte_ring {
    uint32_t size;           // 环容量 (2 的幂次,便于 & 取模)
    struct {
        uint32_t head;       // 下一个将写入的位置
        uint32_t tail;       // 已提交的位置
    } prod, cons;
    void *ring[size];
};

入队流程:原子 CAS 把 prod.head 加 n(预留槽位) → 拷贝数据 → 等 prod.tail 追上自己的原 head → 推进 prod.tail。这套两阶段让多生产者可以并发预留、顺序提交,保证环内数据被消费者看到时已完整。

NUMA 感知

现代服务器多 socket,跨 socket 访问内存延迟翻倍(远端 DRAM vs 本地 DRAM)。DPDK 通过 rte_socket_id() 让你指定 mempool、队列都分配在特定 NUMA 节点。生产代码必须所有资源 NUMA 对齐——网卡、CPU 核、内存池在同一 socket。

RSS 与多队列

单队列单核处理有上限。DPDK 配合网卡的 RSS,把不同流的包分到多个接收队列,每个队列绑一个核,真并行。

RSS 的 hash 基于 5-tuple——同一流的包永远走同一队列(保证流内有序),不同流均匀分布到各核。这个设计是所有现代高速转发的基础。

CPU 亲和性

DPDK 线程必须绑核(--lcores 参数)。避免调度迁移导致 cache miss。结合 isolcpus 内核参数把这些核从 Linux 调度器移除,完全独占。

Batch 处理

所有 DPDK API 都是批处理——rte_eth_rx_burst 一次收 32 个包,rte_eth_tx_burst 一次发 32 个。摊薄每包开销到接近零。应用逻辑也应该 batch 处理——解析 32 个包的以太头、查 32 个流表项等,对 cache 和流水线都友好。

零拷贝

mbuf 数据区直接就是 DMA 区域——网卡把包 DMA 到 mbuf,应用直接读,发送时网卡直接从 mbuf DMA 发出。整条数据路径零拷贝。

2.6.4VPP 向量化转发

定位:构建在 DPDK 之上的高性能可编程转发框架,思路是"对一批包连续做同一处理"。

VPP 的核心创新:图节点 + 向量化

VPP(Vector Packet Processing)是 FD.io 项目(基于 Cisco 开源)。它的两个核心概念:

转发图:处理流程是一张有向图,节点是"处理阶段"(如 ethernet-input、ip4-lookup、ip4-rewrite)。包从图的源节点流入,经过一系列节点,到达汇点(发送或丢弃)。每个节点只做一件事,可插拔扩展。

向量化:不是一次处理一个包,而是一次处理一批。传统"每包过完整转发流程"改成"每个节点把整批包处理完再传给下一节点"。

为什么向量化这么快

假设一批 256 个包,处理有 10 个阶段。传统方式:

for packet in 256:
    for stage in 10:
        do_stage(packet)

每个 stage 的代码被取指 256 次,每次可能 I-cache miss。

向量化方式:

for stage in 10:
    for packet in 256:
        do_stage(packet)

stage 代码只取 1 次,warm 在 I-cache 里。256 次循环迭代,数据可能在 D-cache 里(预取器友好)。I-cache 命中率的改善是 VPP 快的主因,而不是 SIMD 向量化。

这是编译器优化里 loop interchange 的网络转发版本——循环交换带来 cache 局部性改善。

VPP 节点示例

// ip4-lookup 节点的简化示意
static uword
ip4_lookup_node_fn(vlib_main_t *vm, vlib_node_runtime_t *node,
                    vlib_frame_t *frame) {
    u32 *from = vlib_frame_vector_args(frame);
    u32 n_left = frame->n_vectors;   // 这一批包数

    // 先处理 4-at-a-time (用 prefetch 掩盖 cache miss)
    while (n_left >= 4) {
        vlib_buffer_t *b0, *b1, *b2, *b3;
        b0 = vlib_get_buffer(vm, from[0]);
        b1 = vlib_get_buffer(vm, from[1]);
        b2 = vlib_get_buffer(vm, from[2]);
        b3 = vlib_get_buffer(vm, from[3]);

        // 预取下一批的 cache line
        CLIB_PREFETCH(vlib_get_buffer(vm, from[4]), CLIB_CACHE_LINE_BYTES, LOAD);
        // ...

        // 对 4 个包并行查路由
        ip4_fib_mtrie_lookup(...);   // 对 b0
        ip4_fib_mtrie_lookup(...);   // 对 b1
        ip4_fib_mtrie_lookup(...);   // 对 b2
        ip4_fib_mtrie_lookup(...);   // 对 b3

        from += 4; n_left -= 4;
    }

    // 剩余 0–3 个包单独处理
    while (n_left > 0) { /* ... */ }

    return frame->n_vectors;
}

这种 "4-at-a-time + prefetch" 是 VPP 的标志性代码模式。用指令级并行和显式 prefetch 把内存延迟隐藏在计算后面。

VPP 能做什么

  • - IPv4/IPv6 路由转发(FIB)
  • - MPLS
  • - VXLAN、GRE 等隧道
  • - NAT
  • - 基础防火墙 ACL
  • - 负载均衡
  • - IPSec

性能:单核十 Mpps+,双路服务器可做 100Gbps+ 线速。

与 DPDK 的关系

VPP 默认使用 DPDK 作为输入/输出驱动层,也支持 AF_XDP、Netmap 等。VPP 负责"包处理框架",DPDK 负责"和硬件对话"。两者不冲突、常结合使用。

2.6.5TCP / UDP / Raw IP

定位:传输层协议——日常用 socket 但要知道内部发生了什么。

IP 层

IP 是不可靠的包投递。每个 IP 包独立路由——没有"连接"的概念,中间路由器对每个包独立查表转发。IPv4 头 20 字节(含可选字段),IPv6 头 40 字节(但简化了字段)。

IP 层关键机制:分片(当下一跳 MTU 小于包大小)、TTL(每经一跳减 1,归零丢弃防环路)、校验和(只校头部)。

UDP

最薄的传输层——8 字节头(源端口、目的端口、长度、校验)。无连接、无重传、无拥塞控制、无顺序保证。

适合场景:DNS 查询、实时音视频、游戏状态同步、自定义可靠层(QUIC 就是 UDP 之上)。

TCP:核心三大机制

连接:三次握手建立,四次挥手关闭。维护序列号、确认号、窗口等状态。

可靠传输:基于序列号和 ACK 的重传。超时重传 + 快速重传(3 个重复 ACK 即重传,不等超时)。

拥塞控制:AIMD(Additive Increase Multiplicative Decrease)。拥塞窗口慢启动指数增、到阈值后线性增、遇丢包减半。算法有 Reno、Cubic(Linux 默认)、BBR(Google,按带宽/延迟建模,不靠丢包反馈)。

TCP 的性能调优

实际部署要关注的参数:

  • - 缓冲区大小:net.core.rmem_max/wmem_max。长肥网络(高带宽高延迟)需大缓冲。
  • - TCP_NODELAY:禁用 Nagle 算法,小包立即发送——低延迟交互必开。
  • - TCP_QUICKACK:不延迟 ACK。
  • - 拥塞算法:net.ipv4.tcp_congestion_control。数据中心、跨洲链路选 BBR 通常比 Cubic 好。
  • - TCP Fast Open:SYN 包携带数据,省 RTT。

Raw IP / Raw Socket

应用直接构造 IP 包(可以自定义任何协议)。需要 root 权限。用于:ping 工具(ICMP)、路由协议(OSPF、BGP)、自定义协议。socket(AF_INET, SOCK_RAW, IPPROTO_XXX)。

在 AI 里

AI 训练/推理通信主要用:

  • - TCP:gRPC/REST API、分布式训练的控制面
  • - RDMA（将在第 4 章 4.7 节介绍）:训练数据面,完全绕过 TCP/IP
  • - UDP:偶尔用于自定义 AllReduce 实现

2.6.6TLS 与 QUIC

定位:加密传输的两种范式——TLS over TCP vs QUIC(UDP + 加密)。

TLS 握手

TLS 1.3(2018)简化了握手,1 RTT 可用(TLS 1.2 要 2 RTT):

  1. Client Hello:携带支持的算法、ECDHE 公钥分量
  2. Server Hello + Certificate + Finished:服务器返回选定算法、证书、自己的 ECDHE、签名
  3. Client Finished:客户端验证证书,双方算出会话密钥

TLS 1.3 的 0-RTT 模式:已建立过会话的客户端可以在第一个包里直接发请求数据(用之前缓存的 PSK)。延迟为 0,但有重放攻击风险。

QUIC 的设计动机

TCP + TLS 有几个天生限制:

  1. TCP 握手 1 RTT + TLS 握手 1 RTT = 至少 2 RTT 才开始传数据
  2. TCP 是操作系统实现,升级慢(拥塞算法部署周期以年计)
  3. TCP 的 head-of-line blocking:单个包丢失阻塞整个流
  4. TCP 连接标识是 4-tuple,手机切 WiFi/4G 时 IP 变连接断

QUIC(Quick UDP Internet Connections)重新发明传输层,跑在 UDP 之上,解决这四个问题:

  • - 1 RTT 握手(首连),0 RTT(重连),加密和可靠传输合并
  • - 用户态实现:应用层库,升级只需换库
  • - 多流:一个连接内多个独立流,一流丢包不阻其它流
  • - Connection ID:不依赖 IP,网络切换连接保持

HTTP/3

HTTP/3 = HTTP/2 语义 + QUIC 传输。相比 HTTP/2 over TCP+TLS:握手快、多路复用无队头阻塞、网络切换顺滑。2022 年 IETF 标准化,主流浏览器和 CDN 已支持。

实现

QUIC 库:Google 的 QUICHE、Facebook 的 mvfst、Cloudflare 的 quiche、微软的 msquic。都是用户态。

在 AI 里

LLM 推理 API(OpenAI、Anthropic 等)多数已支持 HTTP/3,减少首 token 延迟。移动端 AI 应用(切换网络频繁)从 QUIC 受益更明显。

2.6.7转发管理与控制面

定位:数据面转发靠 DPDK/VPP 跑快,控制面负责下发策略、同步路由表。简短。

控制面 vs 数据面

现代网络设备是 CP/DP 分离:

  • 控制面:路由协议(BGP、OSPF)、策略计算、配置管理。跑在通用 CPU,Python/Go 实现都行。慢一点没关系。
  • 数据面:逐包处理。DPDK/VPP/NP/ASIC。必须快。

CP 算出路由,下发到 DP 的转发表(FIB)。DP 只做查表转发,不参与路由计算。

SDN

Software-Defined Networking 把这个分离推到极致:把控制面从每个设备里拿出来,集中到一个控制器。控制器通过 OpenFlow、gNMI、P4Runtime 等协议下发转发规则。

优点:全局视图优化、策略统一、设备简化(只要实现数据面)。缺点:控制器成为单点。

控制面配置协议

  • CLI:传统,ssh 登设备敲命令
  • SNMP:老牌监控,读多写少
  • NETCONF + YANG:结构化配置,事务支持
  • gNMI:基于 gRPC,流式遥测
  • OpenConfig:厂商无关的 YANG 模型

路由协议简述

  • BGP:互联网骨干协议,基于路径矢量,AS 间路由。TCP 179 端口,策略丰富。
  • OSPF:IGP(内部网关协议),链路状态算法(Dijkstra 最短路径)。收敛快,用于 AS 内。
  • IS-IS:类似 OSPF,ISP 常用。

在 AI 里

AI 训练集群的网络拓扑(fat-tree、dragonfly)对性能有巨大影响。但控制面本身和 AI 关系不大,这里只做整体感知。

2.6.8日志与可观测性

定位:Metrics + Logs + Traces 三位一体——没有可观测性就没有生产运维。

三大观测支柱

Metrics(指标):数值,按时间序列存。QPS、CPU、延迟。低频采样、小体积、易聚合。Prometheus 标配。

Logs(日志):离散事件的文本记录。高基数、大体积。ELK(Elasticsearch + Logstash + Kibana)、Loki 常用。

Traces(分布式追踪):一次请求在多个服务间的调用链。每个 span 记录开始时间、持续时间、标签。Jaeger、Zipkin、Tempo。

结构化日志

传统 printf 日志不利于机器处理。现代做法是结构化日志——JSON 或 logfmt 格式:

{"time":"2024-05-01T10:30:15Z","level":"INFO","service":"inference",
 "request_id":"abc123","model":"gpt-4","ttft_ms":234,"tokens":156}

带结构才能高效检索/聚合。Go 的 slog、Python 的 structlog、Rust 的 tracing 都推动这个方向。

OpenTelemetry

OpenTelemetry(OTel)是 CNCF 毕业项目,目标是统一观测数据采集规范——语言 SDK、数据模型、传输协议(OTLP)。一套 SDK 同时产出 metrics、logs、traces,可以对接不同后端(Prometheus、Jaeger、commercial APM)。

今天写新服务,选 OTel 基本不会错。

采样

全量 traces 太贵——每秒 100K 请求存下来磁盘都扛不住。采样策略:

  • 头采样:入口决定是否采,基于概率或规则
  • 尾采样:trace 完成后决定(可以针对错误、慢请求采)
  • 自适应采样:动态调整采样率

eBPF 可观测性

传统可观测性要应用侵入式埋点。eBPF 让你在内核抓任何函数调用、任何 syscall、任何 socket 事件,无需改应用。Pixie、Cilium Hubble、Parca 都是 eBPF 驱动的观测工具。

在 AI 里

LLM 推理的可观测性特殊指标:

  • - TTFT(首 token 时间):前端体验关键
  • - TPOT(每 token 生成时间):稳定生成速度
  • - tokens/sec throughput:系统吞吐
  • - KV Cache 命中率:prefix cache 效果
  • - batch size 分布:调度健康度
  • - GPU 利用率:硬件效率

训练可观测性:loss 曲线、梯度范数、学习率、各 rank 同步耗时、数据加载延迟。TensorBoard、Weights & Biases、MLflow 是专用平台。
